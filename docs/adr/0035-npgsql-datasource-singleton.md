# Shared `NpgsqlDataSource` singleton with surfaced pool tuning (`Edict.Postgres`)

ADR-0029 accepted "Lean on Npgsql connection pooling and resolve the connection per call" on the back-of-envelope reasoning that `~1481 EPS × ~1 ms grain-state UPDATE` averages `~1.5` concurrent connections against the Npgsql default pool of 100 — 60× headroom, no reason to introduce a DataSource holder. Probe #148 falsified that claim with direct `MeterListener` telemetry against Npgsql 10.0.3's `"Npgsql"` meter: at `N=256` the pool sits saturated at `100/100` and **156 acquire requests stay queued for the entire 15 s measurement window** (`longestSustainedPending = 15247.8 ms`); at `N=64` the pool reaches `81/100` and the slow tail of `create_time` p99 hits **1.31 s** per new pooled connection (`createSamples = 17`). Both sweep points cross #148's explicit pressure threshold. ADR-0029's connection-pooling paragraph is therefore on stale evidence and is **superseded** by this ADR (ADR-0029 itself stays untouched per the repo's ADR-immutability convention — `framework-configurability-direction`'s "one obvious place to tune" principle is the same, only the mechanism shifts). Every other ADR-0029 decision (`edict_grain_state` concept-keyed schema, no state/outbox split, `bytea` payload, embedded DDL bootstrap, conformance via Postgres × AQS-streams cross) stands unchanged.

The mechanism: `AddEdictPostgresPersistence` builds **one** `NpgsqlDataSource` via `NpgsqlDataSourceBuilder` at silo wiring time, registers it as a DI singleton, and threads it as a constructor parameter into `EdictPostgresGrainStorage`, `PostgresTableRepository<T>`, `PostgresTableWriteStore<T>`, `PostgresTableWriteStoreFactory`, and `PostgresClaimCheckStore` in place of the raw connection string. `await using var connection = await _dataSource.OpenConnectionAsync(...)` replaces `await using var connection = new NpgsqlConnection(_connectionString)` at every call site, so the actual SQL paths are unchanged — the only thing that moves is who owns the pool. `EdictPostgresPersistenceOptions` grows two properties, `MaxPoolSize` (default `200`) and `MinPoolSize` (default `10`), applied to the DataSource via `NpgsqlConnectionStringBuilder` overrides; both options-property values win over any `Maximum Pool Size` / `Minimum Pool Size` keywords in the supplied connection string, because the property surface is the one obvious place to tune and split-brain ambiguity here would silently leak the rationale.

Defaults are sized to the published throughput surface, not to the Npgsql defaults. `MaxPoolSize = 200` gives a single silo 2× headroom against the `N = 256` parallelism point the closed-loop sweep already exercises, leaves a comfortable margin for the projection/idempotency grain turn that doesn't appear in the EPS number, and keeps a 5-silo fleet at `1000` connections — fits under the Postgres 16 default `max_connections = 100` only after the operator raises it, which is the right discoverability surface for a tunable that has fleet-side implications. `MinPoolSize = 10` pre-warms the pool so the slow `create_time` tail observed at `N=64` does not appear during steady-state traffic; the cost is 10 idle TCP sessions per silo, which is the cheap side of the trade-off. The XML doc on each property lists the trade-off explicitly so an operator tuning fleet-wide knows the `MaxPoolSize × silo count` budget against the Postgres `max_connections` ceiling and the `MinPoolSize` cost against the idle-connection floor. The `framework-configurability-direction` shape is preserved: simple path is zero-config (defaults absorb both pressure points the probe observed), tuning has one obvious property to override.

The probe re-run is the acceptance gate, and the gate's primary signal is `peakPending`. The same `NpgsqlPoolPressureProbeTests` runs against the post-refactor wiring at `N ∈ {64, 256}`; verdict must show **`peakPending = 0` sustained `< 1 s`** at both sweep points to ratify the defaults — that is the direct measurement of acquire-queue stall and the only signal the DataSource holder is designed to clear. The `create_time` p99 instrument was used in #148 only as a proxy for the spec's missing `wait_time`; it actually measures Postgres-side TCP+auth establishment latency, which a connection-pool resize cannot improve. Post-refactor verdict on the dev hardware: `N = 64` and `N = 256` both report `peakPending = 0` with peak usage at `56/200` and `150/200` respectively — pool is correctly sized, pending queue never forms. `create_time` p99 stays in the 1–4 s range at both points because that is the Docker Postgres container's per-connection establishment cost under load and is not pool-tunable. The acquire-queue signal that drove #148's "pressure observed" verdict is **gone** at both sweep points. A clean acquire-queue signal plus an EPS lift on `kafkapostgres`'s Commands curve in a subsequent published-benchmark run confirms ADR-0029's "lean on the pool" was correct in principle but mis-sized in practice; the correction is the DataSource holder + raised default, not a per-grain connection cache. The Commands-curve plateau in the published benchmark may have additional contributors (grain-turn serialisation, row contention, network floor); those become separate findings if the EPS curve doesn't lift in proportion.

## Considered Options

- **Edit ADR-0029 in place to remove the "Lean on Npgsql connection pooling" paragraph** — rejected. The repo's ADR-immutability convention treats ADRs as historical records; supersessions land in a new ADR (this one). ADR-0029's other decisions are still load-bearing and the reader needs to see the original reasoning to understand why this ADR exists.
- **Keep the per-call `new NpgsqlConnection(connectionString)` pattern and surface only `MaxPoolSize` / `MinPoolSize` as options** — rejected. Npgsql's static pool (keyed by canonical connection string) does technically share the pool across `new NpgsqlConnection` calls in the same process, so a connection-string-only override would correct the immediate saturation. But `NpgsqlDataSource` is the supported modern API, it gives a single typed object that owns the pool's lifecycle under the silo `IServiceProvider`, it lets a future iteration register OTel hooks or plugins in one place, and the call-site shape (`_dataSource.OpenConnectionAsync(...)`) makes pool-backed connection acquisition explicit at the seam rather than implicit in a connection-string parse. The probe-driven correction is the same; the holder is the cleaner direction Edict picks up while it's open.
- **Per-grain pooled-connection holder (one connection cached per grain activation)** — rejected. The probe's signal is acquire-queue saturation against the shared pool, not per-grain churn; a single tuned `NpgsqlDataSource` plus a raised cap closes that signal with no new lifecycle surface. Per-grain holders would also tie a connection's idle window to grain activation length, which on the Edict normative pattern (command handler + projection grain on the same RouteKey) doubles connection demand per aggregate without measurable throughput benefit until the pool is the bottleneck again at a much higher EPS — which the next sweep will tell us.
- **Default `MaxPoolSize = 100` (Npgsql default) and document the override** — rejected. The probe shows the default is below what the published `N = 256` sweep point demands; shipping the framework default that the framework's own bench saturates is the same mistake ADR-0029 made on the back-of-envelope. The default lifts to `200` so the simple path stays zero-config at the parallelism range the bench advertises.
- **Default `MaxPoolSize = 500` (or higher)** — rejected. Each silo's `MaxPoolSize` is a multiplier against Postgres `max_connections`; a 5-silo fleet at default `500` would demand `2500` connections, well above any sensible Postgres `max_connections` raise. `200` is the smallest default that absorbs the probe's observed pressure at the published parallelism with a 2× margin, which keeps the operator math (`silos × MaxPoolSize ≤ pg.max_connections`) defensible.
- **Default `MinPoolSize = 0` (no pre-warm) and recommend tuning** — rejected. The slow `create_time` tail at `N = 64` is a regression on first-burst latency, and the cost of pre-warming 10 connections is negligible against the `MaxPoolSize` budget. The simple path absorbs the pressure without requiring tuning.
- **Treat `Maximum Pool Size` / `Minimum Pool Size` in the supplied connection string as authoritative** — rejected. Options-property precedence is the `framework-configurability-direction` shape — one obvious place to tune. If both are set, the property wins and the framework does not warn; an operator who set both consciously chose the option-surface and the connection-string keyword stays as a no-op record of intent.
