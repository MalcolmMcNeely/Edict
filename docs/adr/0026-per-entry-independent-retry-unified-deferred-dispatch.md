# Per-entry independent retry; unified deferred-dispatch engine

**Status:** accepted — supersedes in part ADR 0018 (drops FIFO stop-at-head), ADR 0019 (head-wedge mechanics), ADR 0022 (dead-letter-as-tail-promotion vocabulary), ADR 0023 (the asymmetry that was EdictEventHandler-only generalises to all deferred dispatch), ADR 0024 (receiver-side `BlobMissing` tracker folds into the engine), ADR 0012 (atomicity preserved for inline-envelope dispatch only; weakened for pointer-bearing envelopes on projections). Builds on ADR 0025 (blob substrate removes the row cap that previously constrained the Outbox slice).

ADR 0018 made `OutboxHost.DrainAsync` **FIFO, stop-at-head** to preserve per-aggregate causal order across the four effect kinds. In practice the cost dominates the benefit. A single poison effect blocks every subsequent effect on *that grain key* until `MaxAttempts` exhaustion (~30–40 min with default knobs — `docs/resiliency-analysis.md` §3.2.1). The benefit — "consumers downstream of one aggregate see events in raise-order" — is a guarantee the rest of the at-least-once stack already does not honour: a redelivered older event lands at the consumer *after* a newer event of the same aggregate; the ring dedupes but the ordering is lost. Consumers must already be reorder-tolerant for the cross-aggregate and redelivery cases, so FIFO buys nothing structural for them and costs the per-aggregate wedge under outage. We retract the causal-order guarantee and drop FIFO stop-at-head.

The receiver-side claim-check `BlobMissing` retry loop (ADR 0024) is the other duplicated retry harness. It shares the engine's primitives (`OutboxBackoff`, `EdictOutboxOptions.MaxAttempts`, `IDeadLetterPromoter`) but lives in its own per-key tracker on `GrainEnvelope`, driven by Orleans stream-redelivery instead of the engine's reminder. With FIFO gone, the engine becomes a per-entry-independent-retry surface — a perfect home for the receiver-side path. The stream observer's pointer-envelope branch stages an `InvokeHandler` entry (commit ring slot + entry atomically in one write); the engine retries the entry through its standard backoff; `MaxAttempts` exhaustion promotes via the standard `IDeadLetterPromoter`. The `BlobMissing.Attempts` slice, `HandleBlobMissingAsync`, and the rethrow-to-Orleans dance all delete. One mechanism, one driver.

The `InvokeHandler` entry kind absorbs the receiver-side responsibility. Today its payload is a serialised concrete event; tomorrow it is a serialised `EdictEventEnvelope` (inline-payload or pointer-bearing), and the executor calls `ClaimCheckUnwrap.ApplyAsync` before invoking `deferredDispatch`. The `OutboxEffectKind` enum and its frozen ordinals are **unchanged** — `InvokeHandler` is widened, not renamed and not joined. The executor's branching (inline → deserialise; pointer → fetch) is the existing `ClaimCheckUnwrap` logic relocated, not reimplemented.

Drop FIFO does not mean drop ordering on the happy path. Drain is **sequential, in insertion order** (the `OutboxSlice.Pending` list keeps insertion order; the drain walks it processing one ready entry at a time, writing state after each success or backoff transition). N events raised in one Handle commit, all succeeding on first attempt, publish in raise order — the principle-of-least-surprise property is preserved when nothing fails. Only the **retry path** reorders: a failed entry slips its `NextAttemptUtc` into the future; the drain proceeds past it; later entries fire ahead. This is the reorder consumers must already tolerate.

Two stream-observer paths fall out. **Inline-payload envelopes** (the common case for normal-size events) dispatch directly — `Handle` runs inline, ring slot and any staged effects commit atomically in one write, ADR 0012's projection ring-equals-row atomicity property is preserved. **Pointer-bearing envelopes** (oversized events only) stage an `InvokeHandler` entry — ring slot commits at intake (so Orleans does not redeliver), the entry executes later (fetch blob → dispatch), and ring-commit-before-effect-completion becomes the rule for that specific case. ADR 0012's closer holds for the inline case; the asymmetry ADR 0023 introduced for EdictEventHandler now extends to projections receiving pointer envelopes. We accept the weakening because the case it weakens — oversized-event payloads >30 KB landing in projection rows — is operationally rare and already in tension with the destination's 32 KB per-property cap.

Publisher-side claim-check blob writes stay **synchronous at the commit boundary** (ADR 0024 unchanged on this point). Folding the publisher-side write into the engine would require carrying raw event bytes in `OutboxSlice.Pending` until the executor uploads them. ADR 0025 removes the row-cap obstacle to large grain documents, but the per-write IO (every `WriteStateAsync` rewrites the full blob) and per-activation memory cost (every `OnActivateAsync` reads it) make 100 MB-class events sitting in grain state a silo-hostile pattern at scale. The pointer-in-entry property is size-bounding and worth preserving. "One mechanism" applies to **post-commit, asynchronous** retry; the pre-commit blob write is a deliberate exception with size-bounding as its load-bearing justification.

## Considered Options

- **Keep FIFO stop-at-head; tune `MaxAttempts` down to soften the wedge.** Rejected. A lower `MaxAttempts` (e.g. 3) drops the wedge to ~30 s but also accelerates dead-letter on legitimately transient outages. The trade is favourable on throughput, unfavourable on RCA fidelity — and the root failure mode (one poison effect blocking every other effect on the same grain key) is unchanged. The aggregate is still wedged; just for less time.
- **Drop FIFO only for `UpsertRow`/`SendCommand`/`InvokeHandler`; keep FIFO for `PublishEvent`.** Rejected. The engine would carry a per-effect-kind FIFO concept that exists to preserve a guarantee (intra-handle stream-order) that the rest of the at-least-once stack already violates on redelivery. Half the complexity of FIFO for a guarantee consumers cannot rely on anyway. If a consumer needs intra-handle order for two raised events, the consumer raises one event that carries both pieces of data — modelling, not framework.
- **Layer 1 fold: cosmetic dedup only.** The two retry loops (engine + receiver-side `BlobMissing`) share primitives but stay as separate harnesses; clean up the code symmetry, do not unify the driver. Rejected. The maintainer's stated goal was *one mechanism, one concept*; preserving two drivers fails that test for a cosmetic gain.
- **Layer 2 fold with universal staging at the stream observer** (every event-driven dispatch stages an entry, even inline-payload envelopes). Rejected. Breaks ADR 0012's projection ring-equals-row atomicity for the *common case* (inline-payload, normal-size events) — the case the closer was designed for. Two stream-observer paths is a small, well-bounded asymmetry; universal staging trades it for a worse one.
- **Add a new `MaterialiseAndDispatch` effect kind alongside `InvokeHandler`.** Rejected. EdictEventHandler staging "I have the concrete event, just dispatch deferred" vs. projections staging "I have a pointer envelope, fetch then dispatch" is a runtime branch dressed up as a type distinction. The executor still has to know how to materialise. Widening `InvokeHandler` keeps the enum lean (no new frozen ordinal) and consolidates "deferred dispatch" as one concept.
- **Rename `InvokeHandler` to `Dispatch`.** Rejected as cosmetic. The entry does invoke a handler; the name is correct. The rename is a frozen-ordinal change and a CONTEXT.md churn for no consumer-visible improvement.
- **Parallelise drain via `Task.WhenAll` for N independently-ready entries.** Rejected for v1. Volunteers for reorder on the happy path (Q5's principle-of-least-surprise property) and double-counts on the throughput case — per-grain throughput is gated by `WriteStateAsync` latency, not executor I/O, so parallelising the I/O does not help while writes serialise. Revisit as an optimisation if profiling proves the executor wall-time dominates.
- **Fold publisher-side claim-check write into the engine** (entry carries raw event bytes; executor uploads to blob; engine retries on failure with the same backoff). Rejected. ADR 0025 removed the row cap that would have made this impossible, but the per-write and per-activation IO cost of carrying multi-MB or 100 MB-class events in grain state is silo-hostile at scale. Pointer-in-entry is size-bounding and load-bearing. The "one mechanism" framing scopes to post-commit, asynchronous retry; the synchronous pre-commit write is an honest exception.
- **Inline transient-fault retry at `ClaimCheckPolicy.PutAsync`** (3–5 attempts with short backoff before throwing). Out of scope for this ADR. A separate "transient-fault handling at external service boundaries" concern; structurally different from the engine's durable-retry harness. Worth its own slice if storage hiccups prove a real source of consumer-visible command failures.
- **Schedule the lazy drain reminder for `min(NextAttemptUtc)` instead of a fixed 1-min period.** Rejected. Orleans's reminder service has a 1-minute floor; the dynamic-schedule optimisation cannot beat it. The fixed 1-min reminder + drain-skip-when-gated shape is already optimal.

## Consequences

- **Drain semantics.** `OutboxHost.DrainAsync` walks `Pending` in insertion order. For each entry: if `NextAttemptUtc > now`, skip; else attempt. On success, `Ack(entryId)` removes the entry and writes state. On failure, `FailWithBackoff(entryId, now, options)` bumps `AttemptCount` and `NextAttemptUtc`, writes state; if `AttemptCount >= MaxAttempts`, `Promote(entryId, promoted)` removes the entry and appends the synthetic `EdictDeadLetterRaised` `PublishEvent` entry, single write. Drain **continues** past the failed entry — no break. Reminder is registered when any entries remain (gated or otherwise) and unregistered when `Pending.Count == 0`.
- **`OutboxSlice` transition module.** `AckHead`/`FailHeadWithBackoff`/`PromoteHead` are renamed to entry-id-based `Ack(Guid entryId)`/`FailWithBackoff(Guid entryId, DateTimeOffset now, EdictOutboxOptions options)`/`Promote(Guid entryId, OutboxEntry promoted)`. `Head` accessor is removed (no entry is privileged). The slice remains a `List<OutboxEntry>` keyed by `EntryId` insertion order — happy-path drain follows the list; the order does not encode dispatch priority.
- **Receiver-side path.** `EdictIdempotencyBase.UnwrapAndDispatchAsync` bifurcates: non-envelope and inline-payload envelopes dispatch through the existing `OnStreamEventAsync` (ring check → `DispatchAsync` → ring commit + staged effects atomic, ADR 0012 preserved); pointer-bearing envelopes commit the ring slot for the envelope's wire-frame `EventId` and stage an `InvokeHandler` entry in one atomic write, then return. `HandleBlobMissingAsync` deletes. `GrainEnvelope<TPayload>.BlobMissing` (and its inner `BlobMissingAttempt` type) deletes. The frozen `[Alias("GrainEnvelope`1")]` literal is preserved; the slot removal is silent on first deserialization of any persisted state (pre-release, no production deployments).
- **`InvokeHandlerExecutor` widening.** Payload semantics shift from "serialised concrete event" to "serialised `EdictEventEnvelope`" (inline or pointer). The executor calls `ClaimCheckUnwrap.ApplyAsync` before invoking `deferredDispatch`. The unwrap logic is the existing `ClaimCheckUnwrap` code; the executor becomes its second call site. `IEdictClaimCheckStore` is injected through the same DI surface already used by `ClaimCheckUnwrap`.
- **`IDeadLetterPromoter` consolidation.** `PromoteBlobMissing` collapses into the standard `Promote` path. The engine, on `InvokeHandler` `MaxAttempts` exhaustion where the entry payload is a pointer-bearing envelope, synthesises `EdictEventBlobMissing` as the dead-letter notification's source event. The promoter's `EffectTarget` mapping handles the discriminator; the projection row schema is unchanged (still carries `ClaimCheckKey`, the field issue #74 added).
- **ADR 0012 amended.** The atomicity claim (ring commit ≡ row write commit in one atomic grain-state write) is preserved for **inline-payload envelope delivery to projections** (the common case for normal-size events — the case the closer was designed for). For **pointer-bearing envelope delivery to projections**, ring commit happens at intake (entry-stage time) and row write commits when the entry executes — two writes. This is the same asymmetry ADR 0023 introduced for `EdictEventHandler`. CONTEXT.md's **Table Projection Builder** entry is updated to reflect the asymmetry.
- **ADR 0023 amended.** "Ring commit ≠ effect completion" is no longer EdictEventHandler-specific; it is the rule for any deferred-dispatch path (which now includes pointer-bearing envelopes on every consumer role). The contrast against Projection Builder is now a contrast against *inline-payload* delivery to Projection Builders specifically.
- **ADR 0024 amended.** The receiver-side `EdictEventBlobMissing` dead-letter promotion path is preserved in outcome but its mechanism changes: the in-grain `BlobMissing.Attempts` tracker + rethrow-to-Orleans dance is replaced by the engine's per-entry retry. The publisher-side blob-write at the commit boundary is unchanged.
- **No public-surface changes.** `EdictEvent`, `EdictEventEnvelope`, `IEdictClaimCheckStore`, `IEdictDeadLetterRepository`, `IEdictSender`, `EdictOutboxOptions`, the grain bases, and the `[Alias]`-tagged persisted types are all unchanged. The brand-rule clause (a) surface is untouched. This ADR is a framework-internal mechanism change.
- **Resiliency analysis §3.2.1** ("Head-of-line blocking per aggregate") is marked resolved by this ADR. §3.2.2's *observability* half (saturation metrics, intake-throttle policy) remains open — a deep pending backlog is still operationally suspect and worth surfacing even without a wedge.
- **Test surface (per ADR 0016).** `OutboxHostTests` and `OutboxRecoveryTests` add cases for continue-past-failure drain, per-entry independent backoff gating, and the dead-letter-during-drain path with subsequent entries continuing to fire. `InvokeHandlerFifoTests` is deleted (FIFO is no longer the contract). `InvokeHandlerExecutorTests` adds inline-envelope and pointer-envelope unwrap cases. `ClaimCheckUnwrapDeadLetterPredicateTests` is reviewed for relevance (the predicate may still gate the dead-letter projection's pointer-preservation; check post-refactor). New `Edict.Azure.Tests` case asserts pointer-envelope receiver-side delivery dead-letters via the engine path against Azurite (replaces the deleted `BlobMissing` integration tests in spirit).
- **No migration tooling.** Edict is pre-shipped. Persisted state carrying a populated `BlobMissing` slot loses it silently on first deserialization; persisted `OutboxSlice` documents continue to deserialise (`AttemptCount` / `NextAttemptUtc` / `Pending` shape is unchanged).
- **Slicing.** Land as one PR. The receiver-side fold, the executor widening, the dead-letter promoter consolidation, and the drain-semantics change are mutually entangled — splitting them creates a window where one half ships without the other and the framework's retry behaviour is incoherent. The doc set (this ADR + the six forward-pointers + CONTEXT.md + resiliency-analysis §3.2.1) lands in the same PR.
