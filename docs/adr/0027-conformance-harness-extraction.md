# Conformance harness extraction (`Edict.Tests.Conformance`)

The mechanism battery — Commands, Events, Outbox, Sagas, Projections, EventHandler, Idempotency, DeadLetter, ClaimCheck — lives in `Edict.Tests.Conformance` as abstract scenario classes parameterised over a fixture (`abstract class CommandPipelineEndToEndScenarios<TFixture> where TFixture : ConformanceFixture`); each provider's test project (`Edict.Azure.Tests` today; `Edict.Kafka.Tests` and `Edict.Postgres.Tests` next) supplies a `ConformanceFixture` subtype and an empty sealed inheritor per scenario class binding the provider's fixture under that provider's `[Collection]`. This **extends ADR-0024**, whose "build a conformance harness now" entry was deferred until a second provider concretely existed: that trigger is now firing (parent PRD #133 introduces Kafka + Postgres), and ADR-0024's deferral has expired. The fixture seam is two-layered — `Edict.Substrate.Azurite` (and the soon-to-arrive `Edict.Substrate.KafkaPostgres`) owns *container lifecycle and framework wiring* via the `ISubstrate` / `ISubstrateRuntime` contracts from ADR-0030, while the provider's `ConformanceFixture` owns *test-isolation tweaks* (per-fixture container/table names, shorter visibility timeouts for observable redelivery, in-memory `PubSubStore`) layered on top of the substrate's defaults. **Resilience and Load stay per-provider** because failure modes are substrate-specific (Azurite pause via Docker.DotNet vs Kafka rebalance via consumer-group operations vs Postgres connection drop); the post-fault invariants those tests assert (no lost events, exactly-once eventual effect, per-aggregate ordering preserved) are shared, but the fault-injection step is provider code using native APIs and so does not abstract usefully. Resilience tests are declared serial within their assembly (xUnit `[CollectionDefinition(..., DisableParallelization = true)]`) so cross-test fault leakage cannot corrupt parallel test execution outside Resilience. The **chaos-scope principle** travels with this ADR: Edict's chaos surface (ADR-0025) is a consumer-facing guardrail for the idempotency / at-least-once contract on otherwise deterministic in-process dispatch — it is *not* a network-fault or malformed-data simulator. Real-substrate conformance is already as chaotic as the real substrate; the harness must not introduce new fault-injection surfaces beyond what `Edict.Testing` already owns. The substrate-extraction lands in a per-folder slice sequence (Commands → Events → Outbox → Projections → EventHandler → Idempotency → Sagas → DeadLetter → ClaimCheck), each slice independently shippable, each preserving the Azure test count by replacing scenario-shaped test files with empty inheritors of the abstract scenario classes; the per-folder workload types (commands, events, handlers) move into `Edict.Tests.Conformance` with the `Azure*` brand prefix dropped, because they were never substrate-specific — only their containing namespace was.

## Considered Options

- **One-shot extraction (move all nine folders in one PR)** — rejected: 75 files / ~6 000 LOC moving in one change makes review and bisect-on-regression painful; per-folder slices are independently shippable and each is its own conformance-count delta.
- **Provider-neutral fixture base owning silo wiring + substrate-specific overrides** — rejected: the per-fixture test-isolation overrides (short visibility timeout for redelivery, in-memory `PubSubStore`, per-fixture-unique container names) diverge enough from `AzuriteSubstrate`'s framework defaults that a base owning wiring would force every conformance-only tweak through an extension hook, and the conformance fixture would re-override the substrate's `ConfigureSilo` for most of the wiring anyway. The shared base owns only the contract — `IEdictSender` (and, as later folders move in, the substrate-typed surfaces each scenario needs); the substrate brings the backend up; the fixture owns the silo wiring including test-isolation.
- **Keep workload types under per-provider folders, parametrise scenarios over the workload type** — rejected: the types (`PlaceOrderCommand`, `OrderPlacedEvent`, `OrderCommandHandler`) are substrate-neutral by design — the `Azure*` brand prefix was a folder-derived accident; moving them into conformance and dropping the prefix removes duplication across every future provider's test project.
- **Two test projects (`Edict.Tests.Conformance` for scenarios, separate `Edict.Tests.Conformance.Workload` for workload types)** — rejected: the scenarios and the workload they exercise are one slice; splitting them adds a project boundary that buys no isolation (no other consumer references the workload).
- **Toxiproxy / network-layer fault-injection fixture in `Edict.Tests.Conformance`** — rejected: the chaos-scope principle above forbids it. The PRD's Toxiproxy bullet is contradicted by `Edict chaos scope` (memory note) and `delay not useful in tests`; revisit only if a substrate-agnostic latency invariant emerges in practice — none has yet.
- **Disable collection-parallelism at the assembly level** — rejected: papers over Resilience's fault leakage where the existing `DisableParallelization = true` collections already scope it correctly.
