# Grain state on the blob substrate

Edict's Outbox commits its pending-effect list *inline in the grain-state document* (ADR 0015), and under sustained downstream outage the `OutboxSlice.Pending` list grows by one entry per affected event — so backing `edict-state` with Azure Table (32 KB per-property + 1 MB per-row caps) means a handler raising N events per command, blocked behind a slow downstream during a burst, will reach the row cap **before** claim-check's per-event threshold (ADR 0020) can save it. Once the row overflows the grain is hard-broken: every subsequent write returns an Azure Table error, the framework cannot persist the next state change or even the dead-letter promotion, and the operator's only recovery is forensic out-of-band row repair. We move `edict-state` to **Azure Blob** uniformly across every Edict grain type — block-blob upload via `If-Match` ETag is atomic and single-writer-wins on the same primitive Table uses for row updates, so the Outbox's "one atomic write" guarantee is preserved by construction; `PubSubStore` stays on Azure Table (Orleans-internal, bounded shape, not subject to Outbox-growth dynamics). This narrows ADR 0020's three-surfaces motivation for claim-check to two: the wire transport cap (the primary motivation) and the dead-letter row cap (the binding number in the threshold default); the grain-document surface is retired. Per-write latency is ~2× Table in the small-payload regime — accepted as the cost of cap-overflow elimination; the account-level transaction ceiling is identical, so operators saturating it shard across accounts independent of the substrate choice.
