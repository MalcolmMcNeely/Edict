# Claim check for oversized events

Edict's event hop crosses two infrastructure caps — Azure Queue Storage limits a queue message to ~48 KB, and Azure Table (which backs the dead-letter projection row's `PayloadJson` per ADR 0011/0018) limits a single string property to 32 KB — so a `EdictEventEnvelope` is the **universal wire-format wrapper** carried on every event hop. The envelope is `EdictEvent`-derived and carries either an inline `byte[]?` payload (the common case, ~10-byte tax) or a `string?` claim-check pointer to a blob (the rare oversized case); the receiver's stream observer unwraps before consumer `HandleAsync` is dispatched, so consumer code is identical for in-band and claim-checked events. The publish-time decision (serialise, measure, optionally write blob in parallel via `Task.WhenAll`, wrap, assert post-wrap ≤ 32 KB) happens **at the commit boundary**, not inside `Raise` — keeping `Raise` synchronous, localising the policy in one step, and parallelising N oversized writes per HandleAsync. The blob store is **append-only** (Model B; the framework never deletes — retention is the operator's lifecycle policy); the dead-letter projection row is special-cased to **store the blob key, not the fetched body**, preserving the forensic affordance without trying to materialise a >32 KB payload into a 32 KB property.

## Considered Options

- **Conditional-wrap envelope** (only oversized events wrapped) — rejected: the universal envelope buys a permanent wire-format evolution point (encryption envelopes, schema-version tags, multi-tenancy hints) at a ~10-byte common-case tax — the envelope is not solely a claim-check vehicle.
- **Two independent mechanisms, one for the wire-hop and one for the poison-row case** — rejected: they share a blob store and a serialisation step; one mechanism with a single special-case at the dead-letter projection gives operators one place to look.
- **Model A — framework-managed delete-after-ack** — rejected: when an oversized event is promoted to dead-letter the wire-hop blob would already be deleted before the projection captured it, forcing a second forensic-store copy.
- **Inside-`Raise` blob writes (`await Raise(evt)`)** — rejected: forecloses parallelisation of multiple oversized blob writes in one HandleAsync (each await is sequential); breaking API change unjustified by the local-visibility win.
- **Receiver-side missing-blob → silent skip / throw-forever / block-grain** — all rejected: silent loss contradicts the no-silent-loss stance; throwing forever wedges all events on the route key with no operator surface; blocking on small operator errors creates outages. The chosen path funnels missing-blob into the engine's dead-letter promotion (ADR 0019), with `EdictEventBlobMissing` as the synthetic notification.
- **Threshold lives in central `EdictOptions`** — rejected: the cap is set by the publish-path storage stack (per-provider); the knob lives on the provider's options (`EdictAzureStreamsOptions.ClaimCheckThresholdBytes`, default 30 720 = 32 KB minus 2 KB framing headroom).
