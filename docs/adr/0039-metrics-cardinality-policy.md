# Metrics cardinality policy

Edict metric attribute sets are designed for **ops alerting first, test assertion second**: every dimension a metric carries is bounded at compile time (grain types, event types, command types, the four `OutboxEffectKind` values) or by a framework-owned canonical allowlist; no metric exposes per-aggregate-instance cardinality (no `aggregate_key`, no `grain_key`, no per-saga-instance label). The dedup-window dimension is `consumer_grain_type`, not `consumer_grain_key`. The dead-letter `failure_reason` dimension is a closed allowlist of five values â€” `Timeout`, `Saturated`, `Serialization`, `Substrate`, `Unhandled` â€” populated by an internal classifier mapping the throwing exception type to the bucket; anything the classifier can't bucket lands in `Unhandled` rather than leaking the exception type name. Tests that need per-aggregate or per-key specificity combine a `MeterListener` (for the metric increment) with an `ActivityListener` (for the per-trace span carrying `edict.command.route_key` etc.) â€” the dual-listener pattern is already proven in `Edict.Benchmarks.Throughput.Tests/NpgsqlPoolListener.cs` and falls naturally out of the chaos-default-on test rig. No `EdictMetricsOptions.MaxCardinalityPerInstrument` knob ships: every dimension comes from a compile-time-bounded source, so there is no fire for the fuse to protect against, and a soft cap that silently buckets values into `_other_` would distort metrics under the operator's nose.

The audience tie-breaker is the load-bearing call here. A test that wants to verify "the specific aggregate I just sent to dead-lettered" naturally reaches for `aggregate_key` on the metric, which is the attribute that would explode cardinality on a 10k-aggregate production fleet â€” exactly the kind of test-shaped decision that creates an unrunnable production metric. Pushing per-trace forensic detail onto the span (which is per-trace anyway, no cardinality problem) and keeping the metric coarse means the metric works as an alerting surface and tests still reach the specificity they need by attaching both listeners. The `failure_reason` allowlist has a downstream forcing function: the dead-letter classifier maps from the throwing exception type to one of the five buckets, so new exception types thrown from outbox effect executors need a sensible bucket â€” the existence of the allowlist nudges the framework toward custom `Edict*` exceptions over raw `InvalidOperationException` (which always bucket to `Unhandled`), reinforcing the broader maintainer direction recorded in the `edict-exceptions-only-direction` memory. The "no cap knob" decision is reversible â€” a consumer hitting unbounded cardinality on, say, `edict.event.type` because they author 10k event types can ask for the knob and we add it â€” but the default is "no knob," because shipping the knob today would invite consumers to lean on it instead of bounding their own type fanout.

## Considered Options

- **Tests win, with an OTel `View` stripping high-cardinality dimensions in prod** â€” rejected. The View leaks as a consumer responsibility; ADR-0023 prefers framework-owned config defaults, and a View that consumers must register is a regression. Tests that don't run the View see attributes that prod doesn't carry, so test coverage diverges from production reality.
- **Two attribute sets gated by `EdictMetricsOptions.DetailedAttributes = true`** â€” rejected. Doubles every metric's contract (every assertion has two truth shapes to verify), and a consumer running outside the test profile silently loses test coverage. The dual-listener test pattern gives tests the specificity they need without splitting the metric.
- **Expose `aggregate_key` (Guid string) on outbox metrics** â€” rejected. Per-instance cardinality on a 10k-aggregate silo gives 10k unique series per metric per silo, multiplied by the number of silos in the cluster. Prometheus survives the data but operator dashboards become unreadable; the metric stops being an alerting surface and becomes a forensic stream the dead-letter table already provides better.
- **Use the raw exception type name as `failure_reason`** â€” rejected. Consumer-thrown exceptions live in the call path; an arbitrary consumer-defined `MyCustomDownstreamException` would appear as a `failure_reason` value, blowing the cardinality boundary and forcing operators to enumerate exception types they don't author. The allowlist + `Unhandled` bucket keeps the dimension closed.
- **Drop `failure_reason` entirely; classify in the dead-letter projection only** â€” rejected. The whole alerting story is "PublishEvent timeouts spiking vs UpsertRow serialization failures"; dropping the dimension forces operators to join the metric (a rate) against a table (the dead-letter row) at alert time. The closed allowlist gives the alert decomposition story without the cardinality risk.
- **Ship `EdictMetricsOptions.MaxCardinalityPerInstrument` as a soft fuse** â€” rejected. With every dimension already bounded at compile time there's no fire for the fuse to put out; a soft cap that silently buckets values into `_other_` distorts metrics under the operator's nose without surfacing the breach. Non-breaking to add later if a real consumer asks.
