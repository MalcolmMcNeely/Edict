# Agentic tooling — MCP server and skill bundle for Edict consumers

Edict ships two consumer-facing `dotnet tool`s under a new `/Agentic Tooling/` solution folder: `Edict.Mcp` (an MCP server exposing five tools that observe a consumer's Edict-based solution) and `Edict.ClaudeSkills` (an installer that drops five skill markdown files into the consumer's `.claude/skills/`). The two halves are co-designed: each Roslyn-walk MCP tool ships with a paired clause in a skill body that tells the model when to invoke it, so the agent reaches for the tool at the right moment instead of relying on description-match intuition alone.

The MCP tools split into a docs half (`edict_describe_glossary_term` and `edict_lookup_adr`, served from `CONTEXT.md` and `docs/adr/` embedded as resources in the NuGet package) and a Roslyn half (`edict_list_handlers`, `edict_list_route_keys`, `edict_describe_silo_wiring`, loaded via `MSBuildWorkspace` with full semantic resolution), plus a meta tool `edict_describe_mcp_state` for self-reporting the loaded solution and indexed handler counts. Workspace discovery defaults to CWD walk-up looking for `*.slnx`/`*.sln`, with an explicit `--solution` override for multi-solution repos. Skills are `edict-authoring`, `edict-contracts`, `edict-silo-wiring`, `edict-testing`, `edict-diagnostics` — carved along consumer invocation moments (the *when*, not the *what*), so the model picks one skill per task instead of triaging between five similarly-named type-based skills.

Production-mode dictates the implementation choices: `MSBuildWorkspace` over a syntactic-only walk (correctness over startup speed, since the agent runs on a dev machine where the .NET SDK is present); two separate `dotnet tool`s rather than one with subcommands (clean separation of responsibilities, lockstep release per ADR-0043 neutralises the per-package overhead); architecture-test invariants enforcing that `Edict.Mcp` stays substrate-neutral and `Edict.ClaudeSkills` holds no framework references; a cross-package consistency test asserting every skill's MCP-tool reference resolves to a registered tool and every registered MCP tool has at least one skill caller. The Edict repo itself dogfoods both packages — its own `.mcp.json` wires `Edict.Mcp` and its `.claude/skills/` holds the installed bundle, so regressions surface live during framework development rather than waiting on a consumer to report them.

## Considered Options

- **Single `dotnet tool` with subcommands (`edict-mcp install-skills` + `edict-mcp serve`)** — rejected: the long-running MCP server and the one-shot installer are different operational shapes, and lockstep release means two packages cost nothing extra to maintain.
- **Skill bundle as copy-paste markdown (no installer)** — rejected: portfolio-but-production discipline rules out hand-rolled copy-paste; a versioned `dotnet tool` gives consumers a clean install story aligned with how every other Edict package ships.
- **Syntactic-only Roslyn walk (no `MSBuildWorkspace` dependency)** — rejected: misses transitive inheritance (a consumer's `class OrderHandler : MyBase` where `MyBase : EdictCommandHandler<...>` would be invisible), and the dev-machine assumption makes the SDK-availability cost a non-issue.
- **Carve skills by framework type (one per base class: `edict-command-handler`, `edict-event-handler`, `edict-saga`, `edict-projection-builder`, `edict-table-projection-builder`)** — rejected: forces the agent to triage between five similarly-named skills for any task; folding the type taxonomy into `edict-authoring`'s decision tree keeps "Saga vs Event Handler vs Command Handler" answerable in one place.
- **Skill named `edict-observability` covering both instrumentation reference and failure investigation** — rejected: those are different invocation moments and consumers rarely extend Edict's instrumentation. Renamed to `edict-diagnostics` with the focus on failure investigation; the live Tier-2 query tools (deferred) become its primary callers when they land.
- **`edict_run_analyzers` as a Tier-1 MCP tool** — rejected: analyzers already run at build and the IDE surfaces them; an MCP wrapper would only duplicate the IDE. The slot is taken by `edict_describe_glossary_term`, which serves a need the agent has no other path to satisfy.
- **Defer dogfooding to a sibling demo repo** — rejected: the audience-mixing concern (framework-dev skills vs consumer skills coexisting in this repo's `.claude/`) is smaller than first thought because framework-dev skills are SessionStart-hooked and consumer skills are invocation-based, so they don't contend at invocation time.
- **Explicit `--solution` required in `.mcp.json` (no auto-detect)** — rejected: zero-config install is the right v1 demo experience; auto-detection covers the single-solution-at-root majority and the override is a one-line escape hatch for the multi-solution minority.

## Consequences

- Two new NuGet IDs join Edict's lockstep release pipeline (ADR-0043): `Edict.Mcp` (CLI command `edict-mcp`) and `Edict.ClaudeSkills` (CLI command `edict-skills install`).
- `Edict.Mcp` depends on `Microsoft.CodeAnalysis.Workspaces.MSBuild` + `Microsoft.Build.Locator`; calls `MSBuildLocator.RegisterDefaults()` once at startup; holds a single cached `MSBuildWorkspace` for the server lifetime.
- `CONTEXT.md` and `docs/adr/*.md` are embedded as resources in `Edict.Mcp` via build globs, so the MCP package version is also a snapshot of Edict's docs at that version. A drift-guard test asserts the embedded resource set matches the docs folder at build time.
- Three new test projects, all under `/Agentic Tooling/`: `Edict.Mcp.Tests` (handler unit tests over in-memory Roslyn workspaces, embedded-resource drift guards), `Edict.ClaudeSkills.Tests` (install command), and `Edict.AgenticTooling.Architecture.Tests` (the four invariants plus the cross-package consistency check between skill bodies and the MCP tool registry).
- The four invariants live in `Edict.AgenticTooling.Architecture.Tests` rather than the framework-wide `Edict.Architecture.Tests` so that agentic-tooling quality concerns stay co-located with what they constrain: `Edict.Mcp` has no `Edict.Azure*`/`Edict.Postgres`/`Edict.Kafka`/`Edict.Substrate*` references; `Edict.Mcp` has no `Edict.Testing` reference; `Edict.ClaudeSkills` holds no framework references; skill bodies reference only registered tool names.
- This repo's `.mcp.json` wires `Edict.Mcp` at the repo root, and `.claude/skills/` gains the five `edict-*` consumer skills. Skill descriptions explicitly scope to "consumer app built on Edict" so they don't fire when editing framework code, avoiding contention with the SessionStart-hooked framework-dev skills.
- The five MCP tools are scoped to Tier 1 (static observation of a consumer's solution). Tier 2 (live observation: dead-letter queries, saga progress, projection rows, outbox metrics snapshots) is explicitly deferred until consumer demand surfaces, at which point `edict-diagnostics` becomes the natural caller.
