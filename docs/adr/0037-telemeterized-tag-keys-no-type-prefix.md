# Telemeterized tag keys carry no message-type prefix

`[EdictTelemeterized]` properties land on the span as `edict.{snake_case_property_name}` regardless of the declaring `EdictCommand` or `EdictEvent` type. The snake-case derivation is `System.Text.Json.JsonNamingPolicy.SnakeCaseLower.ConvertName(propertyName)`; in the netstandard2.0 generator that algorithm is reachable only via an in-house port (`Edict.Generators.SnakeCaseLower`) pinned to the BCL output by the worked-examples theory in `TelemeterizedTagNamingTests`. The generator-side prefix `EdictWellKnownNames.TelemeterizedTagPrefix = "edict."` is held parity with `SemanticConventions.Telemeterized.Prefix` by `TelemeterizedTagPrefixParityTests` (ADR-0033's reflection sentinel pattern). Both SendAsync-path emitters write the same tag-key shape: the `SendInterceptorEmitter` fast path that fires on concrete-typed `SendAsync(command)` call sites (ADR-0034) and the `CommandRouteRegistrarEmitter` slow path that fires when `IEdictSender.SendAsync` is called against the `EdictCommand` base type. Same key under both paths is the lock; the architecture test enforces the prefix half, the emitter snapshots enforce the suffix half.

The shipped shape was `edict.{lowertype}.{lowerprop}` — `edict.placeordercommand.sku`, with the message-type baked into the tag key. The span name already disambiguates the message type (`edict.command PlaceOrderCommand`, `edict.event.publish OrderPlacedEvent`), so the key duplicating that identity broke cross-message-type aggregation: an operator slicing handler latency or filtering "show me everything for customer X" had to OR-join one tag key per type, which is the opposite of how OTel semantic conventions work everywhere else (`http.method` everywhere, not `getrequest.method` / `postrequest.method`). Removing the type prefix collapses same-named properties across declaring types onto one key, which is what consumers actually want when the property name is a domain noun (`CustomerId`, `OrderId`, `Reason`) — same domain noun, same tag key. Cases where two unrelated types happen to share a primitive property name with semantically different meanings will collide on the same key; that is rare and usually intentional, and a future analyzer slice (deferred — see PRD #171 Out of Scope) can promote it to a diagnostic if it surfaces as a real footgun. `JsonNamingPolicy.SnakeCaseLower` handles the acronym cases (`SKU` → `sku`, `HTTPMethod` → `http_method`, `CustomerID` → `customer_id`, `OrderId` → `order_id`) the way OTel attribute naming does, so the convention is "do what the rest of the OTel ecosystem does" rather than "do what `ToLowerInvariant` happens to produce". No override knob (e.g. `[EdictTelemeterized("custom_key")]`) ships in this slice; the auto-derive covers 100% of the worked-examples table and the parameter is a non-breaking add when a real consumer asks. The analyzer EDICT005 (`Telemeterized must be primitive`) still enforces the cardinality floor at authoring time; nothing here relaxes that gate.

## Considered Options

- **Keep `edict.{lowertype}.{lowerprop}`** — rejected. Span name already carries the type; tag key duplicating it is dead information that breaks cross-type aggregation. Operators querying "everything for customer X" across the fleet have to enumerate every declaring type, which is exactly the semconv anti-pattern.
- **`edict.{lowertype}.{snake_case_prop}`** (keep the type prefix, fix only the snake-case)** — rejected for the same reason: the prefix is the bit that breaks aggregation, not the property suffix. Fixing only the property-name shape inherits the cross-type query problem and ships a hyphenated middle-ground that nobody actually wants.
- **Ship an override-key parameter (`[EdictTelemeterized("custom_key")]`) now** — rejected (deferred). The auto-derive via `JsonNamingPolicy.SnakeCaseLower` produces what consumers expect for every primitive property name we can name; the optional ctor argument is a non-breaking add when a real consumer asks for it. Shipping the knob without a use case is the kind of speculative surface the framework's `framework-configurability-direction` is meant to avoid.
- **Add a cross-type name-collision diagnostic (EDICT006) in this slice** — rejected (deferred to PRD #171 Out of Scope). Collisions are rare and usually intentional (same domain noun = same key). A diagnostic that flags `OrderCreated.Reason` and `OrderCancelled.Reason` writing the same key would be noise in 95% of cases; the analyzer slice waits for a real footgun.
- **Use `ToLowerInvariant` (the previous algorithm) for the property half** — rejected. `ToLowerInvariant` produces `customerid`, `httpmethod`, `orderid` — flat, unsegmented, and divergent from how OTel attribute names are written across every other instrumentation library in the .NET ecosystem. Aligning to `SnakeCaseLower` matches the OTel attribute style and gives multi-word readability for free.
- **Reference `System.Text.Json` directly from the generator instead of the in-house port** — rejected. The generator targets netstandard2.0; pulling `System.Text.Json` 8+ into a Roslyn analyzer load context is a known load-context hazard. A pinned in-house port plus the BCL-equivalence theory keeps the convention sourced from one file and avoids the analyzer-load-graph weight.
