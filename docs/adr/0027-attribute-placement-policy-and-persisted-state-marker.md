# Attribute-placement policy and `IEdictPersistedState` marker

**Status:** accepted — extends ADR 0010 (alias codegen on commands generalises to also emit `[MessagePackObject(keyAsPropertyName: true)]`); extends ADR 0017's frozen-literal `[Alias]` rule from `IdempotencyState` to every consumer-authored persisted-state type; closes the AQTN dead-letter hole in `docs/resiliency-analysis.md` §3.2.3 (priority #1). Builds on ADR 0007 (MessagePack stays the wire format for commands/events — this ADR is orthogonal to the wire-format choice).

The framework today has an unstated split: the source generator emits `[Alias(nameof(TheClass))]` on consumer commands and events; consumers hand-write `[MessagePackObject(keyAsPropertyName: true)]` on the same types; consumers hand-write `[GenerateSerializer]` + `[Alias]` + `[Id(n)]` on their aggregate state. There is no documented principle for which attribute goes which way, no analyzer for the consumer-written ones, and one mechanism — `UpsertRowEffect.RowTypeName` — that uses `Type.GetType(AssemblyQualifiedName)` for row POCO identity, bypassing Orleans's `[Alias]` machinery entirely. Renaming or moving a row POCO dead-letters every in-flight `UpsertRow` entry forever, with no compile-time signal.

The principle this ADR pins down is narrower than "mechanical vs decision" and load-bearing: **the generator may only emit attribute values that are safe to recompute every build from the current syntax. Anything that must remain stable across source-history events (rename, refactor, property reorder) must be consumer-written.** `[Alias(nameof(TheClass))]` on a command satisfies the test — ADR 0010 explicitly accepts class-rename as a deliberate breaking change for wire identity. `[MessagePackObject(keyAsPropertyName: true)]` on a command satisfies it — the value is always `true` by ADR 0007. `[Alias("Frozen")]` on persisted state **fails** the test — the whole point of "frozen" is "this alias does not change when the consumer renames the class," and only the consumer at the keyboard knows when an evolution is "the same logical type" versus "a deliberate breaking change." A generator that emits `[Alias(nameof(TheClass))]` on aggregate state would, on first class rename, silently make every persisted document in the cluster unreadable. The principle is the asymmetry, not the symmetry.

The generator therefore widens to cover **both** of the framework's wire-format obligations for commands and events (`[Alias]` and `[MessagePackObject(true)]`), and the persisted-state contract moves to a consumer-owned marker. `IEdictPersistedState` — an empty marker in `Edict.Contracts.Persistence` — is the anchor; every consumer-authored persisted-state type (aggregate state, saga progress, table-projection row POCO) implements it; the three grain bases constrain their type parameters to it (`EdictCommandHandler<TState> where TState : IEdictPersistedState, new()`, similarly for `EdictSaga<TProgress>` and `EdictTableProjectionBuilder<T>`). The framework's `EdictUnit` shim implements the marker so the existing `EdictCommandHandler : EdictCommandHandler<EdictUnit>` and `EdictEventHandler : EdictIdempotencyBase<EdictUnit>` paths continue to compile. The analyzer EDICT011 fires on any `IEdictPersistedState` implementation that is missing `[GenerateSerializer]`, has an `[Alias]` whose value is not a string literal (a `nameof` here silently defeats the rename-survival guard — the literal-vs-`nameof` distinction is the analyzer's main contribution beyond presence-checking), or has a public property without `[Id(n)]`. The codefix inserts all three at once, with the alias defaulted to the simple class name as a literal the consumer may edit.

`UpsertRowEffect` changes shape to match: the `RowTypeName` field (assembly-qualified name, resolved at drain via `Type.GetType`) is replaced by `RowAlias` (the frozen string from the row POCO's `[Alias]`); the drain resolves it via `Orleans.Serialization.TypeConverter.Parse`, the same machinery that already resolves every other persisted/wire type in the framework. The publisher (`EdictTableProjectionBuilder.BuildUpsertEntry`) computes the alias by reading the row POCO's `[Alias]` attribute (an Orleans `TypeConverter.Format` call, not by string-templating the simple name — so the consumer's literal choice is what travels). This closes the resiliency-analysis hole as a side effect of the broader policy, not as a one-off patch.

One pre-existing analyzer hypothesis is **dropped**, not introduced. A "concrete `EdictCommand`/`EdictEvent` must carry `[MessagePackObject(keyAsPropertyName: true)]`" rule (EDICT010 as briefly proposed during design) is unnecessary once the generator emits the attribute itself — the C# compiler errors on a duplicate non-`AllowMultiple` attribute across partial declarations if a consumer hand-writes a conflicting one, so the failure mode the analyzer would catch is structurally impossible.

## Considered Options

- **Status quo — generator emits `[Alias]` only; consumer continues to hand-write `[MessagePackObject]`; no marker; `UpsertRowEffect` keeps `Type.GetType(AQTN)`.** Rejected. Leaves the AQTN hole open. Keeps an unstated split (why does Edict generate one Orleans-touching attribute and not the other?). Forecloses an analyzer surface for the silent failure modes (a consumer who forgets `[MessagePackObject]` fails at first cross-process send; a consumer who forgets the frozen-literal discipline on aggregate state fails on first rename, months later).

- **Codegen `[Alias(nameof(TheClass))]` on aggregate state / saga progress / row POCOs too, for symmetry with commands and events.** Rejected and load-bearing in this ADR's reasoning. The frozen-literal requirement on persisted state (ADR 0017) exists precisely so the alias survives class rename. Codegen of `nameof(TheClass)` changes the emitted value on first rename, making every persisted document in the cluster unreadable. The same logic forbids codegen of `[Id(n)]` (integer IDs must remain stable across property reorders, which a generator cannot guarantee from the current syntax tree).

- **Codegen `[GenerateSerializer]` only on persisted state; require consumer to hand-write `[Alias]` + `[Id(n)]`.** Rejected. Requires every consumer state type to become `partial` (so the generator can emit a sibling declaration), for one saved attribute. The consumer's mental model would have to be "Edict generates *one* of the three Orleans-codec attributes on my state; I write the other two." Compared with the clean "Edict generates nothing on persisted state; you own the persisted contract," it is a leaky abstraction for a token line saving.

- **Retract ADR 0007 (drop MessagePack for commands/events, move them onto the Orleans codec).** Rejected. The wire-format choice is orthogonal to the attribute-placement policy. Orleans codec on commands/events would *increase* consumer per-property ceremony (`[Id(n)]` per property + ID-reuse hazard for the type's lifetime) — the opposite of what "push responsibility to the consumer" buys when scrutinised. ADR 0007's no-`[Union]` polymorphism win (the GO-condition for the cross-assembly contracts pattern) would have to be re-proven. Two distinct concerns deserve two distinct decisions; this ADR scopes only the attribute-placement policy.

- **No marker interface; the analyzer discovers persisted-state types by walking generic type arguments at `EdictCommandHandler<T>` / `EdictSaga<T>` / `EdictTableProjectionBuilder<T>` use-sites.** Rejected. Works but loses three properties of the marker design: orphan-detection (a state type declared but not yet wired into a grain is invisible to the analyzer), use-site enforcement (a consumer who passes a non-persisted-state type as `TState` fails to compile *before* the analyzer runs), and self-documentation (a reader of `class OrderState : IEdictPersistedState` sees the role on the type, not via the grain that references it). The empty-marker pattern is cheap; the discovery and constraint benefits are real.

- **Three role-specific markers — `IEdictAggregateState`, `IEdictSagaProgress`, `IEdictProjectionRow`.** Rejected. The three roles have identical attribute requirements, identical rename concerns, identical hazards. The role distinction is already signalled by the grain base that holds the type; encoding it a second time on the type itself buys narrower constraints but at the cost of triple the interface surface and triple the analyzer rule. A consumer who legitimately reuses a type across roles (rare but legal) would fight the type system for no framework benefit. If a future rule needs role-specificity (e.g. "saga progress must have an `IsTerminal` property"), it can attach to a derived marker without retracting the unified base.

- **Ship the `UpsertRowEffect` alias fix as a one-off patch, leave the broader attribute-placement policy unstated.** Rejected. The hole `resiliency-analysis.md` §3.2.3 flagged is the loudest case of a more general pattern: silent consumer failure when a framework-implicit attribute discipline is forgotten. Fixing the loud case without writing down the principle leaves the quiet cases (consumer aggregate state with no analyzer; consumer who hand-writes `[Alias(nameof(X))]` thinking it's frozen) open. Pre-release is the right time to write down the principle in one place.

## Consequences

- **`Edict.Contracts.Persistence/IEdictPersistedState.cs` is the new public marker.** Empty interface, brand-prefixed (rule (a) — consumers implement it). No runtime cost.

- **Generic constraints widen on three bases.** `EdictCommandHandler<TState> where TState : IEdictPersistedState, new()`. `EdictSaga<TProgress> where TProgress : IEdictPersistedState, new()`. `EdictTableProjectionBuilder<T> where T : class, IEdictPersistedState, new()`. The `class` constraint stays where it is (row POCOs round-trip through the Table store SDK as reference types). `EdictUnit` implements `IEdictPersistedState` to keep the `EdictCommandHandler : EdictCommandHandler<EdictUnit>` shim compiling.

- **`EdictCommandGenerator` and `EdictEventGenerator` widen their `EmitAlias` emit pass to also write `[global::MessagePack.MessagePackObjectAttribute(true)]` on the sibling partial declaration.** Single new line per command and per event in the generated `.g.cs`. The consumer deletes the corresponding hand-written `[MessagePackObject(keyAsPropertyName: true)]` from every command and event in the contracts assemblies (Sample + tests + framework-shipped events such as `EdictDeadLetterRaised` and `EdictEventEnvelope`). EDICT006 and EDICT007 (the `partial` requirements) carry across unchanged — same rule, now two reasons.

- **`UpsertRowEffect` reshape.** `RowTypeName : string` (AQTN) is removed; `RowAlias : string` (frozen `[Alias]` literal) is added at the same `[Id(3)]` ordinal — pre-release, no migration tooling, the persisted layout shift is a clean break. `UpsertRowExecutor` resolves the alias via `Orleans.Serialization.TypeConverter.Parse` (which respects every `[Alias]` registered in the Orleans manifest); `EdictTableProjectionBuilder.BuildUpsertEntry` reads the row POCO's `[Alias]` via `TypeConverter.Format(typeof(T))` so the consumer's literal choice is the value that travels. No JSON-body change — the row body stays JSON inside `RowJson` because the destination is a JSON-friendly Table cell, not an Orleans-codec'd byte stream.

- **EDICT011 — new analyzer.** Fires on any `INamedTypeSymbol` implementing `IEdictPersistedState` that (a) lacks `[GenerateSerializer]`, (b) lacks `[Alias]`, (c) has an `[Alias]` whose constructor argument is not a string literal expression (a `nameof(TheClass)` argument is the canonical defeat to catch — diagnostic message names it specifically), or (d) has any public instance property without `[Id(n)]`. Codefix inserts all missing attributes at once, with the alias defaulted to the simple class name as a string literal (`[Alias("OrderState")]`). The `[Id(n)]` codefix assigns the next free integer in declaration order for the type's first apply; subsequent properties are the consumer's responsibility (the analyzer does not renumber). Diagnostic location: the type declaration. Test surface follows the existing `Edict.Analyzers.Tests` pattern (positive + negative cases, codefix snapshot tests).

- **EDICT010 is *not* introduced.** A "concrete `EdictCommand`/`EdictEvent` must carry `[MessagePackObject(keyAsPropertyName: true)]`" analyzer was proposed during design and dropped once the generator absorbed the emit. The C# compiler's duplicate-attribute error catches any consumer who hand-writes a conflicting `[MessagePackObject(...)]`, so there is no silent failure mode the analyzer would cover. The diagnostic-ID space stays contiguous (EDICT001–EDICT009 then EDICT011); no renumber.

- **Consumer-facing attribute counts.** A command shrinks from three consumer-written attributes (`[MessagePackObject]`, `partial`, `[EdictRouteKey]` on the property) to two (`partial`, `[EdictRouteKey]`). An event shrinks similarly, plus its required `[EdictStream]`. A persisted-state type grows from "nothing required" (row POCO) or "consumer learns the discipline" (aggregate state) to the explicit set `[GenerateSerializer]` + `[Alias("literal")]` + `[Id(n)]` + `: IEdictPersistedState` — caught by EDICT011, the codefix writes them all.

- **CONTEXT.md updates.** Brand-prefix rule list adds `IEdictPersistedState`. Command Handler / Saga / Table Projection Builder entries note the generic constraint and EDICT011. The flagged-ambiguity on persisted-state aliasing extends to reference the unified marker + codegen-vs-consumer principle. A new flagged-ambiguity entry records the codegen-on-transient / consumer-on-persisted line. This ADR lands in the same PR as those edits.

- **`docs/resiliency-analysis.md` priority #1** (the `UpsertRowEffect.RowTypeName` AQTN hole) is marked resolved. The §3.2.3 row in the high-severity table becomes a strikethrough referencing this ADR.

- **No migration tooling.** Pre-release. Persisted state carrying an old `UpsertRowEffect.RowTypeName` field deserialises to the default (empty string); any such in-flight entry resolves to a missing type at drain and dead-letters. There are no production deployments to protect.

- **Slicing.** The marker introduction, the three generic constraints, the generator widening, the `UpsertRowEffect` reshape, EDICT011, and the sample-contracts cleanup land as one PR — they are mutually entangled (the constraints will not compile until `EdictUnit` implements the marker; the analyzer cannot run until the marker exists; the sample stops compiling until the row POCOs and state types implement it). The doc set (this ADR + CONTEXT.md + the resiliency-analysis strikethrough) lands in the same PR.

- **Test surface (per ADR 0016).** New `Edict.Analyzers.Tests` cases for EDICT011 — missing `[GenerateSerializer]`, missing `[Alias]`, `[Alias(nameof(X))]` (the rename-defeat case the analyzer's literal-only rule exists to catch), missing `[Id(n)]` on a public property, valid case (no diagnostic), inherited property case (analyzer scopes to declared properties only). New `Edict.Generators.Tests` cases asserting the generator emits both `[Alias]` and `[MessagePackObject(true)]` on the sibling partial for commands and events. New `Edict.Core.Tests` case asserting `UpsertRowExecutor` resolves the alias against the Orleans manifest and that a renamed row POCO with preserved `[Alias]` round-trips successfully — the proof the AQTN hole is actually closed. Existing `CommandWireShapeTests` and `EventWireShapeTests` should be unaffected (the wire bytes are identical; only who-wrote-the-attribute changed).
