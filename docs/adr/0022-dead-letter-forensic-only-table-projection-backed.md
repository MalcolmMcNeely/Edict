# Dead-letter, take two: forensic-only, table-projection-backed, no redrive

**Status:** accepted — supersedes ADR 0019; subordinate to ADR 0018 (the Outbox carries the dead-letter publish itself) and ADR 0015 (the dead-letter projection is just another Table Projection).

ADR 0019 closed the no-silent-loss gap by keeping a poisoned Outbox entry in a capped in-grain `DeadLetter` slice and **blocking aggregate intake** when that slice overflowed, with a grain-method `RedriveAsync` for recovery. In framework-consumer practice that stance has three sharp edges. A long downstream outage **stops the aggregate from accepting commands at all** — the back-pressure is on the producer side of the outage, where the producer cannot do anything about it. Every successful command pays the deserialization and write-amplification cost of carrying the slice (and the cap means the document can only grow within bounds, never shrink past zero). Inspection is **per-aggregate only** — there is no way to ask "show me everything that went wrong, fleet-wide", which is the question an operator triaging a system-wide failure actually asks first. We pivot to a **forensic-only** model: dead-lettering is observability, not back-pressure. When an Outbox entry exhausts `MaxAttempts`, the engine — **in the same one grain-state write** — removes the failed entry from the Outbox slice and appends a new `PublishEvent` Outbox entry carrying an `EdictDeadLetterRaised` event. A built-in singleton `EdictDeadLetterProjectionBuilder` (auto-wired by `AddEdict()`) consumes that stream and upserts to a fleet-wide table. A read-only `IEdictDeadLetterRepository` reads from that table with both per-aggregate and fleet-wide queries. There is **no cap**, **no in-grain slice**, and **no `RedriveAsync`**. Recovery for poisoned entries is **manual re-emission** — for `PublishEvent` / `SendCommand` the source can be re-invoked; for `UpsertRow` the operator repairs the row by hand, informed by the dead-letter projection. This is an honest downgrade from ADR 0019's no-silent-loss soundness story for the `UpsertRow` case (permanent-failure reopens the ADR-0012 double-apply gap, now closed only for the transient-failure case); the consumer has explicitly weighed that downgrade against the cost of a stallable aggregate and chosen the former.

The atomicity guarantee the Outbox exists for is preserved exactly as before: the failed-entry removal and the dead-letter-publish append are the same one grain-document write. The dead-letter publish itself is just another Outbox entry, so it inherits at-least-once retry semantics, trace continuity (ADR 0003 — the captured `traceparent` is propagated onto the projection row), and the framework's standard idempotency (the projection is an `EdictTableProjectionBuilder`, upsert by `EntryId`). The v1 projection is a **singleton grain** (one hot grain handling all fleet-wide dead letters) — a deliberate trade-off accepting hot-grain risk under mass-outage in exchange for cheap fleet-wide reads and a single `"deadletter"` partition. A fanned-out per-aggregate projection + global roll-up is a future framework upgrade that does not require consumer migration.

## Considered Options

- **Keep the block-intake-on-cap stance from ADR 0019** — rejected. The consumer's explicit position is that **dead letters must never stall the aggregate**; the framework's job is to record poison, not to use the producer as a relief valve for downstream outages. The cap also forces a grain-document size cost on every successful command, which the maintainer wants to avoid in pursuit of lightweight grains. The "nothing is ever silently dropped" argument that justified block-intake is preserved by a different mechanism (atomic move into the dead-letter publish entry), not by stalling intake.
- **Keep the per-aggregate-only inspection surface** (`IEdictDeadLetterRepository.ListAsync(grainKey)` only) — rejected. The operator's first triage question during a system-wide failure is "what's broken across the fleet?", not "what's broken on aggregate X?" — and the per-aggregate-only API forces the operator to enumerate aggregates before they can answer the fleet-wide question. The new repository surface keeps the per-aggregate query as `ListAsync(grainKey)` and adds `ListAllAsync()` as the fleet-wide partition scan.
- **Spill dead-letter overflow to an external store** (the old "external DLQ" option from ADR 0019, reconsidered) — still rejected, but for a different reason than before. ADR 0019 rejected it because the pending→dead-letter move would span two stores and reintroduce the gap the Outbox closes. With the new design the move is *still* one atomic grain-state write (the entry just becomes a `PublishEvent` Outbox entry), so the two-store argument no longer applies. We reject it now because the table-projection approach gives a familiar, queryable read surface the consumer already understands (ADR 0015), while an external store is a new transport with its own SDK, retry semantics, and operational story.
- **Per-aggregate projection + global roll-up at v1** — rejected for v1, kept as the upgrade path. The hot-grain risk under mass-outage is real but bounded by the fact that the dead-letter publish destination is the same Orleans stream provider every other event uses; if that bus is healthy the singleton can keep up, and if the bus is unhealthy the producer's aggregates are the larger concern. Shipping the singleton first lets the framework see real consumer load before paying the design cost of a roll-up layer.
- **Keep `RedriveAsync` as a grain-method recovery affordance** — rejected. `RedriveAsync` was load-bearing only for the `UpsertRow` case (the only effect kind the framework cannot trivially re-invoke from the source); for `PublishEvent` / `SendCommand` the source aggregate or saga is already addressable and the operator can re-invoke it directly. Keeping `RedriveAsync` for `UpsertRow` alone would be a one-effect-kind affordance with a per-grain inspection surface, which is the exact stance we are pivoting away from. The honest position is: `UpsertRow` permanent failure is operationally rare (the destination is the consumer's own table; failure modes are config faults or oversized rows), and manual table repair informed by the dead-letter projection is the accepted recovery path.
- **Per-effect-kind `MaxAttempts` / backoff configuration** — rejected for v1 (additive change later if a real consumer proves the need). One global `MaxAttempts` and one backoff curve via `EdictOutboxOptions` is simpler to reason about and matches the [[framework-configurability-direction]] of "knobs with sensible defaults", not "knobs per axis".
- **An OTel signal at dead-letter time** (metric/log/span) — out of scope. Carrying `traceparent` on the projection row gives the operator a jump-point from the dead-letter row to the originating distributed trace, which is sufficient for cross-reference; the projection table is itself a queryable observability surface.

## Consequences

- New ADR-0022 entry under `Edict.Contracts/DeadLetter/` (`EdictDeadLetterRaised`, `EdictDeadLetterEntry`, `IEdictDeadLetterRepository`); new `Edict.Core/DeadLetter/` modules (`DeadLetterPromotion`, `EdictDeadLetterProjectionBuilder`). The old `Edict.Core/DeadLetter/` types (`DeadLetterEntry`, `DeadLetterProjection`, `GrainDeadLetterRepository`) and the old `Edict.Core/Administration/IEdictDeadLetterAdmin` are deleted.
- `GrainEnvelope<TPayload>` shape becomes `{ Payload, Outbox, Idempotency }` — the `DeadLetter` slot is removed. The frozen `[Alias("GrainEnvelope" + backtick + "1")]` literal is preserved verbatim (ADR 0017 — state must survive class rename even if shape changes). Pre-1.0, no migration tooling; persisted documents carrying a populated `DeadLetter` slot lose it silently on first deserialization.
- `EdictDurableConsumerBase`'s `IEdictDeadLetterAdmin` implementation and the `EnsureIntakeNotBlocked()` guard are removed; aggregate intake is unconditional.
- `OutboxDrainEngine`'s max-attempts path changes from "move to `DeadLetter` slice" to "remove failed entry + append `EdictDeadLetterRaised` Outbox entry" — both mutations in the same one grain-state write. The promotion logic itself is a deep module (`DeadLetterPromotion`) in `Edict.Core/DeadLetter/`, pure and unit-testable, owning the effect-kind→`EffectTarget` mapping, JSON payload serialization (display data, distinct from MessagePack wire format per ADR 0007), and exception capture.
- `AddEdict()` auto-wires the dead-letter projection grain and repository. The Azure provider plugs the table-backed repository; `Edict.Testing` plugs the in-memory equivalent. The framework's no-silent-loss guarantee (for the transient-failure case) no longer depends on a manual DI call.
- Soundness downgrade: ADR 0018's claim that the Outbox closes the ADR-0012 double-apply gap stands **for the transient-failure case only**. Permanent `UpsertRow` failure now leaves the destination row missing and requires manual operator repair. CONTEXT.md's `Table Projection Builder` entry is updated to reflect this honestly.
- Recursive dead-letter publish failure (the dead-letter `PublishEvent` itself exhausting `MaxAttempts`) is a documented sharp edge: the recursion is bounded by the framework's `MaxAttempts` and in practice indicates a total event-bus outage, not a per-entry hazard. Not testing this scenario in v1.
- Test surface (per ADR 0016): `DeadLetterPromotion` is unit-tested in `Edict.Core.Tests`; `OutboxDrainEngine` promotion path adds cases to the existing engine test class; `TableBackedDeadLetterRepository` is tested in `Edict.Core.Tests` against the in-memory store and in `Edict.Azure.Tests` against Azurite via Testcontainers (the ADR-0016 provider-conformance slot); a single end-to-end `DeadLetterEndToEndTests` in `Edict.Core.Tests` proves the full loop through `Edict.Testing`. The previously shipped `DeadLetterCapTests` / `DeadLetterRedriveTests` / `DeadLetterRepository*` test classes are deleted along with their snapshots.
- ADR 0019 is marked `superseded by ADR-0022`. The memory entry `dead-letter-grain-backed-design` should be revised after the code-slice issues land.
