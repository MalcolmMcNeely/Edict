# Table-projection consistency

`EdictTableProjectionBuilder` stores its read model in Azure Table Storage (not grain state) so grain activation stays small no matter how large the read model grows — the dedup window nonetheless stays in **persisted** grain state (ADR 0002), with the window size **configurable** (ADR 0023) because a global-singleton projection consumes a far larger event firehose than the per-aggregate default. The projected row and the dedup window are two stores in principle, but atomicity is preserved by expressing the row write as a `UpsertRow` Outbox effect (ADR 0015) committed atomically with the window commit in the one grain-state write, then drained at-least-once (idempotent by partition/row key) — so a crash between commit and drain recovers effectively-once for the **inline-payload** case (the common case for normal-size events). For pointer-bearing claim-check envelopes (ADR 0020), ring commit happens at intake and the row write commits when the deferred entry executes — the same asymmetry ADR 0019 introduces for any deferred-dispatch path.
