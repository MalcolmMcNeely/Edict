# Chaos-default models at-least-once delivery in Edict.Testing

`Edict.Testing`'s in-process executor models the **at-least-once delivery contract the production substrate already imposes** — seeded duplicate redelivery and bounded per-subscriber reorder — so every multi-step test exercises the dedup ring (ADR 0002) and the reorder-tolerance contract (ADR 0015) as a side effect of asserting business behaviour. The contract is **on by default with no opt-out**: there is no `WithoutChaos` / `WithChaosSeed` builder method, no environment toggle, and no per-test escape hatch. Determinism comes from a **single `Seed` (`0xED1C7`) that derives multiple independent RNG streams via XOR constants** — `new Random(Seed)` for duplicate, `new Random(Seed ^ 0x5_EE_D5)` for reorder — so tuning one probability does not re-baseline tests gated by the other; **any future chaos behaviour must derive its RNG from `Seed` the same way** to preserve "one-seed-many-independent-streams". Reorder scope mirrors ADR 0015 line 7: **per-subscriber, across `HandleAsync` invocations on the same aggregate, on the retry path** — a `HeldQueue` keyed by `(GrainClass, RouteKey)` defers arrivals up to `MaxReorderDistance`, released through the same dispatch path on `Drain` settle. The `EdictEventHandler` carve-out (ADR 0019 default-OFF chaos for event handlers) **applies to duplicate redelivery only** — duplicate would baffle mock call-count assertions, but call counts are invariant under reorder, so reorder applies to event handlers too. Explicitly **out of scope** for this contract: no failure injection (no transient throws, no provider timeouts — that is `ControllableExecutor`'s internal seam, not consumer-visible), no global or cross-aggregate reorder (per-subscriber-per-aggregate only), no within-`HandleAsync` raise-order reorder (events raised in one `HandleAsync` publish in raise order per ADR 0015), and no time-based holds (release is `Drain`-triggered, never wall-clock — see `[[delay-not-useful-in-tests]]`).

## Considered Options

- **Opt-in chaos via builder method (`b.WithChaos(...)`)** — rejected: an opt-in default produces tests that pass under exactly-once-shaped assumptions and only fail in production; making the chaos contract the *floor* of every test is the whole point of the harness.
- **Per-test seed override (`b.WithChaosSeed(int)`)** — rejected: encourages flakiness-hunting by reseeding rather than fixing the consumer; the single default seed plus deterministic RNG derivation makes failures reproducible from the first run.
- **One RNG advanced across both behaviours** — rejected: tuning `DuplicateProbability` would re-baseline every test sensitive to reorder, defeating the determinism the contract promises.
- **Reorder across aggregates / globally** — rejected: ADR 0015's reorder-tolerance contract is per-subscriber-per-aggregate; modelling broader reorder in tests would assert a stricter contract than the framework actually offers.
- **Reorder within a single `HandleAsync`'s raised events** — rejected: ADR 0015 publishes raised events in insertion order on the happy path; reordering them in tests would make the harness *less* faithful to production, not more.
- **Time-based held-release (`hold for N ms`)** — rejected: wall-clock delays are never the answer (see `[[delay-not-useful-in-tests]]`); the release-on-settle path is deterministic and `FakeTimeProvider`-friendly.
- **Apply the `EdictEventHandler` carve-out to reorder as well as duplicate** — rejected: mock call-count assertions are invariant under reorder, so excluding handlers from reorder would weaken the contract without buying any consumer-side determinism.
- **Failure injection in the default chaos contract** — rejected: failure injection is a forensic seam (`ControllableExecutor`, internal-only) for proving the dead-letter and retry pathways from `Edict.Testing`'s own tests, not consumer surface; conflating the two would force consumers to author exception-handling for transients the production stack already absorbs.
