# Custom Kafka stream provider (`Edict.Kafka`)

**Status:** Draft. Sections 1 and 3 are complete; section 2 records the per-stream topology that the `#143` hardening slices have landed. Remaining `#143` content (claim-check mix-and-match, resilience suite, throughput parity vs the Azure baseline) is owed before this ADR is marked Accepted.

## 1. Why custom

`Edict.Kafka` ships a hand-authored Orleans `IQueueAdapter` / `IQueueAdapterReceiver` against `Confluent.Kafka` rather than adopting the OrleansContrib `Orleans.Streams.Kafka` community package, forking it, or pivoting to `Microsoft.Orleans.Streaming.EventHubs`. The community package has no .NET 10 / Orleans 10 build path and the maintainer has not signalled one; forking would copy a stalled codebase into Edict's release cadence and inherit its design choices forever, so it was ruled out pre-spike (#134 status note, 2026-05-26). Event Hubs was a viable retreat — Microsoft owns the adapter, the resilience suite would shrink, the fourth test layer below would not be needed — but rejecting it serves two concrete goals: a stronger portfolio surface (Kafka is the substrate operators actually expect from a CQRS framework, and `Edict.Kafka` is the credibility claim that Edict's pluggability is real, not Azure-shaped), and a production claim that survives the next non-Azure shop. The spike (`docs/spikes/0001-kafka-stream-provider.md`) returned GO with all five acceptance criteria plus the `MessagesDeliveredAsync`-after-`HandleAsync` pre-criterion green against a Testcontainers `confluentinc/cp-kafka` container in 27 s wall-clock, so the structural risk that would have forced PIVOT is retired. Bail-out triggers (1) "any happy-path conformance scenario fails" and (3) "adapter contract tests cannot be made deterministic" are cleared by the slices `#142` shipped on top of the spike (4/4 happy-path conformance scenarios green on Kafka × Postgres; 8 adapter contract tests green in <0.5 s without Docker); trigger (2) "performance" is owned by the throughput harness and carries over to `#139b` regardless. This ADR cross-references ADR-0024 for the new fourth test layer (§3) and ADR-0006/0007 for the wire-format discipline the custom adapter inherits — the Orleans `Serializer` hop routes `EdictEvent` subtypes through `AddEdictContractSerializer`, so no .NET type names land on the Kafka wire.

## 2. Topology

**One Kafka topic per `[EdictStream]` domain.** The mapping is `[EdictStream("OrdersStream")]` ⇒ Kafka topic `OrdersStream`, no shared "edict-events" central topic, no consumer-managed translation table. The producer side reads the stream namespace off Orleans' `StreamId` (which `PublishEventExecutor` stamps from `EdictStreamAttribute` via `EventStreamAddress.Resolve`) and writes to that topic; partition selection inside the topic is the stable FNV-1a hash of the stream key, so per-aggregate ordering is preserved by Kafka's per-partition ordering guarantee. The receiver side reads the topic out of the Orleans `QueueId`'s string prefix (`edict-kafka-<topic>`) and the partition out of its numeric id, so `CreateReceiver(QueueId)` knows where to subscribe without an out-of-band queue → topic map. One `IQueueAdapterReceiver` per (topic, partition) tuple holds a `Confluent.Kafka.IConsumer<string, byte[]>` with `_consumer.Assign(...)` (not `.Subscribe(...)`) so the per-partition ownership the framework asserts matches Orleans' queue ownership; consumer-group rebalance is owned by Orleans' ring balancer, not Kafka's group coordinator, which keeps multi-silo ownership deterministic.

**Per-stream partition count.** `EdictKafkaStreamsOptions.PartitionCount` is the fleet-wide default (32); `PartitionCountByStream` overrides on a per-`[EdictStream]` basis so a hot stream can carry more partitions without inflating the cold ones. The provisioner reads the per-stream value when ensuring topics at silo startup; the partition mapper reads it when enumerating the queue set. The two MUST agree — a topic created with N partitions and a mapper enumerating M queues would silently misroute on the (max(N,M) − min(N,M)) gap.

**Stream discovery.** `EdictKafkaStreamRegistry.FromAppDomain()` walks `AppDomain.CurrentDomain.GetAssemblies()` at silo-build time for `EdictEvent`-derived types carrying `[EdictStreamAttribute]` and returns the distinct stream names in sorted-ordinal order. Sorted order is the stability guarantee Orleans' consistent-ring queue balancer requires: two silos sharing the same assembly set MUST enumerate queues in the same order, otherwise the balancer's range-to-queue assignment diverges and partitions get owned by both silos at once. The contracts assemblies must be loaded by the AppDomain when `AddEdictKafkaStreams` runs — in practice, the serialiser-side `AddAssembly(...)` calls have already JIT-loaded them by then. Discovery is not generator-emitted; the assembly walk is the slice-4 simplification, and the test seam (`EdictKafkaStreamRegistry(IEnumerable<string>)`) lets fixtures bypass the scan with a hand-picked name list.

**Producer / consumer floors.** Producer: `acks=all`, idempotent producer on, lz4 compression, `MessageSendMaxRetries = int.MaxValue`. Consumer: `enable.auto.commit=false` with commit deferred to `MessagesDeliveredAsync` (after `HandleAsync` returns), `AutoOffsetReset.Latest` (Edict is event-driven, not event-sourced; ADR-0001). These floors are not consumer-tunable — `AddEdictKafkaStreams` validates the raw-config passthrough at wiring time and throws `InvalidOperationException` on any downgrade. No Kafka transactions: ADR-0002's dedup ring plus at-least-once delivery is the exactly-once-effect strategy.

**Replication factor.** Default 3 with provisioner-side auto-clamp to the available broker count when the option is left default; assigning `ReplicationFactor` explicitly (even back to 3) opts into strict mode and the provisioner throws on a broker-count mismatch. `MinInSyncReplicas` is derived (`max(1, RF − 1)`) so rf=3 ⇒ 2 and rf=1 ⇒ 1; no separate knob, the deriving rule is the floor.

**No schema registry.** Wire stability is the ADR-0006/0007 Verify-snapshot discipline on the contracts assembly's MessagePack encoding, not a registry hop on the broker. The custom adapter routes every event body through Orleans `Serializer` (which goes through `AddEdictContractSerializer`); no .NET type names land on the wire.

## 3. Testing implications — the fourth test layer

`Edict.Kafka` introduces a test layer that ADR-0024's three-layer scheme (unit / integration / load) did not anticipate: **adapter contract tests using in-process test doubles for `IConsumer<string, byte[]>`** that assert deterministic properties of the custom adapter's commit logic — no broker, no Orleans pulling agent, no timing window. Custom providers need this layer because the silent-failure surface (offset committed past an unacked record, partial-batch commits advancing past unacked tail, container from a foreign partition triggering a commit, batch-cache replay after rebalance) is observable end-to-end only as eventual data loss, and end-to-end resilience tests catch those failures only when timing happens to expose them. ADR-0024's three layers are insufficient on custom code because they were written when every Edict adapter was Microsoft-shipped, and Microsoft-shipped adapters get their contract tests from Microsoft; an adapter that *Edict* authors must own the same property guarantees in isolation. The shipped layer (`Edict.Kafka.Tests/AdapterContract/`, 8 facts) asserts: receiver assigns the right `TopicPartition` on `Initialize`; `GetQueueMessagesAsync` yields one `EdictKafkaBatchContainer` per `ConsumeResult` with matching `StreamId` / `Partition` / `Offset` / `Events`; `MessagesDeliveredAsync` commits max(offset)+1, never advances past unacked tail, no-ops on an empty batch, ignores containers from foreign partitions; never calling `MessagesDeliveredAsync` leaves the offset uncommitted (the deterministic shape of rebalance-redelivery); `Shutdown` closes and disposes the consumer. The layer applies *only* to custom-authored providers — `Edict.Azure`'s Azure Queue streams ride a Microsoft-shipped adapter and inherit Microsoft's contract guarantees. ADR-0024 stays **frozen** (not amended in place) per the repo's ADR-immutability convention: ADR-0024 was right for the world it was written in (every provider Microsoft-shipped); this ADR records what changed and adds the fourth layer, rather than rewriting history.

## Considered Options

- **Adopt `OrleansContrib/Orleans.Streams.Kafka` as-is** — rejected: no .NET 10 / Orleans 10 build, no maintainer signal that one is coming. Edict's release cadence cannot depend on an unmaintained upstream.
- **Fork `OrleansContrib/Orleans.Streams.Kafka`** — rejected pre-spike. Copying a stalled codebase into Edict's release cadence inherits its design choices forever (wire format, partition mapping, options shape) and pays the same forking tax as authoring fresh but with worse provenance and a less defensible architecture.
- **Pivot to `Microsoft.Orleans.Streaming.EventHubs`** — rejected for the primary direction. Microsoft owns the adapter (no fourth test layer, smaller resilience suite), but Event Hubs is Azure-only and the framework already has an Azure streams story (`Edict.Azure`). Shipping Event Hubs as Edict's "alternative" substrate would prove Edict's pluggability claim only inside Azure. Kept as the bail-out trigger for `#139b` performance failure or any future Kafka-specific show-stopper.
- **Adopt the spike's MessagePack-direct wire format** (`TypelessContractlessStandardResolver`, .NET type names embedded) — rejected: the production adapter goes through Orleans `Serializer`, which routes `EdictEvent` subtypes via `AddEdictContractSerializer` per ADR-0006/0007. The spike's format choice was scoped to "is the round-trip mechanical" and does not pre-commit production wire shape.
- **Skip the fourth test layer and rely on end-to-end resilience tests for the silent-failure surface** — rejected. The silent-failure modes are deterministic properties of the adapter's commit logic; end-to-end tests catch them only when timing happens to expose them, and a missed-commit regression that leaks past CI is the exact failure mode the ADR-0024 layering was designed to make impossible. Bail-out trigger (3) made this an explicit gate: had the contract layer turned out non-deterministic, the entire custom-adapter direction would have been abandoned.
- **Amend ADR-0024 in place to add the fourth layer** — rejected. The repo's ADR-immutability convention treats decisions as historical records; superseding additions go in the new ADR (this one), not by editing the prior one. Future readers of ADR-0024 see what was decided at the time; ADR-0028's §3 records what changed when a custom provider arrived.
- **Hide the full topology (producer/consumer floors, partition mapper, topic provisioning) inside this ADR even though it ships in `#139b`** — rejected. The topology surface is real today but the *defensible defaults* are tracer-bullet — `PartitionCount = 32`, hardcoded producer floors, single fixed `ConsumerGroupId` shape — and locking them in an ADR before `#139b` proves them at scale would commit Edict to choices the hardening slice may reverse. The §2 stub keeps the record honest about what is and is not decided.
