# Config surface and installation

The framework's installation surface is **three `ISiloBuilder` extensions, each accepting an `Action<T>` over a brand-prefixed options class**, with **all knobs surfaced as properties whose constructor defaults replace every literal in mechanism code**, and a **single host-start validator that fails fast on both missing-provider wiring and invalid option values**. The three calls partition the surface by what the consumer is choosing: `AddEdict(Action<EdictOptions>)` for core mechanisms (outbox, dead-letter projection, sender, dedup window — flat property list `IdempotencyWindowSize`, `OutboxBaseDelay`, `OutboxMaxDelay`, `OutboxMaxAttempts`, `OutboxJitterFraction`, `OutboxDrainReminderPeriod`); `AddEdictAzureStreams(Action<EdictAzureStreamsOptions>)` for the Azure stream-provider wiring plus the claim-check *threshold* (the wire-transport cap concern — `StreamProviderName`, `ClaimCheckThresholdBytes`, `QueuePollingPeriod`, `QueueServiceClient`); `AddEdictAzurePersistence(Action<EdictAzurePersistenceOptions>)` for Azure grain storage (Blob per ADR 0021), the reminder service, the dead-letter table, and the claim-check *blob store* (the storage concern — `GrainStateContainerName`, `ClaimCheckBlobContainerName`, `DeadLetterTableName`, `TableServiceClient`, `BlobServiceClient`). `EdictWiringValidator : IHostedService` resolves the framework's `IOptions<T>` instances at `StartAsync`, asserts both provider markers are present, and accumulates every problem into one aggregated `InvalidOperationException` — invalid values **throw** (no silent clamp on `JitterFraction`); the `IdempotencyWindowSize` global default is overridable per-grain-type via a `protected virtual WindowSize` so a high-firehose singleton can go larger without affecting silo-wide settings.

## Considered Options

- **`services.Configure<T>(IConfiguration)` as the primary entry shape** — rejected: assumes the consumer wants values from a configuration provider, but consumers reach for `Action<T>` even when they have `IConfiguration` (they mix literals and configured values); `Action<T>` composes with `services.Configure<T>(...)` under the standard Options machinery, so the binding case is supported without forcing it.
- **One front-door call (`AddEdict` covering core + streams + persistence in one options bag)** — rejected: the provider is genuinely a separate decision; a future Azure-streams-with-Postgres-persistence mix would make `o.GrainStateContainerName` ambiguous.
- **Five calls, one per mechanism** — rejected: outbox needs the dead-letter projection needs the table repo needs persistence; three calls aligned with the consumer's actual decisions is the right granularity.
- **`IServiceCollection` extensions everywhere** — rejected: the Orleans silo-builder calls only exist on `ISiloBuilder`; lifting Edict's calls to `ISiloBuilder` lets them wrap the Orleans calls internally, collapsing interleaved Orleans-and-Edict registration into three lambdas.
- **Silent clamping on `JitterFraction in [0, 1]`** — rejected: silent value mutation is itself a form of magic; "no magic numbers" cuts both ways.
- **Lazy throw on first use instead of `StartAsync` validation** — rejected: the failure window hides behind workload arrival; `IHostedService` fires before any grain serves a request, which is what the consumer wants when they typo a registration call.
