# Command validator precondition gate

Command validation is a framework layer that runs **server-side, inside the generator-emitted `Dispatch`, before `HandleAsync`, in the same Orleans activation turn** — the consumer authors a FluentValidation `AbstractValidator<TCommand>` (opt-in per command, resolved from grain DI, no-op when absent); current aggregate state arrives via `ValidationContext.RootContextData` under an Edict-defined key so validators stay stateless DI services. The validator may read state but **performs no mutation**; failures map to `EdictRejectionReason`s and return a `Rejected` `EdictCommandResult` — never thrown. Server-side placement makes the gate un-bypassable (the grain is the trust boundary, not the swappable `IEdictSender` seam) and, because validator and `HandleAsync` run in one single-threaded activation turn, the state the validator inspects cannot be raced before `HandleAsync` acts — a correctness guarantee client-side validation cannot offer.
