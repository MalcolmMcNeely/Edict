# Deferred dispatch (dedup ≠ effect completion)

`EdictEventHandler` is the consumer-facing base for stream-subscribed side-effect consumers (e.g. "send an email when `OrderPlaced` arrives"): it inherits `EdictIdempotencyBase<EdictUnit>` and the consumer's authoring shape is identical to a projection builder (`partial` class, one `Handle(TEvent)` per type), but the stream callback **stages an `InvokeHandler` Outbox entry instead of running `Handle` inline** — window slot and entry commit atomically in one write, and the Outbox drain (ADR 0015) re-invokes `Handle` afterwards with the same retry/backoff/dead-letter machinery the other effect kinds use. The base is **terminal** — it cannot `Raise` events (only `EdictCommandHandler` owns events) and cannot `Dispatch` commands (a saga's job per ADR 0016) — enforced structurally (those methods are not exposed on the base). External-API idempotency is the consumer's responsibility; the canonical mitigation is to pass `evt.EventId` (already a stable Guid in scope) as the external API's idempotency key (e.g. Stripe `Idempotency-Key`, SendGrid `X-Message-Id`).

The semantic this introduces, and that consumers must internalise, is that **dedup-window commit and effect completion are no longer the same moment** — for an inline projection-builder write they are synonymous, but for any deferred-dispatch path the window slot commits when the event is *accepted for processing* and the actual `Handle` runs later (possibly minutes later under backoff) and might dead-letter. The same asymmetry applies to **pointer-bearing claim-check envelopes on every consumer role** (ADR 0020): the stream observer commits the window slot at intake and stages an `InvokeHandler` entry that, at drain, fetches the blob and dispatches; a missing-blob fetch is treated as a transient error funnelling into the receiver-side dead-letter promotion path (a synthetic `EdictEventBlobMissing` notification preserves the no-silent-loss contract). The trade is intentional: the framework's existing retry/backoff/dead-letter machinery for free, at the cost of asking the consumer to mentally separate "accepted" from "completed".

## Considered Options

- **Direct stream callback (`EdictEventHandler : EdictIdempotencyBase` marker only)** — rejected: gives the consumer a renamed projection-builder and leaves retry/dead-letter to Orleans's stream-provider redelivery (sized for stream-internal failures, not external I/O).
- **Allow `Raise` from an event handler** — rejected: events express *domain facts an aggregate owns*; a stream-subscribed grain has no source-aggregate Guid to key a raised event with.
- **Allow `Dispatch` from an event handler** — rejected: that is what a saga is; two near-identical bases would confuse consumers picking the right one.
- **`HandlerOutcome { Success, Transient, Permanent }` or a `EdictPermanentFailureException` sentinel** — rejected: failure classification is a domain decision the consumer makes inside `Handle`; uniform throw-as-transient is consistent with the rest of the codebase. Additive later if a real consumer needs it.
- **Stage an `InvokeHandler` entry for every stream event** — rejected: would consume a dedup-window slot for unhandled event types; the generator emits a synchronous `HandlesType` gate so unhandled types remain a pure no-op.
- **Chaos default ON for event handlers in `Edict.Testing`** — rejected: side effects are observed in tests by external footprint (mock call counts); chaos-induced duplicates make those counts baffling on first contact. Default OFF for event handlers specifically.
