# In-grain row cache for table projections

`EdictTableProjectionBuilder<T>` caches the row it last touched in grain memory and skips `IEdictTableWriteStore.GetAsync` when the next event resolves to the same `(PartitionKey, RowKey)`; events on a different key fall through to today's `GetAsync ?? new T()`. The cache is safe because Orleans's `(grainType, primaryKey)` placement plus the projection's PartitionKey defaulting to the grain key make this grain the **single writer** to its rows — the in-memory copy is authoritative across events, and grain reactivation re-reads from the store after the activation drain has replayed any pending `UpsertRow` effects (ADR 0011). A single slot is deliberate over a bounded dictionary: per-aggregate projections (the binding constraint on the Events row of `Edict.Benchmarks.Throughput`) hit one row per grain and pay no GET after the first event, while many-rows-per-grain patterns (e.g. an `OrdersByCustomerProjection`, or the singleton dead-letter projection whose RowKey is the `EntryId`) miss on key change and behave as today — keyed-dictionary caching is a follow-up if a real bench surfaces that pattern. No `InvalidateCache()` hook: the single-writer invariant is taken at face value, in line with Edict's other "no escape hatch for misuse" choices.
