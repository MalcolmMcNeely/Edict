# Silo-local metrics cache for observable gauges

The three observable-gauge metrics (`edict.outbox.pending.count`, `edict.outbox.oldest_entry.age`, `edict.saga.progress.age`) read from an in-process, DI-registered singleton `EdictMetricsCache` rather than fanning out per-scrape to live grains. Every `OutboxHost` state mutation pushes `(grainType, grainKey, depth, oldestEntryEnqueuedAt)` into the cache on the grain task scheduler — same process, no RPC, one `ConcurrentDictionary` write per outbox commit. `EdictSaga` pushes its `(sagaType, sagaKey, lastEventHandledAt)` on every event-handle completion. Grain deactivation lifecycle removes the entry so a deactivated grain stops contributing to the gauge value. The `ObservableGauge` callback runs on the OTel scrape thread, iterates the cache, and emits one observation per grain type with per-type aggregation applied at read time — `sum(depth_i)` for `pending.count`, `max(now - enqueuedAt_i)` for `oldest_entry.age`, `max(now - lastHandledAt_i)` for `saga.progress.age`. The Q1 cardinality decision (per-type, not per-key) means the aggregation collapses the per-key cache entries into a small per-type result set before the OTel SDK ever sees them; cross-silo aggregation (sum across silos, max across silos) is the operator's PromQL responsibility, not the framework's.

Reading per-scrape from live grains is the alternative the PRD called out as quadratic, and it is: a scrape that fans out to every active grain costs O(grains) grain calls per scrape interval, on grains whose primary job is serving traffic. A polling silo-side aggregator (fixed-interval poll of each grain, cache the result, gauge reads cache) decouples scrape rate from sample rate but still pays the N grain calls per poll and bakes in poll-interval staleness — a freshly-stuck outbox waits up to one poll cycle to appear in metrics. The push-from-host model with a same-process cache pays zero RPC, the cache write is amortised into work the OutboxHost is already doing (the outbox commit), and the cache read at scrape time iterates a `ConcurrentDictionary` bounded by the number of *currently active* grains on this silo (concrete: 10k active grains × ~64-byte entry ≈ 640KB per silo, well under any reasonable memory budget). The grain-deactivation cleanup is load-bearing — without it, a grain that deactivates with `depth=5` continues to contribute 5 to the per-type sum forever, which is the same staleness disease that the polling aggregator solves more expensively. The cache lives behind an `IEdictMetricsCache` interface so `Edict.Testing` can swap it for an assertable in-memory variant (the same swap pattern the `IEdictSender` seam established); `MeterListener` already gives the test-side observation surface, but the swap lets tests assert on the cache state directly when needed. The cache instance is registered as a silo singleton via `AddEdict()` and resolved by every grain base that hosts outbox or saga state, so the wiring is invisible to the consumer.

## Considered Options

- **Scrape-time pull** — rejected. O(active-grains) grain calls per scrape on a substrate where grain calls go through the Orleans task scheduler; a 10k-grain silo scraped every 15 seconds drowns the grain scheduler in metric reads. The PRD itself flagged this shape as the dangerous one.
- **Polling silo-side aggregator** — rejected. Still pays N grain calls per poll, just at a slower cadence than scrape; introduces a poll-interval staleness window that's exactly the latency the metric is supposed to surface. The push model eliminates both costs.
- **Histogram per state-change instead of an observable gauge** — rejected for the age-shaped metrics. The age of the oldest pending entry grows continuously while the grain idles; sampling only on state-change misses the very condition (a stuck outbox) the metric exists to expose. Counter-as-depth-delta has analogous problems on grain restart (counter reset breaks the current-depth view PromQL would derive from `enqueued_total - dequeued_total`).
- **Distributed cache (Redis, the substrate's grain storage) instead of a process-local singleton** — rejected. Edict has no current cross-silo coordination requirement for metrics — Prometheus's `sum()` / `max()` across silo-scraped series gives the cluster view for free. Adding a distributed cache buys nothing and introduces a new failure mode (cache outage → metrics gap).
- **Push to OTel `Counter<int>` (`depth_delta`) on enqueue/dequeue and let the SDK accumulate** — rejected. Same depth-delta-on-restart problem as above plus the counter reset/recovery semantics that Prom rate functions handle but operator dashboards do not. The gauge-via-cache model lets the silo report its current truth on every scrape.
- **Make the cache write asynchronous (`Task.Run` or channel)** — rejected. The cache write is a single `ConcurrentDictionary` mutation; queueing it through a channel adds latency and a new failure mode (channel saturated) for no measurable benefit. Synchronous-same-thread is the simpler shape.
