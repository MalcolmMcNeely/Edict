# Saga model

A saga reacts to Events and issues Commands while holding durable progress: `EdictSaga<TProgress>` is the consumer-facing base, derived from the generic `EdictIdempotencyBase<TPayload>` — generic-ifying the idempotency root over `{ Payload, Outbox, Idempotency }` lets event handlers and projection builders close on `EdictUnit` while sagas carry typed progress, keeping ADR 0014's clause-(b) "all three inherit it" literally true. A saga issues **exactly one Command per Event** via `Dispatch(command)` — a single-command API that **throws if called twice in one handler**: a deliberate hard runtime limit, asymmetric with `EdictCommandHandler`'s buffering `Raise`, because a saga fanning out commands is a coordination smell and the API shape (no list, no buffer) makes the constraint unmissable. The dispatched command rides the Outbox's `SendCommand` effect (ADR 0015) so progress, dedup-window commit, and the buffered command all commit in one grain-state write.
