# Integration-first test layering, provider-suite mechanism battery, and shared-container parallelism

**Status:** accepted — supersedes ADR 0016 (provider-scoped test layering). Inverts ADR 0016's "Core.Tests = fast in-mem inner loop; Azure.Tests = full battery" thesis. Drops the "one test file per source class" rule from CLAUDE.md's Testing section. Establishes the rule that internal framework tests do not depend on `Edict.Testing` (the shipped consumer harness is proven via Sample app tests instead). The `edict-test-cluster-wiring` auto-memory is updated; CLAUDE.md's Testing section is rewritten in the same PR.

ADR 0016 split tests by *what they prove* but kept a cluster-based inner loop in `Edict.Core.Tests` against in-memory streams/storage so the dev loop stayed fast. In practice the inner-loop framing decayed: 13 subfolders accreted into `Edict.Core.Tests`, most boot a real `TestCluster` (`EdictClusterFixture`), and the mechanism battery effectively lives there — `Edict.Azure.Tests` shrank to a thin set of Azure-specific adapter tests rather than the conformance suite ADR 0016 envisioned. The duplication ADR 0016 set out to remove returned in a new shape: the same mechanism (dedup, outbox, saga, projections) is exercised once against in-memory infrastructure in Core.Tests and once against Azurite in Azure.Tests, the two copies drift, and neither is the load-bearing proof. Worse, the in-memory copy gives false confidence because it does not exercise the real transport — the exact failure ADR 0002's at-least-once realism story was meant to guard against.

Three additional facts forced the issue. First, three separate Azure cluster fixtures (`AzureClusterFixture`, `AzureUpsertRowRecoveryClusterFixture`, `EdictAzureSiloBuilderExtensionsClusterFixture`) each hold their Azurite connection string and service clients in **`static` fields** while xUnit's default parallelises across collections in the same assembly — a latent race that only "works" today because the collections happen never to run concurrently. Second, there is no resilience test in the repo that exercises a transport failure (Azurite paused, restarted, throttled) — the entire transport-fault category is untested. Third, there is no proof, even loose, that the framework holds up under any concurrent load — a dispatch of a thousand commands with invariants intact has never been demonstrated.

The principle this ADR pins down is that **`Edict.Core.Tests` carries only pure-logic unit tests; the mechanism battery, resilience scenarios, and load proof all live in `Edict.Azure.Tests` against Azurite via Testcontainers, with one shared Azurite container per assembly and per-test resource naming providing parallel isolation.** Pure-logic tests stay in Core.Tests because they are genuinely faster and no less revealing than the integration version; everything that wants a cluster gets a real one. The "fast inner loop" is no longer a goal of the test architecture — integration realism is.

`Edict.Core.Tests` is therefore trimmed to genuinely pure tests: the `OutboxSlice` state transitions, command-route discovery and resolution, the `EdictOptions` validator, the wiring inspector, the claim-check policy, the envelope codec, the claim-check unwrap helpers, the `InvokeHandlerExecutor` pure parts, the MessagePack round-trip and wire-shape Verify guards. `EdictClusterFixture` (the in-memory cluster) is deleted entirely — anything that needs a cluster needs Azurite, anything that does not need a cluster does not get one. The corresponding cluster-based test files (idempotency, outbox engine/host/recovery, saga workflows, dead-letter end-to-end, claim-check end-to-end, event-handler redelivery, command pipeline, projection and table-projection builders, event publishing, command telemetry) move into `Edict.Azure.Tests` against a real Azurite-backed cluster — lift-and-shift, because the existing tests encode hard-won traps (the frozen-`TimeProvider` stream stall, the all-zero `traceparent`, the orphaned command span) that a from-scratch rebuild would forget to test for.

`Edict.Contracts.Tests` is deleted in full. Its contents were reflection-against-record-shape tests asserting things the C# compiler or the type system already guarantees, plus a small set of public-API drift guards. The drift guards are not retained: widening commands, events, and attributes is acceptable as the framework evolves, and the only stability guarantee the framework offers in this area is the wire-shape Verify snapshots already living in `Edict.Core.Tests/Serialization/`. The cost of a project that exists to enumerate `IEdictClaimCheckStore`'s method names is not paid back by any regression it catches.

`Edict.Azure.Tests` becomes the integration-realism suite. Its folder layout is the **union** of `Edict.Azure/` and `Edict.Core/` source folders — `ClaimCheck/`, `Commands/`, `Configuration/`, `DeadLetter/`, `EventHandler/`, `Idempotency/`, `Outbox/`, `Projections/`, `Sagas/`, `TableStorage/` — so a contributor knows where a new test goes by reading the source tree. The ADR-0016 rule that each test file mirrors one source class is dropped: integration tests are inherently scenario-shaped (one test exercises many classes through a real transport), and forcing a per-class mapping either bloats one file or names tests after an arbitrary primary class. The replacement rule is: **folder mirrors source; filename names the scenario, not a source class; multiple test files per folder are normal.** Test method names continue under `Subject_Should{Outcome}[_When{Condition}]`.

Parallelism is reworked to match the new shape. One Azurite container is shared across the assembly; each test collection still spins its own `TestCluster` pointing at the shared Azurite because the cluster *shape* legitimately varies (visibility-timeout, controllable executors, custom stream provider configs). All `static` fields in the three fixtures become instance fields. Each test prefixes its table, queue, and blob container names with a `Guid` so collections may run in parallel without colliding on Azurite resources; resources are not cleaned up between tests (the container is thrown away at the end of the assembly run, which bounds the leak). An explicit `xunit.runner.json` documents `parallelizeAssembly: true, parallelizeTestCollections: true` rather than relying on the default. Tests within a collection still run serially (xUnit default), which keeps the per-cluster scenarios deterministic.

Resilience is scoped to two additions on top of lift-and-shift. The existing controllable-executor tests (which simulate crashes between commit and drain) carry over with the lift; their dedup, outbox-recovery, and saga-resume invariants are re-proven against the real transport. On top of those, **transport-fault tests** use Testcontainers' `PauseAsync` / `RestartAsync` to exercise scenarios the controllable executors cannot — Azurite paused mid-publish then resumed, Azurite unavailable at startup then catching up, Azurite restart mid-saga. One **silo-kill test** uses `TestCluster.StopSilo` / `RestartSilo` to prove the Orleans wiring (not just the gap-closure logic) survives a silo death mid-`HandleAsync`. Toxiproxy is explicitly out of scope: a single provider does not justify a sidecar container per test collection, and the conformance-harness era (when Kafka lands and a second provider exists) is the right time to introduce a network-fault-injection abstraction.

Load is scoped to a single liveness-and-invariants test: dispatch one thousand commands, with completion defined as the *downstream effect observed* (the published event consumed by a projection or handler) rather than the handler returning. The test asserts invariants — no lost events, every event observed exactly once, dedup ring bounded — and a deliberately generous wall-clock ceiling (~2 minutes) acting as a catastrophic-regression tripwire rather than a perf target. The test runs in every CI build; the loose ceiling is the cheapest defence against a 10× regression nobody notices.

Telemetry tests stay in `Edict.Telemetry.Tests` as today, against an in-memory cluster — span assertions are a different test shape (listener subscription, not transport I/O) and the parent-child stitch in `RequestContext` is transport-agnostic by design. **One** end-to-end span-stitch test moves into `Edict.Azure.Tests` to prove the real Azure Queue Storage transport preserves the trace headers across the hop; the targeted listener-shape tests stay where they are.

The shipped consumer test harness, `Edict.Testing`, is consumer-only. **No internal framework test project depends on `Edict.Testing`** — Core.Tests, Azure.Tests, Telemetry.Tests, and Architecture.Tests use Orleans `TestCluster` directly (with Azurite for Azure.Tests, in-memory for the others). The proving ground for `Edict.Testing` is the Sample app tests, where it is exercised the way an external consumer would use it. Conflating the two surfaces would create a circular trust problem (the framework's tests depending on a layer of the framework's shipped product) and tempt internal tests to skip real-transport scenarios because the in-memory harness "already proved it" — exactly the wrong move when the entire point of the integration suite is to exercise the real transport.

## Considered Options

- **Build the conformance harness now (an `Edict.Conformance` shared library with `IProviderFixture` + abstract scenario suites, Azure as the only fixture).** Rejected. Designing an abstraction against one example is the canonical way to design the wrong abstraction; the conformance-strategy doc itself flags Kafka as the load-bearing unknown and lists a timeboxed spike before committing. Defer the harness to when the second provider concretely exists and the genuinely-provider-agnostic seams are visible from two cases, not one.

- **Soft trim — delete the obvious cruft, keep cluster-based "fast inner loop" tests in Core.Tests *also* running against Azurite in Azure.Tests.** Rejected. This is the duplication ADR 0016 attempted and which decayed back into the current state: two copies that drift, neither load-bearing. The maintainer's stated preference is high-quality integration tests over many isolated ones; soft trim retains the low-quality copy "for speed".

- **Keep Core.Tests largely as-is and bulk Azure.Tests with resilience + load on top.** Rejected. Same shape as the status quo; the mechanism battery would still ride on in-memory infrastructure with Azure.Tests carrying only the extras. The whole reason to add resilience and load is that *those failures* are what production faces — gating them on a separate, smaller suite while the bulk of mechanism testing runs against fakes is the wrong centre of gravity.

- **Per-collection Azurite container (de-staticing fixtures but keeping today's pattern of one Azurite per collection-fixture).** Rejected. Fixes the static-field race but pays Azurite startup N times per assembly run; Azurite is not free to start and the cost grows linearly with collection count. A shared-Azurite + per-test Guid-prefixing pattern is the Testcontainers idiom and scales as more collection shapes accrete (saga workflows, load, future fixtures).

- **Disable collection-parallelism in the assembly (`xunit.runner.json` with `parallelizeTestCollections: false`).** Rejected. Cheapest fix for the race, but it lengthens CI noticeably and papers over a latent bug rather than fixing it. Future contributors would have to remember the rule to add new collections without losing parallelism.

- **Big-bang one-PR migration.** Rejected. The diff (delete Contracts.Tests, port ~25+ cluster-based files, refactor three fixtures, add `xunit.runner.json`, add transport-fault tests, add silo-kill, add load, add E2E telemetry) is too large to review meaningfully. Sliced PRs let the parallelism + fixture refactor land standalone and prove the new fixture model before the migration bets on it.

- **One ADR amending ADR 0016 in place.** Rejected. ADR 0016's core thesis ("Core.Tests = fast inner loop; Azure.Tests = full battery") is *inverted* here, not refined — Core.Tests is pure-only, Azure.Tests carries the battery, the in-memory cluster fixture dies. An amendment would rewrite most of 0016's text, paying the cost of a new ADR without the clarity. Superseding is the honest signal that this is a different decision.

- **Retain the public-API drift guards in `Edict.Contracts.Tests` even after deleting the rest.** Rejected. The drift guards exist to fail when someone widens `EdictCommand` or `EdictEvent`, but the maintainer's explicit position is that widening is acceptable. A test that fails on a deliberate, acceptable change is noise; the wire-shape Verify snapshots in `Edict.Core.Tests/Serialization/` remain as the only stability guarantee, and they fail on the things that genuinely matter (the persisted/wire byte shape changed) rather than on the public type's reflective surface.

- **Fold trace-stitch assertions into every Azure.Tests scenario.** Rejected. Maximum integration but overloads every test with span assertions, making each one harder to read and giving the listener-shape proofs a worse home than they have today. Keeping the targeted listener tests where they are + adding one E2E span-stitch test in Azure.Tests is the high-leverage hybrid: focused tests where focus matters, one real proof that the production transport preserves trace context.

- **Add a full §3b fault battery (Toxiproxy network fault proxy + container pause + silo kill + state-store transient failure injection).** Rejected for this scope. A single provider does not justify a per-collection Toxiproxy sidecar; the conformance-harness era (when the second provider is real and a network-fault abstraction has two concrete cases to shape against) is the right time. Container pause + one silo-kill test cover the failure modes the existing controllable executors cannot reach without dragging in Toxiproxy.

- **Track load as an SLO trend in a separate nightly job rather than a CI-blocking liveness test.** Rejected for this scope. The maintainer's framing is "some proof we can dispatch a thousand commands in a reasonable amount of time", not perf engineering — trend-tracking machinery is overkill for one provider. A liveness-and-invariants test with a generous wall-clock ceiling is the cheapest tripwire for catastrophic regressions, and is something the conformance-harness era can promote to a trend if and when that becomes warranted.

## Consequences

- **`Edict.Contracts.Tests` is deleted as a project.** Its csproj, files, and `.slnx` entry go. No replacement; the wire-shape Verify guards in `Edict.Core.Tests/Serialization/` remain the sole stability guarantee on the contracts surface.

- **`Edict.Core.Tests` is trimmed to pure-logic only.** Surviving tests: `OutboxSliceTests`, `RouteDiscoveryTests`, `OptionsValidatorTests`, `WiringInspectorTests`, `CommandRouteResolverTests`, `ClaimCheckPolicyTests`, `EnvelopeCodecTests`, `ClaimCheckUnwrapTests`, the pure parts of `InvokeHandlerExecutorTests`, the wire-shape Verify tests (`CommandWireShapeTests`, `EventWireShapeTests`, `EnvelopeWireShapeTests`, `EnvelopeStateShapeTests`, `DeadLetterContractWireShapeTests`, `CommandMessagePackRoundTrip`), and the `OutboxEffectKindFrozenOrdinalTests`. The duplicate `Saga/` vs `Sagas/` folder under `Edict.Core` is consolidated to `Sagas/` (matching the source) as a separate housekeeping concern.

- **`EdictClusterFixture` and its collection definition are deleted.** All test files that used it move to `Edict.Azure.Tests` against `AzureClusterFixture` or a refactored equivalent.

- **`Edict.Azure.Tests` folder layout becomes the union of `Edict.Azure/` and `Edict.Core/` source folders.** Existing top-level files (`DedupAtLeastOnceTests.cs`, `AzureEventHandlerRedeliveryTests.cs`, `UpsertRowGapClosureAzureTests.cs`, `TableProjectionBuilderAzureE2ETests.cs`, `GrainStateOnBlobSubstrateAtomicityTests.cs`, `EdictAzureSiloBuilderExtensionsTests.cs`, `AzureClientDIOverrideTests.cs`) move into the appropriate folders (`Idempotency/`, `EventHandler/`, `Outbox/`, `Projections/`, `TableStorage/`, `Configuration/`).

- **Lift-and-shift order**: Idempotency → Outbox → DeadLetter → ClaimCheck → EventHandler → Saga → Commands/Grains → ClaimCheck blob-missing. Each folder is its own PR; each PR ports tests, fixes Azurite-specific timing or wiring breakage, and deletes the Core copy. `EdictClusterFixture` deletion lands after the last folder.

- **Fixture refactor.** The three Azure fixtures (`AzureClusterFixture`, `AzureUpsertRowRecoveryClusterFixture`, `EdictAzureSiloBuilderExtensionsClusterFixture`) have all `static` fields converted to instance fields. Azurite is started once at the assembly level (an `IAssemblyFixture` or a shared `AsyncLazy<AzuriteContainer>` exposed to each collection fixture); each collection fixture spins its own `TestCluster` pointing at the shared Azurite. Each test names its table, queue, and blob container with a `Guid` prefix so collections do not collide.

- **`xunit.runner.json` is added** to `Edict.Azure.Tests` with `parallelizeAssembly: true` and `parallelizeTestCollections: true` set explicitly; documented in the file's adjacent README or as inline JSON5-style comments (xUnit's loader tolerates JSON5).

- **Transport-fault test category lands in `Edict.Azure.Tests/Resilience/`** (or distributed across the existing scenario folders if that reads cleaner). Initial tests: Azurite paused mid-publish then resumed; Azurite restarted mid-saga; Azurite unavailable at startup then becomes available. Mechanism: Testcontainers `PauseAsync` / `RestartAsync` on the shared Azurite (with one collection's lifecycle owning the disruption to avoid affecting other collections).

- **One silo-kill test** lands in the same area: `TestCluster.StopSilo` mid-`HandleAsync`, redelivery on restart, effect once.

- **Load test** lands as `Edict.Azure.Tests/Load/DispatchThroughputTests.cs` (or similar). Dispatches one thousand commands across many aggregates; completion defined as downstream effect observed; asserts invariants and a generous wall-clock ceiling. Runs in every CI build.

- **E2E telemetry test** lands in `Edict.Azure.Tests/Telemetry/` asserting the `Command → Publish → Handle` span tree survives the real Azure Queue Storage transport. `Edict.Telemetry.Tests` is otherwise unchanged.

- **Internal-tests-do-not-depend-on-Edict.Testing rule** holds today (a `grep` shows only doc-comment mentions) and is now codified: no test csproj outside Sample app tests may add a `<ProjectReference>` to `Edict.Testing`. No analyzer enforces this; the rule lives in CLAUDE.md and this ADR.

- **`Edict.Architecture.Tests`, `Edict.Generators.Tests`, `Edict.Analyzers.Tests` are unchanged** by this work.

- **CLAUDE.md is updated in the same PR as this ADR.** The Testing section is rewritten to reflect the inversion: Core.Tests = pure-only; Azure.Tests = mechanism battery + resilience + load; the file-per-class rule is dropped in favour of folder-mirrors-source-plus-scenario-named-files; the Edict.Testing prohibition is stated; the parallelism model is named.

- **Cross-cutting pruning principle applied during lift-and-shift.** For each test being ported, ask: *does this prove Edict's contract, or does it prove Orleans works?* Tests that exercise Orleans' own machinery without an Edict-specific invariant on top are deleted, not ported (`GrainStateOnBlobSubstrateAtomicityTests` is the explicit candidate for this review, but the question is asked of every file).

- **Slicing.**
  1. Parallelism + fixture refactor (shared Azurite, instance fields, `xunit.runner.json`) — no test changes, pure infra. Green before slice 2.
  2. Delete `Edict.Contracts.Tests`.
  3. Lift-and-shift folder-by-folder in the order above; one PR per folder; Core copy and in-mem fixture references deleted as each lands.
  4. Delete `EdictClusterFixture` once the last folder has been lifted.
  5. New categories (transport-fault, silo-kill, load, E2E telemetry) — one PR per category.
  - This ADR + the CLAUDE.md update + the duplicate-`Saga`/`Sagas` housekeeping land as slice 0, before slice 1, so subsequent PRs reference a written direction rather than chase a moving target.
