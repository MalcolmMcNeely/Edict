# Grain state on the blob substrate: removing the row-cap as a failure mode

**Status:** accepted — corrects the storage-provider choice the sample app encoded for `edict-state` since `Edict.Azure` was first wired (`AddAzureTableGrainStorage`). Narrows ADR 0024's three-surfaces motivation for claim-check to two surfaces (wire + dead-letter row); the grain-document surface this ADR retires is the (i) of that three-way argument. Does not amend ADR 0018 — the Outbox's atomicity claim ("one grain-state write is the unit") is satisfied on Blob storage by single-blob ETag-based optimistic concurrency, identical in shape to Table's row-level ETag.

The Edict Outbox commits its pending-effect list *inline in the grain-state document* (per ADR 0018) — that co-location is the atomicity. Under sustained downstream outage the `OutboxSlice.Pending` list grows by one entry per affected event, each entry carrying the serialised event payload, until the drain catches up or `MaxAttempts` exhaustion promotes the head to dead-letter. The previous sample app wired `edict-state` to `AddAzureTableGrainStorage`, which puts the entire `GrainEnvelope { Payload, Outbox, Idempotency }` into one Azure Table row — and Azure Table rows have **two hard caps**: 32 KB per property and 1 MB per row total. A handler raising N events per command, blocked behind a slow downstream during a 10 000 EPS burst, will reach the row cap **before** claim-check's per-event threshold can save it — because claim-check shrinks each event payload but not the per-entry overhead × N. Once the row overflows, the grain is hard-broken: every subsequent write returns an Azure Table error, the framework cannot persist the next state change or even the dead-letter promotion, and the operator's only recovery is a forensic out-of-band repair of the storage row. This is precisely the row-overflow risk identified in [docs/resiliency-analysis.md §3.2.2](../resiliency-analysis.md).

We move `edict-state` to **Azure Blob** storage, uniformly, across every Edict grain type — `EdictCommandHandler<TState>`, `EdictIdempotencyBase<TPayload>`, and the consumer-facing roles that derive from them (`EdictSaga`, `EdictEventHandler`, `EdictProjectionBuilder`, `EdictTableProjectionBuilder`, the framework-singleton `EdictDeadLetterProjectionBuilder`). The grain state lives on the blob substrate; the *read models* a Table Projection Builder writes via `UpsertRow` continue to live in Azure Table — those are consumer-shaped query surfaces, not framework persistence. A block-blob upload via `If-Match` ETag is atomic and single-writer-wins on the same primitive Table uses for row updates, so the Outbox's "one atomic write" guarantee is preserved by construction. Orleans's `AzureBlobGrainStorage` is the off-the-shelf provider; this ADR is a *wiring* and *substrate* decision, not a framework code change — `Edict.Core` is already storage-provider-agnostic behind the `[StorageProvider("edict-state")]` attribute on the grain bases.

The framework's **claim-check** mechanism (ADR 0024) is unaffected in outcome but its motivation narrows. ADR 0024 cited three independent surfaces that hit Azure Table's 32 KB per-property cap: (i) the Outbox-entry payload colocated in the grain document, (ii) the queue body during the stream hop, (iii) the dead-letter row's `PayloadJson` column. Surface (i) is **retired** by this ADR — Blob has no per-property structure. Surface (ii) remains and is the **primary motivation** for claim-check existing (the wire transport has a per-message cap that cannot be raised by changing storage substrate). Surface (iii) remains and is the **binding number** in the Azure provider's threshold default (the dead-letter projection's row stays in Azure Table, so its 32 KB per-property cap is what sets `ClaimCheckThresholdBytes = 30_720`). The threshold *value* is unchanged; ADR 0024's "Considered Options" rejection of "wire-hop only" still holds on the strength of (iii) alone — one mechanism still serves both the wire and the forensic poison-row case, which was the original "one mechanism, one store" promise.

`PubSubStore` — Orleans's stream pub-sub state — stays on Azure Table. It is Orleans-internal, bounded in shape (a small known set of subscriber records per stream), and not subject to the Outbox-growth dynamics that motivate this ADR. The rule is **"every Edict-defined grain has its state on Blob; Orleans-internal stores stay on whatever Orleans recommends."** Operator confusion is avoided by being explicit about the carve-out rather than universalising the rule beyond its actual scope.

This is **the** substrate choice the sample app should demonstrate. Consumers copying the sample's wiring into their own silo get the safer default automatically; consumers writing fresh silo wiring without consulting the sample find this ADR via the resiliency analysis they cross-reference when designing for production. Migration tooling is unnecessary: Edict is pre-shipped, no production deployments exist yet, and Orleans's two grain-storage providers do not share an on-disk format anyway — fresh deployments adopt the Blob substrate from the first activation.

## Considered Options

- **Keep Tables-default; rely on claim-check to keep individual entries small.** Rejected. Claim-check shrinks each event payload but not the *count* of pending entries. A handler raising one event per command at 100 deep entries × a few KB of entry framing each (envelope wrapper, MessagePack overhead, traceparent strings) approaches the row cap before any individual entry violates the 32 KB property cap. The failure mode is "row exhausted by many small entries," which claim-check does not address.
- **Per-grain-type knob — Blob only for the dead-letter singleton, Tables elsewhere.** Rejected. The Outbox-growth risk is uniform across every Edict grain type — every grain inherits the same `GrainEnvelope` and the same `OutboxSlice` growth dynamics. A per-type carve-out invites future "which grains went to Blob this time?" confusion. One uniform rule is simpler to operate and simpler to teach.
- **Build an intake-throttle / saturation metric on Tables instead.** Rejected as the primary fix. The intake throttle is *also* a real concern (resiliency-analysis §3.2.2's observability half) but it is a separate decision: an aggregate that has accumulated 200 pending entries is operationally suspect regardless of the row cap. This ADR resolves the *cap-overflow* half by removing the cap; the *observability* half remains open for a future ADR (saturation metrics, intake-throttle policy).
- **Split `ClaimCheckThresholdBytes` into two knobs (wire vs dead-letter row).** Rejected for v1. The framework currently has one stream-transport provider (Azure Queue) and one storage provider for dead-letter rows (Azure Table). There is nothing to compose yet; splitting the knob is speculative generality. When a second wire-transport or storage provider lands (Kafka, Postgres), revisit. Until then `ClaimCheckThresholdBytes` is one number set by `min(wire_cap, dead_letter_row_cap)` which the Azure provider computes internally as `30_720`.
- **Move `PubSubStore` to Blob too, for symmetry.** Rejected. `PubSubStore` is Orleans-internal, bounded in shape, and not subject to Outbox-growth dynamics. Moving it to Blob to match `edict-state` is uniformity for uniformity's sake — and may interact with Orleans's stream-implementation assumptions about its own pub-sub store. Leave Orleans's recommended substrate alone; document the carve-out.
- **Do not assert atomicity in a conformance test; trust Orleans.** Rejected. Orleans's `AzureBlobGrainStorage` has been production for years and the atomicity claim is sound, but resiliency-analysis §3.4.2 already calls out "the composition refactor (#69) might have moved this guarantee around — verify rather than assume." The same posture applies here: a single Azurite-backed test that pounds the substrate with concurrent writes is a tiny investment that converts "Orleans says so" into "Edict's CI proves so." See *Consequences*.
- **Build a sharded-grain-storage resolver into the framework now.** Rejected. The account-level transaction ceiling (~20 000 tx/sec per storage account, identical on Tables and Blobs) is the real upper bound; deployments at that scale shard across accounts. Sharding is a deployment concern independent of substrate, and there is no shipped deployment at that scale yet to design against. A future ADR may add a framework hook if a real workload hits the wall.

## Consequences

- **Sample wiring** (`Sample/Sample.Silo/Program.cs`, `Sample/Sample.Api.Tests/ApiFixture.cs`) swaps `AddAzureTableGrainStorage("edict-state", …)` for `AddAzureBlobGrainStorage("edict-state", …)`. `PubSubStore` keeps `AddAzureTableGrainStorage`.
- **`Sample.AppHost`** provisions an Azure Blob container for grain state (default name `edict-state`) alongside the existing claim-check container. Two distinct containers because the lifecycle policies differ — grain-state blobs are live data with no operator-set retention; claim-check blobs are append-only with operator-set retention.
- **`Edict.Azure.Tests`** gains an Azurite-backed conformance test asserting single-blob ETag-based atomicity under concurrent writes on the same grain key: spawn N concurrent writers attempting to mutate the same `GrainEnvelope`, assert the persisted envelope is exactly one of the writers' versions (no field-level mixing, no torn `Pending` list). The test uses Testcontainers/Azurite per ADR 0016 (provider-scoped — substrate behaviour is provider-specific).
- **ADR 0024** is amended: the three-surfaces rationale narrows to two surfaces (wire + dead-letter row). Surface (i) — the Outbox-entry payload colocated in the grain document — is removed from the motivation. The "Considered Options" rejection of "wire-hop only" still holds on (iii) alone. `EdictAzureOptions.ClaimCheckThresholdBytes = 30_720` is **unchanged as a value** — the binding number was always (iii), the dead-letter row's 32 KB per-property cap.
- **`CONTEXT.md`** — the **Outbox** entry gains one sentence noting the substrate: "The persisted document lives on the blob substrate (ADR 0025); the Outbox slice growth is bounded only by the wire-transport cap (via claim-check), not by a row-property cap."
- **`docs/resiliency-analysis.md` §3.2.2** is marked resolved for the *cap-overflow* half: the grain row cannot overflow because there is no row cap on Blob storage. The *observability* half (saturation metrics, intake-throttle policy) remains an open concern for a follow-up ADR.
- **Per-write latency.** Block-blob writes against Azure Blob are observably ~2× the latency of single-property entity writes against Azure Table in the small-payload regime (Azure storage public benchmarks; small entity ≤ 4 KB, single-region, P99 ~30–50 ms Table vs ~50–120 ms Blob). This is an acceptable cost for the cap-overflow elimination — and the **account-level transaction ceiling is identical** (~20 000 transactions/sec per storage account) on both substrates. Operators saturating the account ceiling shard across multiple storage accounts; that strategy is independent of the substrate choice and is an operator concern, not a framework concern.
- **No new exception types, no new public surfaces.** This is purely a configuration/wiring change at the consumer's silo + an AppHost provisioning change + one new test.
- **No migration tooling.** Edict is pre-shipped; no production deployments to migrate. Fresh deployments adopt the Blob substrate from the first activation.
- **Brand-rule clause (a) consumer-facing surface is unchanged.** No new types are exposed; `[StorageProvider("edict-state")]` is unchanged on the grain bases; the substrate choice is invisible to consumer code.
- **Slicing — one PR.** The change is too small to slice: sample silo wiring (~2 lines × 2 files), AppHost provisioning (~5 lines), one new conformance test, the doc amendments in this commit. Bundle as a single PR.
