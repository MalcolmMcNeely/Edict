# Test layering (integration-first)

`Edict.Core.Tests` carries **only pure-logic unit tests** (the `OutboxSlice` transitions, route discovery, options validators, the claim-check policy + envelope codec + unwrap helpers, the `InvokeHandlerExecutor` pure parts, the MessagePack round-trip and wire-shape Verify guards, the frozen-ordinal guards); the **mechanism battery, resilience scenarios, and load proof all live in `Edict.Azure.Tests` against Azurite via Testcontainers**, with one shared Azurite container per assembly and per-test `Guid` resource naming providing parallel isolation. The "fast inner loop" is no longer a goal of the test architecture — integration realism is; the in-memory `EdictClusterFixture` is deleted, and anything wanting a cluster gets a real one. `Edict.Azure.Tests`'s folder layout is the **union** of `Edict.Azure/` and `Edict.Core/` source folders so a contributor knows where a new test goes by reading the source tree; the per-source-class file rule is dropped in favour of folder-mirrors-source plus scenario-named filenames (integration tests are inherently scenario-shaped — one test exercises many classes through a real transport). Resilience adds two categories on top of lift-and-shift: **transport-fault tests** via Testcontainers `PauseAsync`/`RestartAsync` (Azurite paused mid-publish, restart mid-saga, unavailable at startup), and **one silo-kill test** via `TestCluster.StopSilo` mid-`HandleAsync`. Load is **one liveness-and-invariants test** — dispatch 1 000 commands with completion defined as the *downstream effect observed*, asserting invariants (no lost events, every event observed exactly once, dedup window bounded) and a deliberately generous wall-clock ceiling acting as a catastrophic-regression tripwire rather than a perf target. The shipped consumer harness `Edict.Testing` is **consumer-only**: no internal framework test project depends on it (proven via Sample app tests instead) — conflating the two surfaces tempts internal tests to skip real-transport scenarios because the in-memory harness "already proved it", the exact wrong move when the integration suite's whole point is to exercise the real transport.

## Considered Options

- **Soft trim — keep cluster-based "fast inner loop" tests in Core.Tests *also* running against Azurite in Azure.Tests** — rejected: this is the duplication a prior pass attempted, which decayed into two copies that drift, neither load-bearing.
- **Build a conformance harness now (`Edict.Conformance` shared library with abstract scenario suites, Azure as the only fixture)** — rejected: designing an abstraction against one example is the canonical way to design the wrong abstraction; defer until a second provider (likely Kafka) concretely exists and the genuinely-provider-agnostic seams are visible from two cases.
- **Per-collection Azurite container** (de-staticing fixtures but keeping today's pattern) — rejected: pays Azurite startup N times per assembly run; shared-Azurite + per-test `Guid`-prefixing is the Testcontainers idiom.
- **Disable collection-parallelism (`parallelizeTestCollections: false`)** — rejected: papers over a latent static-field race rather than fixing it.
- **Retain the public-API drift guards in `Edict.Contracts.Tests`** — rejected: widening commands/events/attributes is acceptable; the wire-shape Verify snapshots are the only stability guarantee that matters.
- **Toxiproxy network-fault sidecar** — rejected for this scope: a single provider does not justify a sidecar; revisit when the conformance-harness era lands.
