# OccurredAt is stamped at Raise() time, not drain time

Edict stamps `EdictEvent.OccurredAt` at the moment `Raise(theEvent)` is called inside the command handler's `Handle`, via the framework-injected `TimeProvider` — capturing the wall-clock instant the consumer's code decided the event happened. This **replaces** the prior contract, which stamped `OccurredAt = DateTimeOffset.UtcNow` inside `PublishEventExecutor` at drain time, *after* the entry had been buffered in the outbox and potentially queued under backoff for seconds-to-minutes. Drain time and intent time can diverge arbitrarily under transient stream-provider unavailability or burst backoff; for any consumer reading `OccurredAt` as the time the business event occurred (audit trails, latency analyses, regulator-facing reports — see `docs/trading-system-fit-analysis.md` §3.2.4), the prior contract was silently wrong under exactly the failure modes most worth recording. Stamping at `Raise` makes `OccurredAt` reflect intent — the moment the consumer's command handler decided to publish — and survives any subsequent outbox delay without drift. `EventId` and trace context (`TraceId`/`SpanId`/`TraceState`) remain drain-time-stamped: identity Guids carry no business semantic, and the publish span is correctly the parent of downstream spans regardless of how long the entry sat in the outbox. The framework gains no `DrainedAt` counterpart field — wire time is already the span start time recorded by `EdictDiagnostics.ActivitySource`, and duplicating it onto the event payload would inflate every event for an observability concern telemetry already covers. Grain turn discipline guarantees `TimeProvider.GetUtcNow()` at `Raise` is race-free against any other `Handle` invocation on the same activation; multiple `Raise` calls in one `Handle` get monotonically-increasing timestamps reflecting actual call order, preserving sub-`Handle` event ordering for audit consumers that care about it. The XML doc contracts on `EdictEvent.OccurredAt` and `EdictCommandHandler.Raise` are updated in the same change to advertise the new semantic.

## Considered Options

- **Stamp at `Handle` entry** — rejected: collapses every event raised in one turn to the same timestamp, losing sub-`Handle` ordering that an audit consumer would otherwise get for free; the simplification buys nothing the consumer wants.
- **Stamp at `EdictEvent` constructor** — rejected: unenforceable by the framework (a consumer can construct an event in a helper, store it, and `Raise` minutes later), and the consumer-controlled stamp duplicates the consumer-controlled `Raise` call without adding meaning.
- **Stamp at `Raise` *and* also at drain (twin fields `OccurredAt` / `DrainedAt`)** — rejected: wire time is observable in Aspire as the publish span's start time; adding a payload field for an observability concern that telemetry already covers inflates every event for no consumer-side query benefit.
- **Move `EventId` to `Raise` as well** — rejected: `EventId` is a fresh `Guid` identity with no business semantic on its generation time; moving it would conflate this change with the deferred sequence-number work in `docs/trading-system-fit-analysis.md` §3.2.3.
- **Use `DateTimeOffset.UtcNow` directly instead of `TimeProvider`** — rejected: the outbox backoff path already uses `TimeProvider`; using wall clock here would make `OccurredAt` non-testable and split the framework's time-source story.
