# Event Handler is a deferred-drain side-effect consumer

**Status:** accepted — subordinate to ADR 0018 (the Outbox carries the deferred invocation), ADR 0022 (permanent failure lands in the forensic dead-letter projection), ADR 0003 (trace context propagates onto the deferred invocation as parent-child). **Amended in part by ADR 0026:** the "ring commit ≠ effect completion" asymmetry recorded here as EdictEventHandler-specific is generalised to any deferred-dispatch path, which now includes pointer-bearing envelope delivery to projections and sagas. The contrast against Projection Builder narrows to *inline-payload* delivery specifically.

CONTEXT.md has named "Event Handler" as a domain role since the framework's inception ("a grain that subscribes to an event stream and reacts to events; does not own the events") but no consumer-facing base type has shipped. Today a consumer who wants the role rolls their own subclass of `EdictIdempotencyBase` — which works but loses the framework's failure-handling story for *external I/O*. The motivating use case is "send an email when `OrderPlaced` arrives": the consumer wants the dedup ring to suppress at-least-once redeliveries (as today), and *also* wants framework-managed retry/backoff and a queryable dead-letter row when the email API is permanently down. The naive shape — derive from `EdictIdempotencyBase` and call the email API inline from `Handle` — does the first but not the second: a transient SMTP failure relies on Orleans's stream-provider redelivery cadence (a coarse retry primitive sized for stream-internal failures, not external I/O), and a permanent failure just keeps redelivering until somebody notices.

We ship a new consumer-facing base `EdictEventHandler` whose mechanic is **deferred-drain via the existing Outbox** rather than inline-on-the-stream-callback. The consumer's authoring shape is identical to a `EdictProjectionBuilder` (a `partial` class, one `Handle(TEvent)` per event type, generator emits the dispatch); the runtime difference is that the stream callback **stages an `InvokeHandler` Outbox entry instead of running `Handle` directly**, commits the ring slot and that entry atomically in one grain-state write, and the Outbox drain re-invokes the consumer's `Handle` afterwards. The consumer never names "outbox" — the deferred-drain shape is purely a framework internal. The wire-up is symmetric with the existing `SendCommand` / `PublishEvent` / `UpsertRow` kinds: a fourth `OutboxEffectKind.InvokeHandler`, an executor that calls back into the host grain's dispatch, captured `traceparent`/`tracestate` so the deferred invocation nests under the publish span (ADR 0003), and the same ADR-0022 dead-letter promotion path on `MaxAttempts` exhaustion. The `EdictDeadLetterRaised` event is widened with two optional fields (`SourceEventType`, `SourceEventId`) populated only for `InvokeHandler` so an operator can query the dead-letter projection by event type and event id without parsing payload bytes.

The semantic this introduces, and which CONTEXT.md must document explicitly, is that **ring commit and effect completion are no longer the same moment**. For a `EdictProjectionBuilder` they are synonymous because the "effect" is grain-state mutation in the same write. For an `EdictEventHandler` the ring slot commits when the event is *accepted for processing*; the actual `Handle` runs later (possibly minutes later under exponential backoff) and *might dead-letter*. The dedup ring suppresses redelivery, but the side effect may never have happened. The trade is intentional: we get the framework's existing retry/backoff/dead-letter machinery for free, at the cost of asking the consumer to mentally separate "accepted" from "completed". The same trade has been live for `PublishEvent` and `SendCommand` since ADR 0018; this just extends it to the new consumer-visible role.

The base is **terminal**: an `EdictEventHandler` cannot `Raise` events (only an aggregate's `EdictCommandHandler` owns events) and cannot `Dispatch` commands (a saga's job per ADR 0020). Enforcement is structural — those methods are not exposed on the base — not analyzer-policed. A consumer who needs "react to event → do I/O → issue follow-up command" splits into an `EdictEventHandler` (the I/O leg) and a separate `EdictSaga` (the command leg), with the external system either pinging back into Edict via a Command or the saga reacting to a downstream event the I/O produced.

External-API idempotency is **the consumer's responsibility**. The framework guarantees exactly-once ring commitment and at-least-once handler invocation; if the consumer's `Handle` succeeds at the external API and the post-invocation Outbox-entry removal crashes, the drain re-invokes `Handle` on resume and the external system sees the call twice. The canonical mitigation is to pass `evt.EventId` (already a stable Guid in scope) as the external API's idempotency key (Stripe `Idempotency-Key`, SendGrid `X-Message-Id`, equivalent). The framework does not introduce a new affordance for this — `evt.EventId` is enough and documented as the pattern. Per-handler retry tuning is rejected for v1 (the same one-knob-set `EdictOutboxOptions` governs `InvokeHandler` as the other effect kinds); the [[framework-configurability-direction]] is "knobs with sensible defaults", not knobs per axis.

## Considered Options

- **Shape 1 — direct stream callback (a one-line `EdictEventHandler : EdictIdempotencyBase` marker).** Rejected. It would give the consumer a renamed `EdictProjectionBuilder` and nothing else: the *reason* an event-handler base exists separately from a projection-builder base is that side effects need a different failure-handling story, and Shape 1 leaves that story to Orleans's stream-provider redelivery — which is sized for stream-internal failures, not external I/O. Naming the role without giving it the mechanic the role needs is worse than not shipping the type at all.
- **Inline handler with a separate "outbox effect" the consumer types directly.** Rejected on the explicit maintainer position: outbox is a framework internal, never named in consumer code. The consumer's surface is `Handle(TEvent)`, the same shape as every other event-consuming role. Forcing the consumer to assemble an effect descriptor by hand would be the *worst* of both worlds — a leaky abstraction and a more verbose API for the canonical case.
- **`EdictSideEffect` as the name.** Rejected. CONTEXT.md has named this role "Event Handler" since inception; renaming on type creation would be a meaningful drift from the published domain language. "Side effect" is also overloaded in software (functional-programming pure-vs-side-effecting, "side effect" of any state mutation). "Event handler" is unambiguous in a CQRS framework and the deferred-drain mechanic is just *how* event handling works in Edict — part of what the term means here, not a separate concept that needs its own name.
- **Allow `Raise` from an event handler (Shape "C" — propagating role).** Rejected. Events express *domain facts an aggregate owns*; a stream-subscribed grain is not an aggregate and has no source-aggregate Guid to key a raised event with. Allowing it would either invent a fake source aggregate (a category error) or open the door to consumers building ad-hoc saga-like progress tracking without any of the saga's durability guarantees. A consumer who needs the shape uses an `EdictSaga`.
- **Allow `Dispatch` from an event handler (Shape "B" — saga-flavoured).** Rejected. This *is* a saga (react to event, issue one command, hold durable progress). Allowing it on `EdictEventHandler` would create two near-identical bases (saga and event handler with `Dispatch`), each with subtle progress-tracking differences, and consumers would pick the wrong one. Sharper to keep the roles disjoint: side effects in event handlers, command dispatch in sagas.
- **Sentinel `EdictPermanentFailureException` for skip-retries-and-dead-letter-immediately.** Rejected, on the maintainer's explicit position: failure classification (a 400 from SendGrid vs a 500) is a domain decision the consumer makes inside `Handle`, not a framework concern. The consumer who wants the affordance writes `try { ... } catch (BadInputException) { /* swallow as success */ }` or rethrows for transient retry. The framework treats every throw uniformly as transient → exp backoff → dead-letter at `MaxAttempts`. Adding the sentinel later if a real consumer proves the need is purely additive.
- **A `HandlerOutcome { Success, TransientFailure, PermanentFailure }` return type.** Rejected for the same reason as the sentinel, plus an API-churn cost: every existing `Handle` signature would change. Throwing remains the failure signal, consistent with the rest of the codebase ("exceptions across the grain boundary are reserved for infrastructure faults" — and external I/O failure sits on that side of the line).
- **No pre-flight type-check (stage an `InvokeHandler` entry for every stream event, let drain discover the no-overload case).** Rejected — this would consume a dedup-ring slot for unhandled event types and violate CONTEXT.md's explicit rule "an unhandled type is a pure no-op that consumes no dedup window slot and is never recorded as seen". The generator emits a synchronous `HandlesType(EdictEvent)` switch alongside the existing async `DispatchAsync` so the stream callback can no-op cheaply for events the subclass doesn't have a `Handle` overload for, leaving the ring window for events the handler actually handles.
- **Chaos default ON for event handlers in `Edict.Testing`.** Rejected. Side effects are observed in tests by their external footprint (a mock email sender's call count, an `HttpClient` request capture). Chaos-induced duplicate deliveries make those counts non-deterministic in a way that consumers writing their first `EdictEventHandler` test would find baffling — the test framework would teach the wrong lesson on first contact. Chaos defaults OFF for `EdictEventHandler` specifically; consumers who want at-least-once realism opt in explicitly. (Chaos remains ON by default for the other consumer roles, where it has earned its keep.)

## Consequences

- New `Edict.Core/EventHandler/` folder with `EdictEventHandler` (the consumer-facing base) and `InvokeHandlerExecutor` (internal, bare-named, implements `IOutboxEffectExecutor`). `EdictEventHandler` closes `EdictIdempotencyBase<TPayload>` on `EdictUnit` (no read-model, no progress state).
- `OutboxEffectKind` gains a fourth value, `InvokeHandler`. Persisted enum, frozen ordering — appended, not reordered. Pre-1.0 so no migration tooling; documents written before this change cannot carry the new kind and so are unaffected.
- `EdictDeadLetterRaised` gains two optional fields (`SourceEventType` string, `SourceEventId` Guid?), populated only for `InvokeHandler` promotions. Existing kinds leave them null. Frozen string-literal `[Alias]` preserved verbatim (ADR 0017). Schema bump on a pre-release projection — no consumer migration.
- The dead-letter projection row schema is widened in lockstep with `EdictDeadLetterRaised`. The fleet-wide partition strategy is unchanged.
- The generator's idempotent-consumer dispatch contract is split: `Task<bool> DispatchAsync(EdictEvent)` (runs the matching `Handle` — used at drain time) and a new synchronous `bool HandlesType(EdictEvent)` (used at stream-callback time to gate ring-slot consumption). The contract change is invisible to consumer code.
- `EdictEventHandler.OnStreamEventAsync` *overrides* the inherited behaviour: dedup-check, `HandlesType` pre-flight, build an `InvokeHandler` `OutboxEntry` (capturing the live `traceparent` so the deferred span nests parent-child per ADR 0003), and route the commit through `OutboxDrainEngine.EnqueueAndDrainAsync`. The base never invokes the consumer's `Handle` from the stream callback path.
- `InvokeHandlerExecutor` deserialises the entry's `EdictEvent` payload, restores trace context from the entry's `traceparent`, and calls back into the host grain's `DispatchAsync` to run `Handle`. The executor consumes the same retry/backoff machinery as the other kinds; permanent failure goes through the existing `DeadLetterPromotion` module, with the widened payload.
- `IOutboxHost` gains a callback for the deferred dispatch (one shape that the durable-consumer base implements as a route into `DispatchAsync`); the engine stays a plain unit-testable class.
- `AddEdict()` auto-wires `InvokeHandlerExecutor` in the executor registry alongside the existing three.
- `Edict.Testing` gains a new timeline kind `Invocation` representing one `EdictEventHandler` invocation (event arrived → stage → run or dead-letter). The chaos default for `EdictEventHandler` invocations is **OFF** (per the considered-options entry above); consumers opt in explicitly via the existing chaos surface.
- Sample app gains an `OrderEmailHandler` example reacting to `OrderPlaced` and logging to console (no real SMTP — the sample stays self-contained while riding the real Azurite-backed outbox per CONTEXT.md).
- Test surface (per ADR 0016):
  - `Edict.Core.Tests` — `InvokeHandlerExecutor` unit tests, `EdictEventHandler` stream-callback path (unhandled-type no-op, handled-type stage-then-drain), FIFO stop-at-head under transient failure of an earlier invocation, dead-letter promotion populating `SourceEventType`/`SourceEventId`, dedup interaction with re-delivery.
  - `Edict.Azure.Tests` — Azurite/Testcontainers conformance: real stream redelivery, crash-mid-drain reactivation, end-to-end dead-letter promotion landing in the real projection.
  - `Edict.Generators.Tests` — Verify-snapshot the dispatch surface (`HandlesType` + `DispatchAsync`) for a `partial class` deriving `EdictEventHandler`, including the cross-assembly-contract case (the [[generator-alias-cross-assembly-trap]] guard).
  - `Edict.Telemetry.Tests` — `ActivityListener` assertion that the deferred invocation span is a child of the publish span.
  - `Edict.Architecture.Tests` — placement guards (`EdictEventHandler` in `Edict.Core`, `InvokeHandlerExecutor` internal-and-bare, `OutboxEffectKind` value placement).
- CONTEXT.md updates (inline with the slice that lands the base type):
  - **Event Handler** entry rewritten to Shape 2 / terminal / ring-commit-vs-effect-done.
  - **Outbox** entry: four effect kinds (`PublishEvent`, `SendCommand`, `UpsertRow`, `InvokeHandler`).
  - **Idempotency Base** entry's family list adds `EdictEventHandler`.
  - Brand-rule clause-(b) shared-root list adds `EdictEventHandler` (consumer-facing surface).
  - New Flagged Ambiguity: "Event Handler was a named role with no shipped base type" → resolved by ADR 0023.
- The memory entry [[durable-consumer-unification]] should be refreshed after the engine slice lands to mention the four-kind outbox.
- Slicing — three PRs against the issue tracker:
  1. **Contracts-first:** widen `EdictDeadLetterRaised` (and dead-letter projection row) with `SourceEventType`/`SourceEventId`; add `OutboxEffectKind.InvokeHandler`; land ADR 0023 and the CONTEXT.md updates.
  2. **Engine + base type:** `InvokeHandlerExecutor`, `EdictEventHandler`, generator's `HandlesType` emit, `Edict.Core.Tests` + `Edict.Azure.Tests` + `Edict.Generators.Tests` + `Edict.Telemetry.Tests` + `Edict.Architecture.Tests`.
  3. **Sample + Testing:** `OrderEmailHandler` in the Sample, `Invocation` timeline kind in `Edict.Testing` with chaos-off default for event handlers, an end-to-end test through the in-memory test framework.
