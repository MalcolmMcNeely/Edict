# Substrate seam shared library

The harness-to-substrate seam — `ISubstrate` (a factory that brings a backend up) and `ISubstrateRuntime` (the live runtime that hands back `ConfigureSilo` / `ConfigureClient` callbacks plus an async dispose) — lives in `Edict.Substrate`, with one implementation per pairing in a sibling library (`Edict.Substrate.Azurite` today; `Edict.Substrate.KafkaPostgres` next). The throughput harness owned the seam while it was the only consumer; extracting it ahead of the conformance harness (the second consumer, per ADR-0027) commits both consumers to the same vocabulary and the same lifecycle contract — `Task<ISubstrateRuntime> StartAsync(CancellationToken)` for fixture bring-up, two `Action<...Builder>` properties that the harness's `TestClusterBuilder` configurators invoke against the runtime, and `IAsyncDisposable` for tear-down. The seam stays intentionally tiny: a substrate's job is "stand the backend up, hand back DI-wiring callbacks, tear down" — nothing about scenarios, polling, or workload shapes. **Workload-specific registrations stay in the harness, not the substrate** — the moved `AzuriteSubstrate` no longer registers `IEdictTableRepository<BenchEventRow>` (that was throughput-harness bleed); throughput's own `ActiveRuntime.ClientConfigurator` registers its workload repository after delegating to `runtime.ConfigureClient`, so the substrate library knows only about framework surfaces (`EdictDeadLetterEntry` repository, contract serializer, Azure clients, `AddEdictAzureStreams` / `AddEdictAzurePersistence`). The fixture/runtime split — substrate as factory, runtime as IAsyncDisposable carrier — mirrors the Testcontainers idiom and lets a per-test-collection `StartAsync` produce a long-lived runtime that's reused across scenario sweeps.

## Considered Options

- **Keep the seam in `Edict.Benchmarks.Throughput`, have `Edict.Tests.Conformance` reach into the harness project** — rejected: harness-imports-harness is the wrong direction and ADR-0024's "conformance harness deferred until a second example" trigger has fired; the seam needs to be a peer library both harnesses depend on.
- **Single `Edict.Substrate` library bundling abstractions + every provider impl** — rejected: pulls every substrate's SDK (Testcontainers.Azurite, future Confluent.Kafka, Npgsql) into both harnesses regardless of which substrates the run selects; the per-pairing sibling library is the canonical .NET shape and matches `Edict.Azure` / `Edict.Kafka` / `Edict.Postgres` for the provider packages.
- **Leave `AzuriteSubstrate`'s workload registrations in place (`IEdictTableRepository<BenchEventRow>`)** — rejected: a `Edict.Substrate.Azurite` library cannot reference `Edict.Benchmarks.Throughput.Workload` without a backwards dependency, and the next consumer (`Edict.Tests.Conformance`) has its own scenario row types — registering them in the substrate would force every harness's row types into every other harness's substrate.
- **Promote the seam into `Edict.Core` as a Testing-surface concept** — rejected: the seam is harness infrastructure, not consumer-facing; `Edict.Testing is consumer-only` (memory note) means consumer surfaces and harness surfaces stay separate.
- **Pass the workload registration in as a delegate parameter on `StartAsync`** — rejected: `StartAsync` is a backend-bring-up signal, not a wiring extension point; the existing `ConfigureSilo` / `ConfigureClient` actions already compose at the harness's own configurator layer, so no new parameter is needed.
