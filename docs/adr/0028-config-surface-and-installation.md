# Config surface and installation

**Status:** accepted — supersedes the pre-release `IServiceCollection`-based DI surface (`AddEdict()` + `AddEdictOutbox()` + `AddEdictAzureClaimCheck()`); extends ADR 0021 (the hand-authored front door now lives on `ISiloBuilder`); retires resiliency-analysis priority #5 (the drain-reminder 1-minute floor becomes tunable). Renames `Ring` → `Window` in code and prose to align with CONTEXT.md's already-canonical "dedup window" term (the persisted field `HandledEventIds` keeps its `[Alias]`-frozen name).

The framework's installation story today is three `IServiceCollection` extensions called in an order the consumer must learn (`silo.Services.AddEdict()`, `silo.Services.AddEdictAzureClaimCheck()`, `silo.Services.AddEdictOutbox()`) plus four Orleans silo-builder calls scattered alongside them (`silo.AddAzureBlobGrainStorage("edict-state", …)`, `silo.AddAzureTableGrainStorage("PubSubStore", …)`, `silo.UseAzureTableReminderService(…)`, `silo.AddAzureQueueStreams("edict", …)`). Several values that should be knobs are magic literals buried in mechanism code (`RingSize = 100` as a `protected virtual` override on `EdictIdempotencyBase`; the drain-reminder `TimeSpan.FromMinutes(1)` in `OutboxHost.RegisterDrainReminderAsync`; the stream provider name `"edict"`; the claim-check container name; the dead-letter table name). One option, `EdictOutboxOptions.JitterFraction`, documents itself as silently clamped to `[0, 1]` — the opposite of loud feedback. A host that forgets the claim-check store registration fails at the first oversized event, not at startup. A host that sets `MaxAttempts = 0` dead-letters every entry on its first failure with no startup signal.

The principle this ADR pins down is that **the framework's installation surface is three `ISiloBuilder` extensions, each accepting an `Action<T>` over a brand-prefixed options class, with all knobs surfaced as properties whose constructor defaults replace every literal in mechanism code, and a single host-start validator that fails fast on both missing-provider wiring and invalid option values.** The `Action<T>` shape (rather than `services.Configure<T>(IConfiguration)`) is deliberate: a consumer running on AKS with env-var overrides, a consumer running locally from `appsettings.json`, and a consumer running in a test with literal values all use the same call shape — the consumer chooses where the values come from inside the lambda, the framework does not assume a configuration source. The validator runs as an `IHostedService` at `StartAsync` so a misconfigured host fails before any grain serves traffic, with one aggregated error message listing every missing provider and every invalid option value.

The three calls partition the surface by what the consumer is choosing:

- **`AddEdict(Action<EdictOptions>)`** — the core mechanisms a consumer of any provider needs: outbox engine, dead-letter projection, sender, idempotency dedup window, claim-check policy threshold-decision seam. The options class is flat: `IdempotencyWindowSize`, `OutboxBaseDelay`, `OutboxMaxDelay`, `OutboxMaxAttempts`, `OutboxJitterFraction`, `OutboxDrainReminderPeriod`. Flat — not nested under `o.Outbox.MaxAttempts` — because a flat property list reads top-to-bottom and IntelliSense surfaces it without a category-discovery step.

- **`AddEdictAzureStreams(Action<EdictAzureStreamsOptions>)`** — the Azure stream-provider wiring (the Orleans `AddAzureQueueStreams` call folded in) plus the claim-check threshold policy. Options: `StreamProviderName`, `ClaimCheckThresholdBytes`, `QueuePollingPeriod`, `QueueServiceClient`. The claim-check *threshold* lives here because the cap is dictated by the wire transport (Azure Queue body + Azure Table dead-letter per-property cap), not by the grain-state substrate. A future Kafka provider exposes its own equivalent with a much higher threshold.

- **`AddEdictAzurePersistence(Action<EdictAzurePersistenceOptions>)`** — Azure grain storage (Blob for `edict-state`, Table for `PubSubStore`), the reminder service, the dead-letter table, and the claim-check *blob store*. Options: `GrainStateContainerName`, `ClaimCheckBlobContainerName`, `DeadLetterTableName`, `TableServiceClient`, `BlobServiceClient`. The claim-check *store* lives here because the bytes physically sit in Azure Blob storage — same family as grain state, same SDK, same connection-string story.

The split between "threshold lives with streams, store lives with persistence" is the load-bearing partition. It is the same mechanism (Claim Check) configured at two distinct seams because the *threshold* is a wire-cap concern and the *store* is a storage concern; collapsing them into one provider would force a consumer who mixes-and-matches (Azure persistence + a hypothetical Kafka streams) to either pick the wrong cap or hand-coordinate two providers. Splitting them keeps each provider call self-consistent — the streams call knows the cap it can serialise into; the persistence call knows the blob it can write to.

Service clients (`TableServiceClient`, `BlobServiceClient`, `QueueServiceClient`) go on the options bag so the sample reads top-to-bottom in one place. The provider extension internally prefers a DI-registered client when one is present (a power-user pattern: `AddAzureClients()` from the Azure SDK, shared connection across multiple features), falling back to the options-supplied instance otherwise. The simple path is the bag; the override is DI; either works and neither requires both.

The startup validator (`EdictWiringValidator : IHostedService`) does two things in `StartAsync`: (a) resolves `IEnumerable<IEdictWiringMarker>`, asserts a `StreamsProviderMarker` and a `PersistenceProviderMarker` are present, and accumulates missing-call descriptions if not; (b) resolves every registered `IOptions<T>` for the framework's options classes and triggers the `ValidateOnStart` validators on them. Both failure modes accumulate into one `InvalidOperationException` listing the full set, so a host with two problems sees two problems, not one then the other. Validation rules: `OutboxMaxAttempts >= 1`; `OutboxBaseDelay > TimeSpan.Zero` and `OutboxBaseDelay <= OutboxMaxDelay`; `OutboxJitterFraction in [0, 1]` (throws — no clamp); `OutboxDrainReminderPeriod >= TimeSpan.FromMinutes(1)` (Orleans' floor); `IdempotencyWindowSize >= 1`; `ClaimCheckThresholdBytes` in `(0, 30_720]` on the Azure provider (above 30 720 silently corrupts the dead-letter row's 32 KB per-property cap); `QueuePollingPeriod > TimeSpan.Zero`; non-empty name strings.

`IdempotencyWindowSize` is the global default; the per-grain-type `protected virtual int WindowSize` override on `EdictIdempotencyBase` stays. A singleton consumer at 5 000 EPS legitimately needs a much larger window than a per-aggregate handler at 1 EPS; the options value sets the silo-wide default, the override lets a specific consumer go larger or smaller. This preserves the per-type tunability while eliminating the unexplained `=> 100` magic.

The sample app (`Sample.Silo/Program.cs`) lists every option on its own line with the literal default value. The sample's job is to teach the surface, and a top-to-bottom catalogue of every knob does that better than an opaque `services.Configure<T>(config.GetSection("Edict"))` — a reader sees every knob the framework offers in the file they were going to read anyway. A consumer who wants the terser shape in their own app converts to `services.Configure<T>(...)` themselves; the framework supports both because `Action<T>` registration composes with later `services.Configure<T>(...)` calls under the standard Options post-configure order.

A new skill, `surface-config`, lands alongside this ADR. It triggers when an agent edits framework code in `Edict.Core` / `Edict.Azure` / `Edict.Contracts`, introduces a `TimeSpan` or `int` literal in mechanism code, or authors a new `Add*` extension. Its checklist enforces this ADR's principle at authorship time: every new knob is a property on an options class with a default in the constructor, a validation rule on the same class, and a line in `Sample.Silo/Program.cs` demonstrating the literal default. The skill is the in-the-moment nudge; this ADR is the durable record; CLAUDE.md carries a one-line summary so the principle is visible from turn one of every conversation.

## Considered Options

- **Status quo — three `IServiceCollection` calls, magic literals in mechanism code, silent `JitterFraction` clamp, no startup wiring check.** Rejected. The values most likely to need tuning under production load (drain reminder period, window size, max attempts) are the ones least visible: a `protected virtual` override, a hardcoded `TimeSpan.FromMinutes(1)`, an XML doc comment. The split between "consumer-typed" and "internal" is right but is fragmenting under growth — every new mechanism added a `services.Add*` call in the sample. The resiliency-analysis doc already flagged the reminder-period floor; "no magic numbers" is the principle that retires the flag.

- **`services.Configure<EdictOptions>(IConfiguration)` binding instead of `Action<T>`.** Rejected. Binding directly to `IConfiguration` assumes the consumer wants their option values to come from a configuration provider, but in practice consumers reach for `Action<T>` even when they have an `IConfiguration` available, because they want to mix literals (the threshold their team has agreed on) with configured values (the connection string from the host). `Action<T>` is the lowest common denominator: a consumer who wants the binding pattern can call `services.Configure<EdictOptions>(config.GetSection("Edict"))` themselves alongside or instead of our `Add*` lambda — the standard Options machinery composes both. Choosing `Action<T>` as the framework's primary entry point keeps the simple case simple and does not foreclose the binding case.

- **One front-door call (`AddEdict(o => …)`) that covers core, streams, and persistence in a single options object.** Rejected. The provider is genuinely a separate decision from the framework — a consumer might use Azure streams with Postgres persistence in a future version of the framework. Collapsing both providers into one options object makes that mix-and-match incoherent (which provider does `o.GrainStateContainerName` belong to?). The three-call shape names exactly the decisions a consumer makes (core knobs, which streams provider, which persistence provider), and the missing-call validator is the loudness that the collapsed shape can't offer.

- **Five calls — one per mechanism (`AddEdictOutbox`, `AddEdictIdempotency`, `AddEdictClaimCheck`, `AddEdictAzureBlobStore`, `AddEdictAzureTableStore`).** Rejected. The mechanisms are not independent enough to warrant a registration per mechanism — the outbox needs the dead-letter projection, the dead-letter projection needs the table repo, the table repo needs the persistence provider. A consumer asked to call five extensions in the right order before the host builds is fighting the framework. Three calls aligned with the three decisions (core, streams provider, persistence provider) is the granularity that matches the consumer's actual choices.

- **`IServiceCollection` extensions everywhere (current shape).** Rejected. The Orleans silo-builder calls (`AddAzureBlobGrainStorage`, `AddAzureQueueStreams`, `UseAzureTableReminderService`) only exist on `ISiloBuilder`, not `IServiceCollection`. Keeping our extensions on `IServiceCollection` means the consumer still writes the gnarly nested-Configure ceremony for streams (`configure.ConfigureAzureQueue(opt => opt.Configure(o => …))`) and the storage registrations remain a separate stack of `silo.Add*` calls. Lifting all three Edict calls to `ISiloBuilder` lets the streams call wrap `silo.AddAzureQueueStreams` internally and the persistence call wrap the three storage registrations — the sample shrinks from interleaved Orleans-and-Edict calls to three `silo.AddEdict*` Action lambdas.

- **Silent clamping on `JitterFraction in [0, 1]` (today's behaviour) rather than throwing.** Rejected. Silent value mutation is itself a form of magic — a consumer reading `o.OutboxJitterFraction = 1.5;` and getting `1.0` at runtime has no signal that their intent was changed. The "no magic numbers" principle cuts both ways: the framework neither hides a default behind a literal in mechanism code nor silently rewrites a value the consumer typed. `ValidateOnStart` throwing with a specific message is the loud-and-correct path.

- **Lazy throw on first use (sentinel `IEdictClaimCheckStore` that throws on resolve) instead of startup validation.** Rejected. The whole point of "loud on missing" is fast feedback. A sentinel that throws on first event publish only fires after the silo has accepted traffic — the failure window is hidden behind workload arrival. The `IHostedService.StartAsync` check fires before any grain serves a request, which is what the consumer wants when they typo a registration call. The lazy-throw path becomes dead code once the startup validator is in place; carrying both is two ways to fail at two timings, both needing to stay in sync.

- **Keep the `RingSize = 100` `protected virtual` override and skip the options surface for it.** Rejected. The `protected virtual` is the per-type tunability story and stays — but the *default* still needs to come from somewhere, and "100" with no documented rationale is the magic number the principle exists to eliminate. Sourcing the default from `EdictOptions.IdempotencyWindowSize` removes the literal from mechanism code; the override survives as the per-grain-type escape hatch. The two coexist: a consumer who doesn't override gets the silo-wide default the operator picked; a consumer who overrides picks their own.

## Consequences

- **`Edict.Contracts.Configuration/EdictOptions.cs` collapses the old `EdictOutboxOptions`** into a flat options class: `IdempotencyWindowSize` (default 100, replaces `RingSize` literal at `EdictIdempotencyBase.cs:72`), `OutboxBaseDelay` (2 s), `OutboxMaxDelay` (5 min), `OutboxMaxAttempts` (8), `OutboxJitterFraction` (0.2), `OutboxDrainReminderPeriod` (1 min, replaces literal at `OutboxHost.cs:246`). `EdictOutboxOptions` is deleted; pre-release, no migration shim.

- **`Edict.Azure/EdictAzureStreamsOptions.cs` and `Edict.Azure/EdictAzurePersistenceOptions.cs` are introduced.** `EdictAzureOptions` (current home of `ClaimCheckThresholdBytes` + `ClaimCheckContainerName`) is deleted and its two properties split: threshold to streams, container name to persistence. `StreamProviderName` (default `"edict"`), `QueuePollingPeriod` (default 500 ms), `QueueServiceClient`, and `ClaimCheckThresholdBytes` (default 30 720) on the streams options; `GrainStateContainerName` (default `"edict-state"`), `ClaimCheckBlobContainerName` (default `"edict-claim-check"`), `DeadLetterTableName` (default `"edict-dead-letter"`), `TableServiceClient`, `BlobServiceClient` on the persistence options.

- **Three `ISiloBuilder` extension methods land**: `AddEdict`, `AddEdictAzureStreams`, `AddEdictAzurePersistence` — replacing the four `IServiceCollection` extensions today (`AddEdict`, `AddEdictOutbox`, `AddEdictAzureClaimCheck`, plus the implicit silo-builder calls). `AddEdictOutbox` and `AddEdictAzureClaimCheck` are deleted; their wiring folds into `AddEdict` and `AddEdictAzureStreams`/`AddEdictAzurePersistence` respectively. `AddEdict` itself moves from `Edict.Core/EdictServiceCollectionExtensions.cs` to a new `EdictSiloBuilderExtensions.cs` (the `IServiceCollection` overload is dropped, not parallel-kept).

- **`Ring` → `Window` rename through code and prose.** `RingSize` → `WindowSize` on `EdictIdempotencyBase`; `EnsureRingInitialized` → `EnsureWindowInitialized`; xmldoc/comment references; CONTEXT.md prose (already done in this PR). The persisted field `IdempotencyState.HandledEventIds` keeps its name — it has a frozen-literal `[Alias]` and is the wire identity of the dedup-window storage slot.

- **`EdictWiringValidator : IHostedService` is registered by `AddEdict`.** Its `StartAsync` resolves `IEnumerable<IEdictWiringMarker>` and accumulates a missing-call list for any of `StreamsProviderMarker` / `PersistenceProviderMarker` not present; resolves every framework `IOptions<T>` and runs the `ValidateOnStart` validators; throws one `InvalidOperationException` whose message lists every problem found. The marker types live in `Edict.Contracts.Configuration` so provider packages can reference them without taking a dependency on `Edict.Core`.

- **`OptionsBuilder.Validate(...).ValidateOnStart()` is the validation seam** for every options class, used inside the `Add*` extensions. The validators are pure functions over the options instance (no DI access) so they run identically under `EdictTestApp` and a production silo. No `JitterFraction` clamp survives; the clamp comment is replaced by a validation rule.

- **Sample.Silo/Program.cs shrinks and grows in different directions.** Shrinks: the Orleans-and-Edict interleave (storage registrations + stream-provider configure + Edict DI calls) collapses to three `silo.AddEdict*` Action lambdas. Grows: every option is on its own line with its literal default value, so the file is the consumer's first-pass config catalogue. `var tableServiceClient = new TableServiceClient(...)` and the corresponding `AddSingleton` calls move *into* the persistence Action lambda (as the option-bag values), not as a separate DI stack — a power user who wants DI-registered clients still gets them (the extension prefers DI).

- **`surface-config` skill lands at `.claude/skills/surface-config/SKILL.md`.** Triggers: editing files under `Edict.Core/`, `Edict.Azure/`, `Edict.Contracts/`; introducing a `TimeSpan`/`int` literal in framework code; authoring a new `Add*` extension. Content: the five-step checklist (options property + default + validation rule + sample line + provider marker if applicable).

- **CLAUDE.md gains one line** under Conventions: "Every new framework knob is an options property with a default in the constructor, a `ValidateOnStart` rule, and a line in `Sample.Silo/Program.cs` — never a literal in mechanism code (ADR 0028)."

- **`docs/resiliency-analysis.md` priority #5** (reminder-table sizing + 1-minute floor) is updated: the tunable-floor half is resolved by `OutboxDrainReminderPeriod`; the sizing-assumption half (operator docs at fleet-wide outage scale) remains open and is now an operator-docs concern rather than a code concern.

- **No migration shim.** Pre-release, breaking changes are clean. Existing consumer code (Sample.Silo + tests) is updated in the same PR; the contracts assembly no longer ships `EdictOutboxOptions`; the Azure assembly no longer ships `EdictAzureOptions`; the `IServiceCollection` extension `AddEdict` is deleted in favour of the `ISiloBuilder` overload.

- **Slicing.** The whole surface lands in one PR: options class additions/deletions, `Ring` → `Window` rename across code + tests, three `ISiloBuilder` extensions replacing the four `IServiceCollection` extensions, validator service, sample-app rewrite, skill, CONTEXT.md edits (already in this PR), CLAUDE.md line. The pieces are tightly coupled (the rename touches the same files as the new options class; the validator depends on the marker types; the sample compiles only against the new extensions) and a staged landing would force a half-renamed intermediate state that is confusing to review.

- **Test surface (per ADR 0016).** `Edict.Core.Tests` covers the validator's missing-provider and bad-value paths against a minimal in-memory cluster, the `WindowSize` override per-grain-type, and the `OutboxDrainReminderPeriod` knob's effect on the reminder registration call. `Edict.Azure.Tests` covers the `Add*` extensions wiring real Azurite-backed storage/streams/blob through the same paths, asserting an `ISiloBuilder.AddEdictAzureStreams()` + `AddEdictAzurePersistence()` host boots and round-trips one command. `Edict.Architecture.Tests` gains a boundary test that no `TimeSpan`/`int` literal appears in mechanism code outside the options-class constructors (the literal-in-mechanism-code regression guard).
