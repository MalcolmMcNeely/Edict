# Claim Check: a universal Event Envelope with an off-wire payload escape hatch

**Status:** accepted — subordinate to ADR 0018 (commit-boundary claim-check piggy-backs on the Outbox flush), ADR 0022 (poison-row case lands in the forensic dead-letter projection), ADR 0003 (trace context survives the off-wire hop because the envelope carries it). Extends the dead-letter conceptual surface introduced by ADR 0019/0022 from "an Outbox effect failed at the producer" to also include "an inbound event could not be materialised at the consumer."

Edict's event hop crosses two infrastructure caps that the framework currently has no story for: Azure Queue Storage limits a queue message to roughly 48 KB of raw bytes, and Azure Table Storage — which backs both grain-state documents (per ADR 0018) and the shipped fleet-wide dead-letter projection (ADR 0022) — limits a single string property to 32 KB. An event whose MessagePack-serialised body crosses 32 KB starts to be hostile to the framework's persistence on three independent surfaces: (i) the Outbox-entry payload colocated in the grain document, (ii) the queue body during the stream hop, and (iii) the dead-letter row's payload column if that event ever permanently fails. The pragmatic mitigation is the well-known **claim-check** pattern: serialise the payload to a separate blob store, replace the in-band bytes with a small pointer, and fetch the pointer on the receiving side. CONTEXT.md is silent on this, and consumers who happen to raise a large event today get a deeply confusing Azure storage error from inside the Outbox commit instead of a designed framework outcome.

We ship one mechanism that serves both the wire-hop case and the poison-row case, sitting on a single appended-only blob store with **storage-policy-managed retention** (Azure Blob lifecycle rules), and exposing **one universal `EdictEventEnvelope`** as the wire-format wrapper on every event that crosses an Edict stream. The envelope is `EdictEvent`-derived, carries either an inline payload (`byte[]?`) or a claim-check pointer (`string?`), and is the framework-internal type the stream observer always sees on `OnNextAsync`. When the threshold is not exceeded, the inline-payload branch is taken and the envelope is a ~10-byte tax around the actual event bytes; when it is exceeded, the framework writes the body to a blob at commit time, replaces the buffered bytes with the pointer-bearing envelope, and the grain-state document, the queue message, and the dead-letter row downstream all carry the small pointer instead of the body. The receiver's stream observer unwraps the envelope before the consumer's `Handle` is dispatched, fetching from the blob store on the pointer-bearing branch, so consumer code is identical for in-band and claim-checked events. The dead-letter projection — and only it — is special-cased to **store the blob key in its row instead of the fetched payload**, preserving the forensic affordance (operator clicks through the key to the blob) without ever trying to materialise a >32 KB payload into a 32 KB property.

The same blob is referenced by both the wire hop and the dead-letter row, with retention owned by the storage account's lifecycle policy rather than the framework. Edict never deletes blobs. This is **Model B (append-only)** — chosen because the alternative (framework-managed delete-after-ack) requires reference counting across implicit subscribers and forces a *second* blob copy for the forensic dead-letter case, breaking the "one mechanism, one store" promise on first contact with the poison-row scenario. The append-only choice does *not* re-introduce event sourcing: there is no replay affordance, no `Rehydrate`, and no framework path that reads a blob outside the live-event hop or operator-led forensic inspection. Blobs are an **oversized-payload escape hatch**, not an event log; ADR 0001's "no event store, no replay" stays literally true.

The publish-time decision — serialise, measure, optionally write blob, wrap in envelope — happens **at the commit boundary**, not inside `Raise`. `Raise` remains the synchronous in-memory buffer call CONTEXT.md describes; the buffered events are flushed in one atomic moment alongside the Outbox-entry list and the grain-state write. This keeps every existing `Raise(evt)` call site untouched (no awaiting; no breaking API change), localises the entire claim-check policy in one pipeline step easy to reason about and easy to evolve, and allows the framework to parallelise the N blob writes for a Handle that raises N oversized events (`Task.WhenAll`) — something a per-call async `Raise` cannot do because each await is sequential. The measurement itself is exact and free: the serialised byte length is `bytes.Length` after MessagePack serialisation, which the commit pipeline computes anyway. There is no managed-memory size estimation, no reflection, no headroom guess for the *measurement*.

There **is** headroom in the threshold default — for envelope framing, not for measurement. The threshold gates the *event payload* size, but the Outbox entry stores the *envelope-wrapped* bytes. MessagePack framing for `EdictEventEnvelope` adds ~30–60 bytes around the payload, and the Read-3 universal-envelope decision (see Considered Options) reserves room for future framework wire-format fields. The Azure provider default is `ClaimCheckThresholdBytes = 30_720` (30 KB) — 2 KB of slack against the 32 KB per-property cap — and the commit pipeline asserts post-wrap that the entry payload is ≤ 32 KB, throwing a typed `EdictEnvelopeOverflowException` if it isn't. That converts an Azure Table 400-with-confusing-error into a designed framework failure surfaced cleanly to the operator.

The receiver-side failure mode for a missing blob is a small but real conceptual extension of the dead-letter surface. The stream observer fetches the blob before the consumer's `Handle` runs and before the dedup-ring slot commits, so a fetch failure has no good place to go in the existing model — Orleans's stream-internal retry is sized for transport blips, not for "the lifecycle policy ate this blob three weeks ago"; loud-but-wedging would block all events on a route key forever; silent skip is data loss the framework promised not to inflict. We treat fetch failure as a **delivery failure** that funnels into the **receiver-side** dead-letter promotion path: the Outbox machinery (which `EdictIdempotencyBase<TPayload>` already hosts post-#69, per the [[outbox-composition-refactor]]) treats a missing-blob fetch as a transient error, applies the exponential backoff already configured on `EdictOutboxOptions`, and on `MaxAttempts` exhaustion promotes a synthetic `EdictEventBlobMissing` failure into the same fleet-wide dead-letter projection that publisher-side failures use. The dedup-ring slot is committed on dead-lettering so the event does not redeliver. The operator's forensic view names the missing blob key, the route key, and the consumer's grain identity — enough to attempt restoration or, more usefully, to identify the lifecycle-policy misconfiguration that caused it. This is the conceptual extension worth ADR-marking: previously dead-letter meant "an outbound effect failed at the producer"; now it also means "an inbound event could not be materialised at the consumer."

Configuration sits in the provider package, not in `Edict.Core` — matching the [[framework-configurability-direction]] pattern already established by `EdictOutboxOptions`. `EdictAzureOptions.ClaimCheckThresholdBytes` defaults to 30 720 and is consumer-overridable; a future Kafka provider would set its default to `int.MaxValue` (effectively off — Kafka's natural limit is several orders of magnitude above what makes claim-check worthwhile, and a Kafka deployment using Azure Table grain storage still inherits the 32 KB per-property cap on the Outbox-entry side, which is the more restrictive bound on the publish path). The store seam — an `IEdictClaimCheckStore` with `PutAsync(ReadOnlyMemory<byte>) → string` and `GetAsync(string) → ReadOnlyMemory<byte>` — lives in `Edict.Contracts` as a framework-internal seam alongside `IEdictTableWriteStore`. No `DeleteAsync` (Model B forbids deletion), no `ExistsAsync` (a get-with-missing produces the dead-letter path; existence checks are redundant indirection that invite races), key generation is the store's responsibility (it knows its own naming conventions). The Azure implementation in `Edict.Azure` is an Azure Blob container with the lifecycle policy explicitly the operator's job to configure (the AppHost provisions the container; retention policy is documented but not framework-set).

## Considered Options

- **Two independent mechanisms, one for the wire-hop and one for the poison-row case.** Rejected. The two cases share a blob store and a serialisation step; coupling them into one mechanism with a special-case at the dead-letter projection (store the pointer in the row, don't fetch) gives the operator one place to look for oversized-event forensics. Two mechanisms force Kafka deployments to bring a blob store for the dead-letter case even when the wire-hop case doesn't apply to them, and force Azure Queue users to think about forensic retention when they only wanted to ship a big event.
- **Model A — framework-managed delete-after-ack blob lifecycle.** Rejected on the dead-letter scenario: when an oversized event is promoted to dead-letter, the wire-hop blob would already be deleted (or about to be) before the dead-letter projection could capture it, forcing a second forensic-store copy. That is precisely "two mechanisms," with a hand-off between them, and reference counting across implicit subscribers as the only alternative — both contradict the single-mechanism intent. Model B (append-only, lifecycle-policy-managed) is the only design that makes "one mechanism" hold for both cases.
- **Conditional-wrap envelope** (only oversized events get wrapped; small events ride the wire as themselves). Rejected after initially being chosen — discussed inline in the grilling session and overridden by the maintainer's preference for evolution headroom. The universal-envelope variant (Read 3) buys a permanent wire-format evolution point (encryption envelopes, schema-version tags, multi-tenancy hints) at a per-event cost of ~10 bytes and one extra deserialisation indirection. The `[[common-case-no-overhead]]` principle is set against this trade in the record: the conscious choice is that the envelope is *not* solely a claim-check vehicle, so the common-case tax is buying something more than the rare case.
- **Inside-Raise blob writes (`await Raise(evt)`).** Rejected. Makes the common-case `Raise` async for the exclusive benefit of making the rare-path blob write visible at the call site, and forecloses parallelisation of multiple oversized blob writes in one Handle (each await is sequential, vs `Task.WhenAll` at the commit boundary). The breaking API change is unjustified by the local-failure-visibility win — the same failure surface is available at the commit boundary as a typed exception, distinguishable from any other storage-write failure.
- **Drain-time blob writes (decision deferred to `PublishEventExecutor`).** Rejected on the grain-document-size argument: the Outbox entry would hold the full event bytes until drain, and a grain that raised three 80 KB events would overflow its own state document (32 KB per Azure Table property; ~1 MB total per row with splitting) before the queue is even touched. Raise-time / commit-boundary keeps the grain document small no matter how oversized the events that pass through it.
- **Marker event type (`EdictClaimCheckEnvelope`) only for the oversized case, with sniffing on receive.** Rejected. Sniffing serialised bytes to discriminate envelope-vs-event creates a hidden wire-format coupling that is fragile to MessagePack-Orleans-manifest changes. The universal envelope rides the existing Orleans type-manifest layer cleanly — the receiver sees `EdictEventEnvelope` as the concrete runtime type and the framework unwraps it before consumer dispatch.
- **`EdictDeadLetterProjectionBuilder` auto-fetches and stores the body.** Rejected. The dead-letter row's storage cap is *exactly* the reason claim-check exists; auto-fetching and writing the body to the row reintroduces the failure mode the design is trying to eliminate, with the additional bad outcome that the dead-letter projection silently truncates or rejects. The projection writes the pointer; the operator clicks through.
- **Receiver-side missing-blob → silent skip.** Rejected outright by the maintainer position "don't irreparably break their production system." Silent data loss in the face of a recoverable infrastructure misconfiguration is the opposite of what the framework should do.
- **Receiver-side missing-blob → throw and let Orleans retry forever.** Rejected. A misconfigured lifecycle policy wedges all events on the route key indefinitely with no operator surface beyond Orleans logs. The framework's existing dead-letter machinery is the better forensic destination.
- **Receiver-side missing-blob → block the grain with a typed exception, no dead-letter.** Rejected as too brittle — small operator errors (a deleted blob that the consumer doesn't need urgently) become outages. The configurable backoff already softens the transient case; permanent failure landing in dead-letter softens the operator-deletion case.
- **Threshold lives in central `EdictClaimCheckOptions` (not in the provider).** Rejected. The constraining cap is set by the publish-path storage stack — Azure Table per-property today, something else for a future Cosmos/Kafka pairing — and that knowledge lives in the provider package, not in `Edict.Core`. Centralising the knob would force the consumer to know their grain-storage backing's per-property cap, which the provider already encapsulates.
- **No envelope framing assertion at commit time.** Rejected. Without the post-wrap assertion, the failure mode for "envelope framing pushed us over 32 KB" is an Azure Table 400 with a confusing error from deep inside Orleans's storage layer. The assertion converts that to `EdictEnvelopeOverflowException` with the route key, event type, and measured byte length — operator-meaningful information at the moment of failure.
- **`ExistsAsync` / `DeleteAsync` on the store seam.** Rejected. `ExistsAsync` invites TOCTOU races (the answer can be stale between the check and the fetch); `DeleteAsync` violates Model B and would re-introduce all of Model A's lifecycle complexity through the back door of "what if we just deleted some blobs sometimes." The seam stays minimal.
- **Test framework defaults claim-check to off (`int.MaxValue` threshold) for ergonomic reasons.** Rejected. Consumer tests don't write claim-check tests directly — the mechanic is a framework internal — but the in-memory test framework should exercise the same code path that prod takes when an event happens to be large, so that a regression in the framework's commit pipeline surfaces in the framework's own test runs. The in-memory store mirrors the production threshold (30 720 default) and is overridable per test for the rare case a consumer wants to force the path with a small payload.

## Consequences

- New `Edict.Contracts/ClaimCheck/` folder with `IEdictClaimCheckStore` (internal-visible) and the typed `EdictEnvelopeOverflowException`. The seam parallels `IEdictTableWriteStore` (also internal, also bridging Contracts→Azure).
- New `EdictEventEnvelope : EdictEvent` in `Edict.Contracts/Events/` as the universal wire-format wrapper. Frozen string-literal `[Alias]` per ADR 0017 (persisted on the queue, so wire-format identity must survive class rename). Fields: `byte[]? InlinePayload`, `string? ClaimCheckKey`; inherits the trace fields and `EventId`/`OccurredAt` from `EdictEvent`. Exactly one of `InlinePayload`/`ClaimCheckKey` is set per instance — the framework asserts on construction.
- `EdictEvent` itself is unchanged. Consumers continue to derive concrete events from `EdictEvent`. `EdictEventEnvelope` is brand-prefixed under clause (a) (the framework receives it on streams the consumer subscribes to) even though no consumer code derives from it or applies it.
- `OutboxHost<TPayload>` (post-#69, [[outbox-composition-refactor]]) gains a commit-time claim-check pass: for each buffered event, serialise with MessagePack, measure, optionally `await store.PutAsync(bytes)`, wrap in `EdictEventEnvelope`, and assert the wrapped bytes ≤ 32 KB. Multiple oversized events parallelise via `Task.WhenAll`. The `IEdictClaimCheckStore` is DI-injected.
- `OutboxHost<TPayload>`'s stream-observer path (inherited from `EdictIdempotencyBase<TPayload>` post-#69) gains an unwrap step: when the incoming `EdictEvent` is an `EdictEventEnvelope`, fetch via the store (if `ClaimCheckKey` is set), deserialise the inner event, and continue dispatch with the materialised concrete event. Consumer `Handle(TEvent)` is never called with an envelope.
- `PublishEventExecutor` is unchanged structurally — the entry payload is already the envelope bytes by the time the executor runs; it deserialises to `EdictEvent` (which polymorphically materialises as `EdictEventEnvelope` for claim-checked entries), and Orleans's stream `OnNextAsync` carries the envelope onto the queue.
- A new receiver-side dead-letter promotion path: missing-blob fetch → transient → backoff → `MaxAttempts` → synthetic `EdictEventBlobMissing` promoted into the dead-letter projection via the same `IDeadLetterPromoter` the existing publisher-side path uses. `EdictDeadLetterRaised` is widened with two optional fields (`ClaimCheckKey` string, `FailureKind` enum extension `BlobMissing`); existing kinds leave them null. Frozen `[Alias]` preserved.
- `EdictDeadLetterProjectionBuilder`'s row schema gains a `ClaimCheckKey` column; the projection writes the key directly into the row instead of fetching and embedding the body. The framework's auto-fetch on receive is **suppressed** for the dead-letter projection specifically — the simplest implementation hooks the unwrap step on a per-consumer-type basis (the projection's type is known to the framework; one `if` in the unwrap path). Documented as a one-off framework special-case, comparable to the existing fleet-wide partition-on-`"deadletter"` constant.
- New `EdictAzureOptions.ClaimCheckThresholdBytes` (default 30 720) and `EdictAzureOptions.ClaimCheckContainerName` (default `"edict-claim-check"`). `Edict.Azure/ClaimCheck/AzureBlobClaimCheckStore.cs` is the production implementation. `Sample.AppHost` provisions the blob container against Azurite alongside the existing queues/tables wiring.
- `Edict.Testing/ClaimCheck/InMemoryClaimCheckStore.cs` is the in-memory store registered by the test framework. Threshold default mirrors the Azure provider (30 720) so test runs exercise the same commit path; per-test override available. No chaos for claim-check in v1 (missing-blob already has dead-letter exercises in `Edict.Azure.Tests`; chaos would duplicate that without adding consumer-meaningful surface).
- Telemetry: `edict.event.claim_check.put` child span inside the commit-time flush (only when claim-check fires), `edict.event.claim_check.get` child span inside the stream observer's unwrap (only when fetching). Tags on both: `edict.event.type`, `edict.event.size_bytes`, `edict.claim_check.key`. Boolean tag `edict.event.claim_checked = true` on the publish span itself so span-query "how many events claim-checked" is one filter. No span for the envelope wrap itself (pure CPU).
- Brand-rule clause (a) consumer-facing surface list gains `EdictEventEnvelope` and `EdictEnvelopeOverflowException`. `IEdictClaimCheckStore` stays internal (framework-internal seam, not consumer-facing).
- Test surface (per ADR 0016):
  - `Edict.Core.Tests` — commit-time wrap/unwrap, threshold boundary cases (just-under, just-over, with framing tax), `Task.WhenAll` for multiple oversized events in one Handle, `EdictEnvelopeOverflowException` on post-wrap overflow, in-memory store key round-trip.
  - `Edict.Azure.Tests` — Azurite Blob conformance: real put/get round-trip across a stream hop, dead-letter projection writing pointer not body, missing-blob → receiver-side dead-letter promotion end-to-end, framework-provided blob lifecycle policy assertion (the test asserts the framework *does not* delete; the operator's lifecycle policy is the only deletion path).
  - `Edict.Telemetry.Tests` — `ActivityListener` assertion that the claim-check spans nest correctly under publish and consumer-handle, with the `edict.event.claim_checked` tag set.
  - `Edict.Architecture.Tests` — `EdictEventEnvelope` placement in `Edict.Contracts`, `IEdictClaimCheckStore` in `Edict.Contracts`, `AzureBlobClaimCheckStore` in `Edict.Azure`, the in-memory store in `Edict.Testing`.
- CONTEXT.md updates (inline with the slice that lands the wire types):
  - New term **Event Envelope** documenting the universal wrapper and its role on every stream hop.
  - New term **Claim Check** documenting the off-wire payload escape hatch, the append-only blob lifecycle, and the dead-letter projection's pointer-preserving special case.
  - **Dead Letter** entry widened to document the receiver-side missing-blob failure mode.
  - **Outbox** entry's commit-boundary paragraph widened to mention the claim-check decision step.
  - Brand-rule clause (a) surface list adds `EdictEventEnvelope` and `EdictEnvelopeOverflowException`.
  - New Flagged Ambiguity: "Oversized events would corrupt the grain document / dead-letter row" → resolved by ADR 0024.
- Slicing — four PRs:
  1. **Contracts-first:** introduce `EdictEventEnvelope`, `IEdictClaimCheckStore`, `EdictEnvelopeOverflowException`; widen `EdictDeadLetterRaised` with `ClaimCheckKey`/`FailureKind=BlobMissing`; land ADR 0024 and the CONTEXT.md updates.
  2. **Publisher side:** commit-pipeline claim-check at the `OutboxHost<TPayload>` flush, in-memory store, Azure Blob production store, telemetry spans, `EdictAzureOptions` knobs, `Sample.AppHost` Azurite blob container, `Edict.Core.Tests` + `Edict.Azure.Tests` for the put path.
  3. **Receiver side:** stream-observer unwrap, missing-blob → receiver-side dead-letter promotion path, telemetry on the get span, `Edict.Core.Tests` + `Edict.Azure.Tests` for the fetch/missing-blob/dead-letter path.
  4. **Dead-letter projection pointer-preservation:** the unwrap special-case for `EdictDeadLetterProjectionBuilder`, the projection row schema widening, `IEdictDeadLetterRepository` surface unchanged externally (the `ClaimCheckKey` shows up on `EdictDeadLetterEntry`), end-to-end test through the in-memory test framework.
