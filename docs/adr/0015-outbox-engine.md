# Outbox engine

A command handler that mutates state and then publishes events has a crash window — Orleans offers no cross-store transaction, only a single grain-state document write — so Edict makes durable side effects **pending entries co-located in that one write**, drained at-least-once afterwards (consumer idempotency per ADR 0002 makes at-least-once effectively-once). The persisted envelope `GrainEnvelope<TPayload> { Payload, Outbox, Idempotency }` is one shape across `EdictCommandHandler<TState>` and `EdictIdempotencyBase<TPayload>`; the drain engine (composed as `OutboxHost<TPayload>` per the #69 composition refactor) supports four effect kinds — `PublishEvent`, `SendCommand` (saga, ADR 0016), `UpsertRow` (table-projection write, closing the ADR 0011 gap), and `InvokeHandler` (deferred dispatch, ADR 0019). Drain is **per-entry-independent retry, sequential in insertion order on the happy path** (so N events raised in one Handle publish in raise order when nothing fails), with exponential backoff per entry (`AttemptCount`, `NextAttemptUtc`) and a **lazy Orleans Reminder** as the crash-recovery net (registered when the slice is non-empty, unregistered on full drain, plus drain-on-activation — steady state holds zero reminders); on `MaxAttempts` exhaustion an entry is promoted to a dead-letter publish in the same one write (ADR 0018). The inline-drain-after-commit path preserves ADR 0003's parent-child span stitch — each entry carries the captured `traceparent` restored as the publish span's parent.

## Considered Options

- **FIFO stop-at-head per ADR 0018's original mechanic** — superseded: one poison effect blocked every subsequent effect on the same grain key for the full `MaxAttempts` window (~30–40 min), and the "per-aggregate causal order" benefit it preserved is already not honoured by the rest of the at-least-once stack (a redelivered older event lands after a newer one of the same aggregate; the window dedupes but ordering is lost). Consumers must already be reorder-tolerant.
- **Outbox as its own grain / its own store** — rejected: a second store is a second non-atomic write, reintroducing the very gap.
- **Reminder-only drain** — rejected: every event would eat the Orleans 1-min Reminder floor of latency.
- **Standing reminder per grain** — rejected: unbounded reminder table cost; lazy register-when-dirty scales to zero.
- **Parallelise drain via `Task.WhenAll`** — *initially rejected, subsequently adopted (2026-05).* The v1 assumption was that per-grain throughput was gated by `WriteStateAsync` latency, not executor I/O. The throughput benchmark (`docs/benchmarks/throughput.md`) refuted this: Commands → RaiseOnly was 642 → 83 EPS @ N=16, an 8× cliff explained by the per-entry queue PUT inside the executor, not the blob write. The drain now snapshots every ready entry (those past `NextAttemptUtc`), fires their executors concurrently via `Task.WhenAll`, then applies outcomes (`Ack` / `FailWithBackoff` / `Promote`) to the slice serially on the grain task scheduler — the slice stays a pure value, no cross-task contention, and the ack-batching / inline-failure-write semantics of v1 are preserved. Soundness rests on two existing invariants: executors are independent (no shared mutable state across kinds) and consumers are already reorder-tolerant (the dedup ring is keyed by `EventId`, not delivery order, per ADR 0001's at-least-once stack).
