# Edict

Edict is a CQRS, event-driven framework built on Microsoft Orleans. It provides reusable grain-based building blocks — command handlers, event handlers, sagas, projection builders — with implicit stream subscriptions, OpenTelemetry observability, and idempotent event delivery. Edict is **event-driven, not event-sourced**: there is no event store, no replay, and no rebuild from history.

## Naming convention (brand)

Edict is treated as a brand. A type carries the **`Edict` prefix** if and only if **(a)** a *consumer, or its generator-emitted code,* types it — derives from it, applies it as an attribute, or receives/returns it — **or (b)** it is an inheritance root shared by the consumer-facing grain bases. Internal infrastructure that satisfies neither stays unprefixed and descriptively named. This supersedes the earlier "no prefix" decision (the `Command`/`Event` bare names); ADR 0017 then dropped the `Grain` suffix ("Grain" is an Orleans implementation noun a consumer never names). Branded surface includes `EdictCommand`, `EdictEvent`, `EdictEventEnvelope`, `EdictCommandResult`, `EdictRejectionReason`, `EdictEnvelopeOverflowException`, `[EdictRouteKey]`, `[EdictStream]`, `[EdictTelemeterized]`, `[EdictRoutes]` (generator-applied on the consumer's assembly, ADR 0021), the grain bases (`EdictCommandHandler`, `EdictEventHandler`, `EdictProjectionBuilder`, `EdictTableProjectionBuilder`), the shared idempotency root `EdictIdempotencyBase` (clause b), `IEdictSender`, `IEdictTableRepository`, `IEdictPersistedState` (the marker every consumer-authored persisted-state type — aggregate state, saga progress, row POCO — implements; ADR 0027), and the options classes a consumer passes to the three `Add*` calls (`EdictOptions`, `EdictAzureStreamsOptions`, `EdictAzurePersistenceOptions`; ADR 0028). Consumer subclasses are named `{Name}{Role}` (`OrderCommandHandler`, `OrdersByStatusProjectionBuilder`). Internals (the sender implementation, command-route resolver, dedup state, the claim-check store seam) stay bare.
_Avoid_: bare `Command`/`Event` base names; the `Grain` suffix on any Edict abstraction or consumer subclass; prefixing internal types the consumer never references (the prefix must keep signalling "this is your contract with the framework"). Raw Orleans test doubles that genuinely derive from `Grain` keep "Grain" — they are not Edict abstractions.

## Language

**EdictCommand**:
An expression of intent to change state, addressed to exactly one grain via a **direct grain call** (not streamed). Handled by a Command Handler. Concrete commands derive from the abstract base class `EdictCommand` (carries `CommandId` only). A command does **not** carry trace-correlation fields: a direct grain call propagates `Activity` context natively, so trace fields live solely on `Event` (ADR 0003). Imperative naming (`PlaceOrder`, `UpdateOrder`) — past-tense names are Events.
_Avoid_: trace fields on `Command`, past-tense command names.

**Event**:
A notification that something happened, broadcast on a domain stream to zero or more subscribers. Transient — delivered, handled, then discarded. Never persisted to a replayable log. Concrete events derive from the abstract base class `Event` (carries `EventId`, `TraceId`, `SpanId`, `TraceState`, `OccurredAt`). A concrete event marks exactly one `Guid` property with `[RouteKey]` (the same attribute commands use); that Guid is the stream key the subscriber is activated with. An event's `[RouteKey]` is set by the consumer inside `Handle` and **may differ** from the command's `[RouteKey]` — re-keying is a first-class case (e.g. a `Sales` command raising an event consumed in the `Orders` domain).
_Avoid_: assuming the event key equals the command key; treating the command→event Guid as a guaranteed-continuous correlation id (it is the common case, not an invariant — trace context, not the Guid, is what reliably stitches the chain).

**Telemeterized**:
An attribute placed on a primitive property of a `Command`/`Event` subclass. A source generator emits code that writes the property as an OpenTelemetry tag (`edict.{type}.{property}`) on the active span. Placing it on a non-primitive is a **compile error**.
_Avoid_: runtime reflection, auto-tagging properties that are not annotated.

**Command Handler**:
The aggregate grain — Guid-keyed — that accepts Commands, performs the state change, and may raise Events. **The framework owns durable aggregate state**: the base is the generic `EdictCommandHandler<TState> where TState : IEdictPersistedState, new()`, whose persisted document is the single-write envelope `{ TState aggregate, Outbox, DeadLetter }`. `IEdictPersistedState` is the marker every consumer-authored persisted-state type implements (ADR 0027); the analyzer EDICT011 errors at the type's declaration if the consumer forgets `[GenerateSerializer]`, the frozen-string-literal `[Alias]`, or `[Id(n)]` on any public property. The consumer mutates `State` inside `Handle` and never persists fields by hand — the state change and the raised events commit atomically in that one write (this is what makes the Outbox sound). One grain *type per aggregate* handles *many* command types; the consumer writes one strongly-typed `Handle(TCommand)` method per command on a `partial` grain class and never authors the Orleans interface (the source generator emits it, the dispatch, telemetry, and the sender).
_Avoid_: holding aggregate state in plain grain fields (volatile — defeats the Outbox); a non-generic stateless `EdictCommandHandler` as the consumer base (every handler needs the envelope; a `EdictCommandHandler : EdictCommandHandler<Unit>` shim is the stateless convenience).

**RouteKey**:
The `[RouteKey]` attribute marking the single `Guid` property that addresses a message. One framework concept, two transports: on a **Command** it selects the aggregate grain (the grain key); on an **Event** it selects the stream key the subscriber is activated with. Exactly one per message; must be `Guid` (analyzer-enforced). The command key and the event key it raises *often* coincide (the in-domain case) but are **independent addressing concerns** — the event may carry a different Guid (the cross-domain re-keying case). The continuous command→event correlation is provided by trace context (ADR 0003), not by a guaranteed-equal Guid.
_Avoid_: `[Key]` (collides with `System.ComponentModel.DataAnnotations`), non-Guid keys, more than one per message, assuming the event key equals the command key.

**Command Result**:
The outcome envelope a Command Handler returns: `Accepted` or `Rejected` (with reasons). Rejection is a first-class *outcome*, never an exception (exceptions across the grain boundary are reserved for *infrastructure* faults — timeout, dead grain). Produced in two places that share this one envelope: a **Command Validator** (precondition failure) or `Handle` (transition-time outcome). Carries no domain data.
_Avoid_: returning domain payloads through a command; throwing for expected rejection.

**Command Validator**:
A server-side **precondition gate** for a Command, run inside the generated `Dispatch` *before* `Handle` and within the **same Orleans activation turn**, so the state it inspects cannot be raced before `Handle` acts. It may read the aggregate's current state but performs **no mutation**; it answers "is this Command admissible against the aggregate as it stands?" Authored by the consumer as a FluentValidation `AbstractValidator<TCommand>` (opt-in per command); current state is supplied via the validation context, never via a validator field. Failures map to `RejectionReason`s and yield a `Rejected` **Command Result**.
_Avoid_: mutating state in a validator; client-side validation; throwing for validation failure; expressing transition-time outcomes (only discoverable while mutating) as validator rules.

**Event Handler**:
A grain that subscribes (implicitly) to an Event stream and reacts to Events by performing external side effects (sending email, calling an HTTP API, writing to a non-Edict store). Base is `EdictEventHandler` (closes `EdictIdempotencyBase<TPayload>` on `EdictUnit` — no read-model, no progress state). The role is **terminal**: an Event Handler **cannot raise events** (only an aggregate's Command Handler owns events) and **cannot dispatch commands** (a saga's job); enforcement is structural (those methods are not exposed on the base), not analyzer-policed. Invocation is **deferred-drain via the Outbox** (ADR 0023): the stream callback dedup-checks, runs the generator-emitted synchronous `HandlesType` pre-check (so unhandled event types consume no dedup-window slot, preserving the rule below), stages an `InvokeHandler` Outbox entry, and commits the dedup-window slot and that entry atomically in one grain-state write; the Outbox drain re-invokes the consumer's `Handle` afterwards, with the framework's exponential backoff and dead-letter promotion at `MaxAttempts`. The consumer never names "Outbox" — the deferred-drain shape is a framework internal. **Dedup-window commit and effect completion are not the same moment** for any deferred-dispatch path (ADR 0026 generalises this from EdictEventHandler-only to also include pointer-bearing envelope delivery on every consumer role): the window slot commits when the event is *accepted for processing*; the actual `Handle` may run minutes later under backoff and may *permanently dead-letter*. The dedup window suppresses redelivery; the side effect may never have happened — observe completion via the dead-letter projection, not via window state. External-API idempotency is the consumer's responsibility: pass `evt.EventId` (a stable Guid already in scope) as the external API's idempotency key (Stripe `Idempotency-Key`, SendGrid `X-Message-Id`, equivalent) so the receiver dedups the at-least-once invocation.
_Avoid_: owning events; calling `Raise`/`Dispatch` from a handler (the base does not expose them — a consumer who needs the shape splits into an `EdictEventHandler` and a separate `EdictSaga`); treating dedup-window commitment as "the email was sent"; inlining external I/O on the stream callback by rolling your own `EdictIdempotencyBase` subclass instead of using `EdictEventHandler`.

**Saga**:
A grain that coordinates a multi-step workflow by reacting to Events and issuing Commands. Base is `EdictSaga<TProgress> where TProgress : IEdictPersistedState, new()`; it inherits the generic `EdictIdempotencyBase<TProgress>`, so its persisted document is the single-write envelope `{ DedupWindow, Outbox, DeadLetter, TProgress }` — dedup, outbound command, and workflow progress all commit atomically. EDICT011 enforces the persisted-state attribute set on `TProgress` exactly as on `TState` (ADR 0027). A saga reacts to an Event and issues **exactly one** Command per Event handled, via `Dispatch(command)` — a single-command API that **throws if called twice within one event**. This is a deliberate hard runtime limit and a deliberate asymmetry with the Command Handler's buffering `Raise` (events fan out; a saga that fans out commands is a coordination smell). Holds its own durable progress state; does not replay events to reconstruct it.
_Avoid_: dispatching more than one command per handled event; expecting `Dispatch` to buffer like `Raise`; reconstructing progress by replay.

**Projection Builder**:
A grain that consumes the live Event stream and maintains a current-state read model. The word "projection" is borrowed from event sourcing, but there is **no rebuild and no replay** — it only ever processes the live stream forward.
_Avoid_: implying replay, rehydration, or "rebuild the projection".

**Sender**:
The DI-injected `IEdictSender` with a single `Task<CommandResult> Send(Command)`. The source generator backs it: it reads the command's `[RouteKey]`, resolves the one owning aggregate grain, and dispatches. It is the substitution seam — `Edict.Testing` registers an in-memory implementation so consumer code is identical under test and in production.
_Avoid_: static/extension-method send (bypasses DI, defeats the in-memory test swap), per-command overloads.

**Domain Stream**:
A named Orleans stream that carries *all* event types for one domain (e.g. `Sales`, `Orders`) — streams are domain-scoped, not event-type-scoped. The name is declared once via a `[Stream("Name")]` attribute on the concrete event type — the single token both the publisher (`Raise` flush target) and every subscriber (generator-emitted `[ImplicitStreamSubscription]`) derive from. There is **no default**: omitting `[Stream]` is an analyzer error. An event may belong to a stream outside its producer's domain (cross-domain: a `Sales` handler raising an `Orders`-stream event). A subscriber to a domain stream is activated for *every* event type on it and only acts on the types it has a `Handle` overload for.
_Avoid_: per-event-type streams; inferring the stream name from the CLR namespace; a publisher and subscriber naming the stream independently.

**Table Projection Builder**:
A **Projection Builder** whose read model lives in an external composite-key store instead of grain state, so grain activation stays small no matter how large the read model grows. The mechanism is **persistence-agnostic** — Azure Table Storage is one implementation (shipped in `Edict.Azure`); a different backing store (e.g. a future `Edict.DynamoDB`) is another. The grain base `EdictTableProjectionBuilder<T> where T : class, IEdictPersistedState, new()` lives in `Edict.Core` and depends only on a framework-internal write store seam (a "dumb" `Get(pk,rk)` / `Upsert(pk,rk,row)`); the grain itself owns the per-event load→apply→writeback orchestration so it is identical across providers. The row is a consumer POCO that carries no storage type and no keys but **does** carry the persisted-state contract: `[GenerateSerializer]`, frozen-literal `[Alias]`, `[Id(n)]` on every property (analyzer-enforced via EDICT011, ADR 0027). PartitionKey/RowKey travel through the seam as parameters (the grain computes pk from its primary key, rk via `GetRowKey`). The application reads the result only through the read-only **Table Repository** — never the grain. The row write is expressed as an Outbox `UpsertRow` effect committed atomically with the dedup-window commit in one grain-state write, then drained at-least-once (the upsert is idempotent by pk/rk). The effect carries the row POCO's frozen `[Alias]` string (not its assembly-qualified name); the drain resolves it via `Orleans.Serialization.TypeConverter.Parse` so a class rename or namespace move does not dead-letter in-flight effects (ADR 0027 closes the former AQTN hole). This closes the former double-apply gap **for the transient-failure case only**, and **for inline-payload envelope delivery only** (ADR 0026 — pointer-bearing envelope delivery to a projection stages an `InvokeHandler` entry whose dedup-window commit happens at intake and whose row write commits when the entry executes, so window-equals-row atomicity does not hold for that case; pointer-bearing-to-projection means oversized events, which are operationally rare and already in tension with the destination's 32 KB per-property cap). ADR 0022 removed `RedriveAsync`, so a *permanently* failing `UpsertRow` (max attempts exhausted) leaves the destination row missing and reopens the gap for that case. Recovery is **manual operator repair** — informed by the dead-letter projection, which carries the failed effect's table/pk/rk in `EffectTarget` and the JSON payload — not a framework-driven redrive. In practice `UpsertRow` permanent failure on a healthy system is operationally rare (the destination is the consumer's own table; failure modes are config faults or oversized rows).
_Avoid_: reading the store directly instead of via the Table Repository; putting the read model in grain state "to be safe"; treating "Table" as Azure-specific (it is the generic composite-key-store mechanism) or putting an `ITableEntity`/storage type on the row; assuming the row write is a direct synchronous store call (it is an Outbox effect); assuming the Outbox closes the double-apply gap for *permanent* `UpsertRow` failure (it does not — see **Dead Letter**).

**Table Repository**:
The framework-provided **read-only**, persistence-neutral interface (`IEdictTableRepository`, in `Edict.Contracts`) the application uses to read a **Table Projection Builder**'s output: point-get by (PartitionKey, RowKey) and a partition-scoped query. It is a distinct seam from the framework-internal *write* store the grain uses — the application never writes, and never depends on the write seam. The Azure implementation lives in `Edict.Azure`.

**Outbox**:
The single durable-delivery engine. A composed internal component (folder `Outbox/`, bare-named — no consumer types it) owned by both grain roots (`EdictCommandHandler<TState>` and `EdictIdempotencyBase<TPayload>`); it never gets its own grain (a separate grain would be a separate store, reintroducing the gap it exists to close). An **outbox entry** is a durable *pending effect* recorded **in the same single grain-state write** as the consumer payload and the dedup-window slot — the persisted document is the unified envelope `GrainEnvelope<TPayload> { Payload, Outbox, Idempotency }` (Command Handlers simply never touch the `Idempotency` slot). That co-location is the atomicity, not a distributed transaction (Orleans gives no cross-store transaction; one grain-document write is the unit). The grain document itself lives on the **blob substrate** (ADR 0025) — Outbox slice growth is bounded only by the wire-transport cap (via **Claim Check**), not by a row-property cap; an aggregate caught behind a slow downstream during a burst cannot overflow its own persisted document. One engine, **four effect kinds**: `PublishEvent` (→ domain stream), `SendCommand` (→ aggregate grain, the saga's path), `UpsertRow` (→ Table Projection write store), `InvokeHandler` (→ deferred dispatch on the host's own idempotent-consumer surface; widened by ADR 0026 to carry an `EdictEventEnvelope` payload so the same entry kind serves both off-the-stream-callback deferral (ADR 0023) and pointer-envelope materialisation on the receiver side). Each entry carries the captured `traceparent`/`tracestate` so the deferred dispatch restores the command context as **parent** (ADR 0003 stays literally true — parent-child, no span links). At the commit boundary (the same write that lands the entry list), the flush pipeline performs the **Claim Check** decision per buffered event — serialise, measure, optionally write to the off-wire blob store, wrap in `EdictEventEnvelope` — so an oversized event is replaced in-place by a small pointer-bearing envelope before the entry is persisted (ADR 0024). Drain is **per-entry independent retry, sequential in insertion order** (ADR 0026 — supersedes ADR 0018's "FIFO, stop-at-head"); it runs **awaited inline after the commit** (happy-path trace identical to the pre-Outbox code) with a **lazy Reminder** as the durable crash-recovery net — registered only when the outbox is non-empty, unregistered when fully drained, so steady state holds zero reminders. Each entry has an `AttemptCount`/`NextAttemptUtc`; backoff is a pure function of `AttemptCount` gating that same one reminder (no second scheduling primitive). A failing entry bumps its own `NextAttemptUtc` and the drain **continues** past it; later entries are not blocked. On `MaxAttempts` exhaustion the entry promotes to dead-letter (atomic remove + append synthetic `PublishEvent(EdictDeadLetterRaised)` in the same write). Happy-path ordering (N events raised in one Handle, all succeed first try) is preserved by insertion-order sequential drain; only the retry path reorders — the reorder consumers must already tolerate for redelivery. This **closes** the ADR-0012 Table-Projection double-apply gap as the `UpsertRow` special case (the prior "planned, not yet built" status is retired); the closure holds for **inline-payload envelope** delivery (ADR 0026 amends ADR 0012 — pointer-bearing envelope delivery to projections is two writes). The injected clock is a `TimeProvider` seam so the Test Framework can virtualise backoff.
_Avoid_: an Outbox grain; a second store for entries; span links instead of parent-child; treating it as table-projection-only; assuming exactly-once publish (it is at-least-once — consumer dedup, ADR 0002, makes it effectively-once); assuming per-aggregate causal order for `PublishEvent` across multiple events raised in one Handle once any entry has failed (happy path is in-order; retry path reorders — model accordingly); folding the publisher-side claim-check blob write into the engine (it is deliberately synchronous at the commit boundary so raw oversized event bytes never sit in grain state — pointer-in-entry is size-bounding, ADR 0026).

**Dead Letter**:
The terminal tail of the **Outbox**, intrinsic to it and **forensic-only** — observability, not back-pressure (ADR 0022 supersedes ADR 0019). An entry that keeps failing is retried with exponential backoff; only a *permanently* failing entry (max attempts exhausted) is **dead-lettered**. There is **no in-grain `DeadLetter` slice**, **no cap**, and **no intake block** — the aggregate keeps accepting commands regardless of how many effects have ever poisoned. Promotion is one atomic grain-state write: the engine removes the failed entry from the `Outbox` slice and appends a new `PublishEvent` Outbox entry carrying an `EdictDeadLetterRaised` event. That event flows through the standard Outbox path (at-least-once, `traceparent` preserved per ADR 0003) and is consumed by the built-in singleton `EdictDeadLetterProjectionBuilder`, which upserts a row into a fleet-wide table partitioned on the constant `"deadletter"`. The projection is the **one framework-special-cased consumer**: when an incoming `EdictEventEnvelope` is pointer-bearing the framework's stream observer suppresses the auto-fetch for this projection specifically, and the projection writes the **Claim Check** key directly into its row instead of the materialised body — preserving the forensic affordance without ever trying to fit a >32 KB payload into a 32 KB property (ADR 0024). The dead-letter surface also covers a **receiver-side** failure mode: an `EdictEventBlobMissing` synthetic outcome promoted when the framework cannot materialise an inbound pointer-bearing envelope after `MaxAttempts` (e.g. the operator's lifecycle policy reaped the blob), so a misconfigured retention does not wedge a consumer indefinitely. A read-only `IEdictDeadLetterRepository` (consumer-facing, `Edict.Contracts`) exposes both per-aggregate (`ListAsync(grainKey)`) and fleet-wide (`ListAllAsync()`) queries. There is **no `RedriveAsync`** — recovery is **manual re-emission**: for `PublishEvent` / `SendCommand` the source aggregate or saga is re-invoked; for `UpsertRow` the operator repairs the row by hand, informed by the dead-letter projection. The dead-letter projection and repository are auto-wired by `AddEdict()` so the framework records poison by default rather than depending on a manual DI call.
_Avoid_: an in-grain dead-letter slice or cap; blocking aggregate intake when downstreams fail; expecting a redrive affordance; treating dead-lettering as a recovery mechanism rather than an RCA surface; reading the dead-letter table directly instead of via `IEdictDeadLetterRepository`.

**Event Envelope** (`EdictEventEnvelope`):
The universal wire-format wrapper carried on every Edict stream hop (ADR 0024). It derives from `EdictEvent`, inherits the trace fields and identity stamping the framework already applies, and carries either an inline payload (`byte[]?`) or a **Claim Check** pointer (`string?`) — exactly one is set per instance, asserted on construction. The framework's commit-time flush wraps every raised event in an envelope; the receiver's stream observer unwraps it before dispatching to the consumer's `Handle`, so consumer code never names this type and never sees it on a `Handle` signature. The envelope is a deliberate evolution point — it is the load-bearing place for future framework wire-format metadata (schema-version tags, encryption envelopes, multi-tenancy hints) — and that future-headroom is the reason every event pays the ~10-byte universal-wrapping tax instead of conditionally wrapping only oversized events.
_Avoid_: deriving consumer event types from `EdictEventEnvelope` (concrete events still derive from `EdictEvent`); reading it on a consumer `Handle` signature (the unwrap happens before dispatch); treating it as solely a claim-check vehicle (it is the wire envelope, claim-check is one use).

**Claim Check**:
The escape hatch for oversized events. An event whose MessagePack-serialised body exceeds the per-provider threshold (default 30 720 bytes on the Azure provider — 2 KB of headroom under the 32 KB Azure Table per-property cap for envelope framing) is written at the **Outbox** commit boundary to a separate **append-only** blob store, and the in-grain Outbox entry, the queue body, and any downstream dead-letter row all carry a small pointer string instead of the body. Lifecycle is owned by the storage account's policy (Azure Blob lifecycle rules), never by the framework — Edict only writes, never deletes — which is the only design that lets one mechanism serve both the wire-hop case and the forensic poison-row case without a second store or reference-counting (ADR 0024). The decision is made at the commit boundary (deliberately synchronous so raw oversized event bytes never sit in grain state — pointer-in-entry is size-bounding, ADR 0026) so `Raise` stays a synchronous in-memory buffer call; multiple oversized events in one Handle parallelise (`Task.WhenAll`). The receive-side fetch is automatic for every consumer **except** `EdictDeadLetterProjectionBuilder`, which is special-cased to store the pointer (see **Dead Letter**). Missing-blob on fetch is folded into the engine's per-entry retry as part of the widened `InvokeHandler` entry (ADR 0026 supersedes the original in-grain `BlobMissing.Attempts` tracker): the stream observer's pointer-envelope path stages an `InvokeHandler` entry whose executor fetches the blob; transient fetch failures backoff via the same primitive as every other effect; `MaxAttempts` exhaustion promotes via the same `IDeadLetterPromoter` as the synthetic `EdictEventBlobMissing` notification. The store seam (`IEdictClaimCheckStore`) lives in `Edict.Contracts`; the production Azure Blob implementation lives in `Edict.Azure`; an in-memory implementation ships with the test framework and mirrors the production threshold so test runs exercise the same commit pipeline as production. This does **not** reintroduce event sourcing — there is no replay affordance, no `Rehydrate`, no path that reads a blob outside the live-event hop or operator-led forensic inspection (ADR 0001 unchanged).
_Avoid_: deleting blobs from framework code (append-only is load-bearing); estimating event size by anything other than the serialised byte length (which is exact and free at the commit boundary); fetching the body into the dead-letter row (the pointer is the row's value); treating the threshold as a stream-only concern (the grain-document per-property cap is the more restrictive bound on the publish path, so the threshold is a provider-package concern, not a stream-provider concern); treating blobs as an event log.

**Idempotency Base** (`EdictIdempotencyBase<TPayload>`):
The abstract generic base that Event Handlers (`EdictEventHandler`), Sagas, and Projection Builders inherit (the shared inheritance root for the idempotent-consumer family — brand-rule clause b stays literally true; it derives in turn from the outer shared root `EdictDurableConsumerBase<TPayload>`). `TPayload` is `EdictUnit` for Event Handlers and Projection Builders, `TProgress` for a Saga; the persisted document is the unified envelope `GrainEnvelope<TPayload> { Payload, Outbox, Idempotency }` — the dedup state is a sibling slot, not a wrapper around the payload. A non-generic `EdictIdempotencyBase : EdictIdempotencyBase<Unit>` shim keeps payload-free subclasses from sprouting `<Unit>`. Its job is idempotency: a configurable bounded per-grain window of recently handled `EventId`s that suppresses at-least-once redeliveries, committed *after* the subclass's dispatch succeeds. It does **not** decide which stream to subscribe to — the consuming type declares that itself.
_Avoid_: implying it owns or configures stream subscription; the old `EdictEventDeduplicationGrain`/`EdictEventIdempotentGrain` names (renamed in ADR 0017).

## Relationships

- A **Command Validator** gates a **Command** server-side, in the same activation turn, *before* its **Command Handler**'s `Handle` runs; it reads but never mutates aggregate state and yields a `Rejected` **Command Result** on failure
- A **Command Handler** handles **Commands** (direct grain call), mutates framework-owned durable `State`, returns a **Command Result**, and may raise **Events**; the state mutation and the raised events commit in one atomic grain-state write via the **Outbox**
- A **Saga** reacts to an **Event** and issues exactly one **Command** (`Dispatch`, hard limit); its progress, dedup window, and outbound command commit atomically via the **Outbox**
- The **Outbox** is one engine with three effect kinds (`PublishEvent`, `SendCommand`, `UpsertRow`); a permanently failing entry is **dead-lettered**; delivery is at-least-once and made effectively-once by the **Idempotency Base** dedup window
- A **Command** is routed to exactly one aggregate grain by its single `[RouteKey]` **Guid** property; that Guid is the grain key and the future event stream's `sourceAggregateGuid`
- A creation command (e.g. `PlaceOrder`) routes identically — the caller mints the Guid; Orleans' virtual grains make the not-yet-activated aggregate addressable
- A consumer issues a **Command** through the **Sender**; the **Sender** is the seam `Edict.Testing` swaps for an in-memory implementation
- **Event Handlers**, **Sagas**, and **Projection Builders** subscribe to **Events** via implicit stream subscriptions and all inherit the **Idempotency Base** (`EdictIdempotencyBase`). An **Event Handler** is terminal (no `Raise`, no `Dispatch`) and its `Handle` is invoked via the **Outbox**'s `InvokeHandler` effect kind — dedup-window commit happens when the event is *accepted*; `Handle` runs later, possibly under backoff, and permanent failure lands in the **dead-letter** projection (ADR 0023)
- An **Event** is published to its **Domain Stream** (named by `[Stream]` on the event), keyed by the event's `[RouteKey]` Guid; every subscriber to that stream is activated with that Guid and acts only on event types it has a `Handle` overload for. A fixed-Guid singleton is the explicit escape hatch for a global read model.

## Example dialogue

> **Consumer:** "My `ProjectionBuilder` for `OrderPlaced` — how does it get the historical orders when it first starts?"
> **Edict author:** "It doesn't. Edict is event-driven, not event-sourced — there's no replay. A Projection Builder only ever sees events from the moment it's subscribed, forward."
> **Consumer:** "And if the same `OrderPlaced` is delivered to it twice?"
> **Edict author:** "It inherits `EdictIdempotencyBase`. The `EventId` dedup window suppresses the second delivery for *that* projection — but the same event still reaches your `OrderEmailHandler`, because dedup is per consuming grain, not global."
>
> **Consumer:** "Where does 'can't cancel an already-shipped order' go — a Command Validator or `Handle`?"
> **Edict author:** "Either could read the state, but the line is mutation. The validator is a precondition gate: it inspects current state and rejects *before* any transition, no writes. `Handle` owns the transition. If the rule is knowable from current state without attempting the change, it's a validator; if the rejection only emerges *while* mutating, it's a `Handle` outcome. Both return the same `Rejected` — they differ by *when* and *whether they mutate*, not by the envelope."

## Flagged ambiguities

- "Projection" implied event-sourcing replay — resolved: Edict projections only consume the live stream forward; no event store, no replay, no rebuild.
- Events called both "transient" and inputs to projections/sagas/idempotency — resolved: events are transient (discarded after handling); durability/replay is never assumed.
- `Command` was said to carry "trace correlation" — resolved: it does not. Direct grain calls propagate `Activity` context natively (ADR 0003); only `Event` carries trace fields because only the stream hop loses context.
- "One command produces one event" — clarified: a **simplifying assumption**, not an enforced constraint. `Raise` buffers and the flush handles N events mechanically; nothing analyzer-enforces single-event. `Raise` is the *only* sanctioned publish API — a handler reaching Orleans streams directly is caught in code review, not by an analyzer.
- A domain stream carries many event types, so a subscriber is woken for event types it has no `Handle` for — resolved: an unhandled type is a **pure no-op that consumes no dedup window slot and is never recorded as seen**. The bounded window of handled `EventId`s (ADR 0002) is reserved for events the subscriber actually handled, so ignored types cannot evict real entries and let genuine duplicates of handled events slip.
- "Single correlation id command → grain → event → handler" implied the same Guid flows end-to-end — resolved: the command's `[RouteKey]` and the event's `[RouteKey]` are independent addressing keys that coincide in the in-domain case but diverge on cross-domain re-keying (`Sales` raising an `Orders` event). The reliably continuous correlation is **trace context** (ADR 0003), not the Guid. The retired term `sourceAggregateGuid` is replaced by "the event's route key".
- Bare base names `Command`/`Event` vs `EdictCommand`/`EdictEvent` — resolved: Edict is a brand; the consumer-facing surface is `Edict`-prefixed, internals stay bare (see *Naming convention*). This supersedes the original "no prefix" decision. Wire identity is unaffected: ADR 0010 keys it on the *concrete* command's simple class name (`[Alias(nameof(PlaceOrder))]`), not the base type name.
- "Table Projection Builder" implied Azure Table Storage was intrinsic — resolved: it is the generic external-composite-key-store mechanism; Azure is one implementation in `Edict.Azure`. The grain base stays persistence-agnostic in `Edict.Core` behind a dumb write-store seam; the row is a plain POCO.
- "Validation" conflated structural input checks with business rejection — resolved: a **Command Validator** is a no-mutation *precondition gate* (admissibility against current state, *before* the transition); `Handle` owns the *state transition* and returns `Rejected` only for outcomes discoverable while mutating. Same `Rejected` envelope, two distinct homes, distinguished by *when they run* and *whether they mutate* — not by structural-vs-business.
- The `Grain` suffix on base classes (`EdictCommandHandlerGrain` etc.) and the brand rule's strict "iff a consumer types it" — resolved (ADR 0017): `Grain` is dropped everywhere ("Grain" is an Orleans noun a consumer never names); consumer subclasses are `{Name}{Role}`. The brand rule gains clause (b): a shared inheritance root (`EdictIdempotencyBase`) keeps the prefix even though no consumer derives from it directly. Raw Orleans test doubles that truly derive from `Grain` keep "Grain".
- Code/doc drift on the dedup root — resolved: the type the docs called `EdictEventDeduplicationGrain` and the code had renamed to `EdictEventIdempotentGrain` is now `EdictIdempotencyBase` (ADR 0017); the `Dedup/` folder is `Idempotency/`.
- Whether persisted serializable state should alias like commands — resolved (ADR 0017, extended by ADR 0027): commands and events use `[Alias(nameof(TheClass))]`, **emitted by the generator** as a sibling partial declaration (ADR 0010 — wire id is intentionally the concrete simple name; renaming a command/event is a deliberate breaking change). Alongside the alias the generator also emits `[MessagePackObject(keyAsPropertyName: true)]` so the consumer writes neither wire-format obligation by hand (ADR 0027 — "the generator emits invariant scaffolding, the consumer owns decisions"). **Persisted grain state** (`IdempotencyState`, consumer aggregate state, saga progress, table-projection row POCOs) uses a **frozen string literal** `[Alias]` because state outlives deploys and must survive a class rename; the consumer writes it themselves (the generator cannot — frozen-literal semantics require knowing "I renamed but the wire identity stays put", which only the consumer at the keyboard knows). Every consumer-authored persisted-state type implements the marker `IEdictPersistedState`; the analyzer EDICT011 errors when the implementation lacks `[GenerateSerializer]`, the frozen string-literal `[Alias]`, or `[Id(n)]` on any public property. `ORLEANS0010` is never suppressed.
- "Outbox" meant two different things (ADR 0012 table-projection row+window atomicity vs general outbound-message reliability) and was "planned, not built" — resolved: it is **one engine**, three effect kinds (`PublishEvent`/`SendCommand`/`UpsertRow`), entries co-located in the single grain-state write (atomicity by co-location, not a distributed transaction Orleans cannot give). Enabling reversal: `EdictCommandHandler` is no longer a stateless bare `Grain` — it is the stateful generic `EdictCommandHandler<TState>` (amends ADR 0004), because the Outbox needs a durable state document to be atomic *with*. The current sample's volatile grain fields are now wrong and must move to `State`. Drain is awaited inline + lazy Reminder net; trace stays parent-child (ADR 0003 untouched). Closes the ADR-0012 gap as the `UpsertRow` case. Dead-lettering (exponential backoff, in-doc capped slice, block-intake on overflow, grain-method redrive, read-only `IEdictDeadLetterRepository`) is intrinsic to it. `EdictIdempotencyBase` becomes generic `EdictIdempotencyBase<TPayload>` (`Unit` for handlers/projections, `TProgress` for saga) so brand clause (b) and "all three inherit it" stay literally true. Saga is `EdictSaga<TProgress>` with a **one-command-per-event hard runtime limit** (`Dispatch`), a deliberate asymmetry with `Raise`. Folders: `Outbox/`, `Saga/`, `DeadLetter/`.
- Oversized events would silently corrupt the grain document (Azure Table per-property cap 32 KB), the queue body (Azure Queue raw ~48 KB), and the fleet-wide dead-letter row (Table per-property 32 KB again) — resolved (ADR 0024): one **Claim Check** mechanism, append-only blobs with storage-policy lifecycle (Model B), universal `EdictEventEnvelope` wrapper on every wire hop, raise-time decision at the Outbox commit boundary (so `Raise` stays a synchronous in-memory buffer call), per-provider threshold (`EdictAzureOptions.ClaimCheckThresholdBytes`, default 30 720) with 2 KB of headroom for envelope framing, dead-letter projection special-cased to store the pointer instead of the body, missing-blob on fetch funnels through the existing dead-letter promotion path as a synthetic `EdictEventBlobMissing` outcome (extends the dead-letter conceptual surface from "outbound effect failed at producer" to also include "inbound event could not be materialised at consumer"). Does not reintroduce event sourcing — blobs are an oversized-payload escape hatch, not a replay log, and ADR 0001 stays literally true.

- "FIFO, stop-at-head" and "per-aggregate causal order" were load-bearing in ADR 0018's drain mechanic and the explanation a consumer was given for ordering guarantees — resolved (ADR 0026): both retracted. Drain is **per-entry independent retry, sequential in insertion order**; happy-path ordering is preserved (N events raised in one Handle, all succeed first try, publish in raise order); only the retry path reorders. The per-aggregate causal-order claim was already weak in practice (the stream is at-least-once; a redelivered older event lands at the consumer *after* a newer one of the same aggregate, the dedup window dedupes but ordering is lost), so the explicit retraction matches the real consumer-observable contract. The receiver-side claim-check `BlobMissing.Attempts` tracker folds into the engine's per-entry retry on a widened `InvokeHandler` entry; the publisher-side blob write at the commit boundary is deliberately unchanged (folding it in would require carrying raw oversized event bytes in grain state, which is silo-hostile at scale — ADR 0025 removes the row-cap obstacle but the per-write/per-activation IO cost remains the load-bearing reason). ADR 0023's "dedup-window commit ≠ effect completion" asymmetry generalises from EdictEventHandler-only to any deferred-dispatch path (inline-payload delivery to projections still preserves ADR-0012's atomicity; pointer-bearing delivery does not).

- "Event Handler" was a named role with no shipped base type — resolved (ADR 0023): `EdictEventHandler` is the consumer-facing base for reacting to events with external side effects. Mechanic is deferred-drain via the Outbox (new `InvokeHandler` effect kind), not inline-on-the-stream-callback: the dedup-window slot and an `InvokeHandler` Outbox entry commit atomically in one write, then the drain re-invokes the consumer's `Handle` with exponential backoff and dead-letter promotion at `MaxAttempts`. The role is **terminal** — no `Raise`, no `Dispatch` — enforced by the base not exposing those methods. **Dedup-window commit ≠ effect completion** for this role only (a deliberate asymmetry with `EdictProjectionBuilder`, where they are synonymous); the trade buys the framework's existing retry/backoff/dead-letter machinery for the external-I/O case. External-API idempotency is the consumer's responsibility (`evt.EventId` is the canonical idempotency key). Chaos in `Edict.Testing` defaults **OFF** for `EdictEventHandler` invocations specifically (side effects are observed by mock call counts; chaos-induced duplicates would flake the canonical first test).
- Whether the sample app may use in-memory infrastructure — resolved: it may not. The Sample is a *real-system* reference (Azure Table grain storage for PubSub + the dedup window, Azure Queue streams, Azure Table projections); `Sample.AppHost` provisions Azurite and injects the `queues`/`tables` connection strings. In-memory wiring is the shipped **Test Framework**'s job, never the sample's. `UseLocalhostClustering` is kept (a single-silo sample's clustering provider is not "storage").

- The framework split between codegen-emitted attributes and consumer-written attributes had no stated principle — resolved (ADR 0027): **the generator emits invariant wire-format obligations on transient types (commands and events get `[Alias(nameof(TheClass))]` + `[MessagePackObject(keyAsPropertyName: true)]`); the consumer writes long-lived contracts on persisted state (aggregate state, saga progress, row POCOs all get hand-authored `[GenerateSerializer]` + frozen-string-literal `[Alias]` + `[Id(n)]`)**. The line is "can the value be recomputed safely from current syntax on every build?" — values that can are codegen'd, values that must survive class rename / property reorder cannot be. The consumer-owned persisted-state contract is signalled by the marker `IEdictPersistedState` (generic-constrained on `EdictCommandHandler<TState>`, `EdictSaga<TProgress>`, `EdictTableProjectionBuilder<T>`) and enforced by the analyzer EDICT011 at the type's declaration. As a knock-on, `UpsertRowEffect` no longer persists the row's assembly-qualified name; it persists the frozen `[Alias]` string and the drain resolves it via `Orleans.Serialization.TypeConverter.Parse` — closing the AQTN dead-letter hole `docs/resiliency-analysis.md` §3.2.3 (and resiliency priority #1) flagged.

- The framework's installation and config surface was scattered across three `IServiceCollection` calls (`AddEdict()`, `AddEdictOutbox()`, `AddEdictAzureClaimCheck()`), several magic numbers buried in mechanism code (`RingSize = 100`, drain reminder period `1min`, stream provider name `"edict"`, claim-check container name `"edict-claim-check"`, dead-letter table name), a silent clamp on `EdictOutboxOptions.JitterFraction`, and no startup failure for a half-wired host — resolved (ADR 0028): three `ISiloBuilder` extensions (`AddEdict` core, `AddEdictAzureStreams` for the stream provider + claim-check threshold policy, `AddEdictAzurePersistence` for grain storage + table repo + claim-check blob store + dead-letter table), each accepting an `Action<T>` lambda so the consumer reads `IConfiguration` however they want. Every tunable value moves onto an options class as a property with its previous magic value as the constructor default — no literals in mechanism code. An `IHostedService` runs at `StartAsync` and (a) verifies markers from each required provider call are present, throwing an aggregated message listing every missing call, and (b) calls `OptionsBuilder.Validate().ValidateOnStart()` — invalid values throw at startup, no silent clamping. `Ring` in code and prose was renamed to `Window` to align with CONTEXT.md's already-canonical "dedup window" term (`RingSize` → `WindowSize`; the persisted field `HandledEventIds` keeps its `[Alias]`-frozen name). `Sample.Silo/Program.cs` demonstrates every option with its literal default value on its own line, so the sample doubles as the framework's config catalogue. The principle for future mechanisms: every new framework knob is an options property with a default, a validation rule, and a line in `Sample.Silo/Program.cs` — never a literal in mechanism code.
